## Context

The profile knowledge base is the running app's per-profile knowledge layer: expert bands, UGS positions, analysisFlags, aliases, category/family/roast, and the prose fed to the LLM. Today it is authored as `resources/ai/profile_knowledge.md` and parsed by `ShotSummarizer::loadProfileKnowledge` (`src/ai/shotsummarizer_kb.cpp`) via `line.startsWith("UGS:")`-style string scraping into a `QMap<QString, ProfileKnowledge>`. The same per-profile facts also live in a hardcoded C++ `kBands` table (`expertBandForKbId`). A second hand-maintained copy, `docs/PROFILE_KNOWLEDGE_BASE.md` (1170 lines vs the runtime 707), carries the full citations and drifts independently. (Note: `shotsummarizer.cpp:1290/1302`, earlier mischaracterized as inlined facts, are actually prompt instructional text — `:1290` a family-switch teaching example, `:1302` an anti-hallucination guardrail — not stranded facts; they are retained verbatim.)

Two failure classes are in scope. **(A) Malformed data:** a typo in a scraped key (`Ugs:`, stray indent) silently drops the field and the detector computes a wrong answer with no signal. **(B) Mis-resolution:** `matchProfileKey` (`shotsummarizer_kb.cpp:303–314`), after an exact-match miss, runs an order-dependent greedy `startsWith`/`contains` scan over `QMap::keys()` and *always returns something*. Because `"d-flow"` sorts before and is a prefix of `"d-flow / q"`, a user title `"D-Flow / Q - Jeff"` silently resolves to the band-less `## D-Flow` default. This has shipped wrong analysis more than once (shot 819; the `dialing_blocks.cpp:847` comment records the `"D-Flow / Q - Jeff"` case; design D14a documents the stale-kbId variant). `flag-off-expert-band-in-shot-summary` is shipped, so the `kBands` table is live debt; this is a clean standalone paydown with no sequencing constraint.

## Goals / Non-Goals

**Goals:**
- One structured, schema-validated JSON source for every app-specific per-profile fact; C++ holds zero facts.
- Malformed data and should-resolve-but-doesn't are **build/test failures**, never silently-shipped wrong lookups.
- An exact-match-or-explicitly-unresolved resolver with a corpus test as a hard gate; the greedy fallback deleted, not ported.
- Collapse the docs/runtime two-file drift to one authored source.
- Preserve recompute-on-load, the `ExpertBand` type, every C++ consumer signature, and the assembled LLM prose blob.

**Non-Goals:**
- The duplicated editor-membership rule (`title.startsWith("D-Flow"/"A-Flow")`) — that is logic, a separate shared-helper dedupe.
- Factory default profile names; MCP A-Flow editor-param schema descriptions (different domain).
- Any new badge, persistence/DB-schema, Visualizer, QML, or LLM-prompt *behavior* change beyond sourcing the same facts from the KB.
- Mirroring portable profile parameters (frames/limits/setpoints) into the KB.
- Re-deriving any band/UGS/citation evidence — the migrated values are transcribed verbatim from the shipped sources.

## Decisions

**D1 — Authored format is JSON, parsed by `QJsonDocument`; runtime stays JSON (Option B).** JSON is the codebase-native structured format (every profile is JSON; `QJsonDocument` is already linked), so zero new dependency and a real parser that fails loud on malformed input — directly answering failure class A. *Alternative (rejected): keep markdown + add an `ExpertBand:` grammar line.* That is exactly the typo-fragile string-scraper this change exists to remove. *Alternative (deferred, scoped fallback): compile JSON to a build-time read-only SQLite resource read via SQL.* Its only edge over B is engine-enforced declarative `UNIQUE`/FK and a relational corpus join; at ~43 sections that is marginal and adds a generated binary artifact through the 6-platform build path that broke last week (`2b6e0965`, qrc custom-command on the macOS Xcode generator). Taken **only if** D5's integrity guarantee cannot be expressed as a simple enumerate-and-assert and needs hand-rolled relational checks complex enough to be under-specified. Decision point: validator design (D5).

**D2 — Single source of truth: all per-profile facts move into the KB; C++ keeps only logic.** The `kBands` table is deleted; `expertBandForKbId` reads the KB. (The `shotsummarizer.cpp:1290/1302` prompt text is *not* a stranded fact — `:1290` is a teaching example, `:1302` an anti-hallucination guardrail; both stay verbatim, underlying facts are KB-canonical.) The split is *facts vs logic*: a per-profile branch selecting a different *value or sentence* is data → KB; a branch selecting different *behavior* (detector algorithms, analysisFlag→suppression semantics, resolver normalization) is logic → stays code. This prevents the inverse failure of encoding behavior as data strings. *Alternative (rejected): leave bands in C++ "resolved fresh as shipped code".* The shipped-code property is equally true of a qrc JSON resource (ships in the binary, parsed once at load), so it never justified the duplication.

**D3 — Structure only the load-bearing facts; everything else is re-authored prose.** Each entry carries typed, validated fields and a single `prose` string. Final field set: **`id`** (stable kebab-case key — the *sole* identity/resolution/cross-reference key), **`displayName`** (human label, prose, never a key), `alsoMatches`, `defaultForEditorType`, `ugs{value,inferred,note}`, `analysisFlags`, `skipCatalog`, `family`, `expertBand{axis,lo?,hi?,src?,provenance,confidence,rationale,srcArchive?}`, `prose`. **`id` vs `displayName` is the core solidity fix:** today the resolver and band lookup key on a prose display string (`pk.name`/normalized title), so a rename silently breaks identity — the D14a stale-kbId bug class. A stable slug as the only key retires that class entirely; the display name is free to change. `family` is a **validated enum** over the closed ≈13-token vocabulary (`lever-decline`, `flow-adaptive`, …): the advisor's switching-recommendation logic is built on family membership, so a typo is a build failure, not a silent miscluster. **`category`, `roast`, and `summary` are NOT fields.** They were near-unique descriptive paraphrase consumed only by the one catalog line; `family` already carries the actionable cross-profile signal and `prose` carries the richer description, so they earn their keep nowhere (and any future user-profile categorization is `family`'s job — a controlled axis — not free-text category's). The catalog line becomes `displayName [family: <family>]`. `prose` is **one opaque, re-authored string**: nothing parses inside it, so a typo there cannot break a lookup. It is **not** required to byte-match today's `pk.content` — only the app ever produced that text and it is not a contract; prose is rewritten for clarity/correctness where that helps (D-numbers and unhelpful prose may be fixed). What *is* gated is **fact-value fidelity** (D5): band numbers, UGS, flags are transcribed faithfully and any deliberate change is enumerated/reviewed, never silent. `expertBand.rationale` is the home for the citation/validation audit (the former `kBands` C++ comment, verbatim — JSON has no comments). `lo`/`hi` are each optional so one-sided rails (Allongé flow floor) stay first-class, matching the existing `ExpertBand` type. *Alternative (rejected): decompose prose into named blocks.* ~8 hand-cut fields × ~43 profiles serving no parse/lookup need — prose is never parsed — adds error surface without advancing typo-safety. File granularity (one document vs per-profile files) is a free, non-load-bearing choice; default is one document so the cross-profile reference (`skipCatalog`) sections stay together.

**D4 — Resolver is exact-match-or-explicitly-unresolved; the greedy fallback is deleted.** Resolution: normalize the title (the existing accent/punctuation normalization is logic, retained) → exact lookup in an explicit **alias→`id`** map built from every entry's `alsoMatches` + `displayName` + the `defaultForEditorType` entries. The resolver yields an `id` (never a prose string); all consumers key on `id`. A miss returns an explicit *unresolved* outcome. The order-dependent `startsWith`/`contains` scan is removed entirely — it is the root cause of class B and cannot be made safe. *Alternative (rejected): keep a "best prefix" heuristic.* Any heuristic that always returns something reproduces silent mis-resolution; the requirement is that wrong-but-confident cannot exist.

**D5 — Two validators, both hard build/test gates.** (a) **Schema/structure validator** (build-time, fails the build): every entry conforms to the schema; unknown/typo'd keys rejected; `id` unique and kebab-case; `expertBand` obeys `lo<hi` (when both present), positive bounds, axis ∈ enum; `provenance` ∈ enum with the cross-field rule **`src` required when `provenance=cited`, forbidden when `provenance=inferred`**; `src` (when present) a well-formed `http`/`https` URL (external *or* our GitHub) or a closed intrinsic token; a set `srcArchive` points to an existing repo file; every `alsoMatches`/`displayName`/`defaultForEditorType` key resolves to exactly one `id` with no duplicate or orphaned alias. Plus the **fact-value parity** check (band/UGS/flag values equal pre-migration unless the change is on the enumerated reviewed-corrections list — no silent value change) and the **D9 best-effort lint**. (b) **Corpus resolution test** (`tst_kb_resolution`, hard gate): every profile title appearing in `tests/data/shots/`, the shipped starter profiles, and representative D-Flow/A-Flow editor outputs resolves to exactly one canonical entry; the shot-819 / `"D-Flow / Q - Jeff"` cases are explicit fixtures. This is the test that would have caught the shipped bugs. The integrity checks in (a) are the B-vs-fallback decision gate: if alias uniqueness/referential integrity stays a clean enumerate-and-assert over the parsed map, B holds; if it needs hand-rolled relational machinery, escalate to the SQLite fallback (D1).

**D6 — Loud/silent splits by expectation, not globally.** A profile that *should* resolve (in the corpus / a shipped or editor profile) but doesn't → loud build/test failure (D5). A genuinely unknown profile at runtime (new community/custom title with no KB entry) → silent no-op: resolver returns unresolved, `expertBandForKbId` returns `std::nullopt`, the check is byte-identical to today's absence-intentional behavior (shipped D5 of `flag-off-expert-band-in-shot-summary`). Making runtime-unknown loud would spam every custom profile and contradict shipped design; this is the only internally consistent reading.

**D7 — KB stays separate from the portable profile and never mirrors its parameters.** Profiles round-trip between apps (Visualizer, de1app); KB knowledge is Decenza-unique and must not be embedded in or copied into profile JSON. The KB↔profile join is the resolver only. Portable parameters (frames, limits, setpoints — including the limiter value the D1 corroboration clause needs) are read from the profile at analyze time, never stored in the KB; copying them in would create a new KB-vs-profile drift surface. The KB *rationale* prose may *quote* a profile's limit as cited evidence (it already does) — that is content, not the app reading a parameter from the KB.

**D8 — One authored source; rollup generated; the old doc is removed ONLY after a duplication differential proves no knowledge is lost.** The JSON is the single authored source of *facts*. The runtime `resources/ai/profile_knowledge.md` is fully superseded by the JSON and is removed outright. `docs/PROFILE_KNOWLEDGE_BASE.md` (1170 lines, the fuller design-doc with full citations) is **not** deleted on faith: proving "everything in it is captured elsewhere" by hand is prohibitive research, so instead a **duplication differential** (task 6.1a) strips everything now represented in the JSON and surfaces the residue; each residue item is either folded into the JSON (`prose`/`rationale`) or preserved in a slimmed `PROFILE_KNOWLEDGE_BASE.md` that holds *only* the residue. Deletion is permitted **only** once the residue is 100% accounted for (Jeff's hard rule — un-captured knowledge must never be silently lost). Any browsable rollup is generated from the JSON, never hand-edited. This still forces drift to zero, without risking knowledge loss.

**D9 — Prose numbers are commentary, not injected; the expert band exists exactly once.** Numbers inside `prose` are descriptive commentary and are NOT substituted at runtime. Most describe the *portable profile's* own parameters (setpoints, limits), which the KB deliberately does not own (D7) and which have no single well-defined value to inject (a profile has many frames); the prose↔profile descriptive drift there is pre-existing and accepted. The one fact that must exist exactly once is the **expert band**: it is rendered into the assembled LLM blob from the `expertBand` struct (struct → one cited sentence), and `prose` is authored to NOT also state the band as the band — so the band number lives only in `expertBand.lo/hi/src`. *Alternative (rejected): generic "scan prose for numbers, inject from structured fields".* Adaptive v2 is the disproof — its `expertBand` is 6.0–9.0 (the deliberately wide judge-by-taste envelope) while its prose dial-in says "peak 8-9 bar … too coarse below 7 bar" (the ideal target); these numbers are intentionally different, so mechanical substitution would corrupt meaning. The validator carries a **best-effort lint** that flags a prose line that restates the struct band verbatim/near-verbatim, catching the regression where an author reintroduces a third copy.

**D10 — Honest, structured provenance; a citation is preferred but not mandatory.** Every `expertBand` carries a required `provenance` ∈ {`cited`, `author-stated`, `inferred`} and a required `rationale` stating *what is real and why we believe it*. `cited` ⇒ `src` is a followable real-doc URL — external, **or our own GitHub** (the relocated `docs/kb_sources/` copy at its repo URL is a valid stable reference). `author-stated` ⇒ the profile's own shipped notes are the source (`src` = `profile-notes`, intrinsic). `inferred` ⇒ no source exists; that is acceptable, but `src` is forbidden and `rationale` must justify the reasoning. The point of the URL is epistemic hygiene — it separates grounded truth from reasoning — not bureaucratic completeness; absence is fine when stated honestly. Making provenance a validated enum (not buried in prose) keeps "what is real" structured and checkable, matching this change's whole ethos. *Alternative (rejected): keep provenance prose-only.* It is exactly the kind of load-bearing epistemic fact this change exists to lift out of fragile prose into validated structure.

**D11 — The LLM prompt must be equivalent or improved, never degraded.** The end-state assembled shot-analysis prompt the model receives is held to a hard non-regression bar: same or better, never worse, as a result of this change. Concretely: the `shotsummarizer.cpp` instructional examples and the `:1302` anti-hallucination guardrail are retained verbatim (their accidental deletion was the design-issue this decision was added to prevent); the expert-band claim is now struct-rendered once (D9) rather than hand-typed; re-authored prose is clearer but factually faithful (D3 + the D5 fact-value gate). Validated at the end by a prompt-equivalence check (a fixed representative profile set; assembled-prompt diff old-vs-new) where every difference must be a deliberate, reviewed improvement — an unintended or degrading change fails the gate. This is the explicit guarantee that a KB-structure refactor cannot quietly worsen the user-facing AI behavior.

## Risks / Trade-offs

- **Deleting the greedy fallback changes resolution for some real titles** → That is the intended fix; the corpus test (D5b) pins every known title to its correct canonical entry before the fallback is removed, so any regression is a red test, not a shipped surprise.
- **A migrated band/UGS/flag value is transcribed wrong** → D5 fact-value parity: the post-migration value equals pre-migration for every fact (UGS, flags, every shipped `kBands` band/axis/confidence) *unless* the change is on the enumerated reviewed-corrections list. Deliberate fixes are allowed (unhelpful prose, a nonsensical number) but never silent — every divergence is enumerated and reviewed.
- **The build-time validator becomes complex enough to be under-specified** → That is precisely the D1 escalation trigger to the SQLite fallback with engine-enforced integrity; decided at validator design, not deferred indefinitely.
- **Cross-platform build-codegen fragility (the `2b6e0965` class)** → Option B ships the JSON as a plain qrc resource (no generated artifact in the build graph); only the *validator* runs at build time and it is a pass/fail check, not a code generator, minimizing the failure surface that bit the repo.
- **Prose is re-authored, so it could lose a real fact** → only *prose* is free to be rewritten (it is not a contract; only the app produced today's text). The load-bearing *facts* (bands/UGS/flags/aliases) are typed fields under the D5 parity gate, so a fact cannot be lost in a prose rewrite — it no longer lives in prose. Deliberate prose changes are fine and reviewed; what is forbidden is a *silent* fact change.
- **Scope creep into the editor-membership rule** → Explicitly out of scope (Non-Goals); that rule is logic and a separate dedupe.
- **Prose silently re-states a struct band and the two drift** → D9: the band is rendered once from `expertBand`; prose is authored not to duplicate the band claim; the validator's best-effort lint flags a prose line that restates the struct band verbatim/near-verbatim.
- **Cited URL rots and the citation becomes unverifiable** → for `cited` bands a durable local copy is kept under `docs/kb_sources/` where capturable, referenced by `expertBand.srcArchive` (and `src` itself may be the GitHub URL of that repo copy); the validator hard-fails if a referenced archive file is missing. `inferred` bands have no source by definition — honesty is enforced by the `provenance` enum + required `rationale`, not a URL.
- **Relocating existing saved sources breaks inbound links** → only live references are updated (e.g. `docs/CLAUDE_MD/AI_ADVISOR.md`); archived `openspec/changes/archive/*` are frozen historical records and are deliberately not rewritten; the move is `git mv` so history is preserved.
- **The assembled LLM prompt silently degrades** → explicit gate (D11): the shot-analysis prompt the model receives must be **equivalent or improved**, never worse, as a result of this change. The instructional/guardrail text (`:1290/:1302`) is retained verbatim; the band sentence is now struct-rendered (D9); a final prompt-equivalence check over a fixed profile set diffs the assembled prompt old-vs-new and any difference must be a deliberate, reviewed improvement.

## Migration Plan

1. Land the JSON schema, the authored `profile_knowledge.json` (re-authored from the current `.md` files + the `kBands` table — facts transcribed faithfully, prose rewritten for clarity where it helps), and the build-time validator — validator green before any C++ change.
2. Switch `loadProfileKnowledge` to parse JSON; keep every consumer signature (the `kbId` token they pass/return is now the stable `id`). Land the fact-value parity test (post == pre for every fact except the enumerated reviewed corrections) and the `tst_kb_resolution` corpus gate.
3. Delete the greedy `matchProfileKey` fallback; delete the `kBands` table (the `shotsummarizer.cpp:1290/1302` prompt text is retained — not stranded facts); remove the markdown resource and `docs/PROFILE_KNOWLEDGE_BASE.md` (replaced by the single source / generated rollup).
4. Verification gate: full Qt Test suite green; migration-parity + corpus-resolution tests green; a `shot_eval` run over `tests/data/shots/` showing analysis output unchanged except where a previously mis-resolved profile now resolves correctly (an intended, enumerated improvement); a **KB-coverage check** confirming every profile/title that had a KB section under the old md still resolves to a valid entry (no section dropped, no alias lost); and a **prompt-equivalence check** confirming the assembled LLM prompt is equivalent-or-improved, never degraded, by this change. Rollback is a single revert (additive JSON + parser swap; no DB/schema/persistence change).

## Open Questions

- **B-vs-fallback trigger:** **RESOLVED (task 2.3) — Option B confirmed, SQLite fallback NOT triggered.** Implementing the integrity check in `tools/validate_kb.py` showed alias/identity integrity is a clean ~15-line dict enumerate-and-assert (cross-entry alias→id collision map + per-editor-default count) — no relational joins, no engine. The under-specification risk that would have escalated to SQLite did not materialize.
- **Generated rollup:** whether a browsable human-readable doc is regenerated from the JSON at all, or the JSON is the sole reference. Default: generate only if a reader is actually needed; otherwise drop it (D8).
- **File granularity:** one `profile_knowledge.json` vs a per-profile directory — non-load-bearing per D3; default single document, revisitable without schema impact.
