## Context

Today's grind model (documented in `docs/CLAUDE_MD/RECIPES.md` and `src/history/recipestorage.h:23-27`) has two live, keystroke-level write paths for a single logical value:

- **Inherit** (`recipes.grind_pinned` empty): `SettingsDye::setDyeGrinderSetting`/`setDyeGrinderRpm` (`src/core/settings_dye.cpp:288-320`) write through to the linked bag on every edit (`writeThroughToBag`, `settings_dye.cpp:678-687`), gated by `m_grindBagWriteThroughSuspended` being false.
- **Pin** (`recipes.grind_pinned` non-empty): the same setters skip the bag write (suspended), and a separate `MainController` connection (`maincontroller.cpp:941-949`) stamps the recipe row instead (`stampActiveRecipe`, `maincontroller.cpp:1365-1374`).

The suspension is the root problem: it deliberately lets the bag's stored grind diverge from a pinned recipe's live value, and that divergence is exactly what **Bug B1** exploits — `MainController::onShotEnded`'s post-shot bag stamp (`maincontroller.cpp` ~2942-2960) writes to the bag through a path that bypasses `SettingsDye::m_pendingSelfWrites`, so its echo is misread as an external edit, triggering `SettingsDye::applyActiveBag()` (`settings_dye.cpp:620-667`) to unconditionally reapply the bag's (deliberately stale) `grinderSetting`/`rpm` over the pinned recipe's just-edited value.

Separately, **Bug B2**: `RecipeStorage::requestRecipeForActivation` (`src/history/recipestorage.cpp:328-366`) always does a fresh, uncoordinated DB read; `MainController::applyActivatedRecipe`'s grind block (`maincontroller.cpp:1151-1164`) unconditionally reapplies whatever it finds. A same-recipe re-activation (trivially reachable since commit `6207cc11` removed the "already active" guard in `RecipesPage.qml`) can race an in-flight grind write-through to the recipe row and lose it.

**Bug A** is unrelated in mechanism but shares the same trigger moment: `SettingsBrew::setTemperatureOverride` (`settings_brew.cpp:792-801`) always sets `m_hasTemperatureOverride = true`, and nothing in the ordinary profile/recipe/shot-load flows ever calls `clearTemperatureOverride()` — so the Shot Plan's orange highlight (`ShotPlanText.qml:75-80,276-278`) is driven by coincidence rather than real user intent, contrary to what `brew-overrides`' own "Overrides cleared on profile switch" scenario already promises.

The repo owner's direction: collapse the two-live-path model to one, and keep the fix as simple as possible. A recipe's grind is a field the recipe owns (no inherit mode). The bag's stored grind is **always the most recent value dialed** — updated live, immediately, exactly like every other bag-backed field already behaves — whether the bag is active directly or because a recipe linking to it is active. There is no deferral, no suspension, and (per an earlier draft of this design that was rejected as unnecessarily complex) no need to batch grind into a post-shot write: keeping the bag always current sidesteps the staleness Bug B1 depends on without inventing a new mechanism.

## Goals / Non-Goals

**Goals:**
- Grind/rpm live in exactly two places that both update immediately, in parallel, off the same edit: the active bag (always, unconditionally) and the active recipe's own field (when a recipe is active). Neither is ever deliberately stale relative to the other, which removes the condition Bug B1 depends on.
- `hasTemperatureOverride`/`hasBrewYieldOverride` genuinely reflect "a deliberate override is active right now" and are reliably cleared, matching the existing (currently unmet) `brew-overrides` spec promise.
- New recipes still get a sensible grind/rpm starting point (the bag's last dial) without any live linkage after creation.
- The recipe wizard knows grinder rpm-capability before the user reaches the grind field.
- Existing recipes in today's inherit mode are migrated losslessly (no silent "recipe now has no grind" regression).

**Non-Goals:**
- Redesigning brew-by-ratio, yield override, or steam/hot-water blocks — untouched.
- Changing the bean-less recipe grind model — it already stores grind locally by construction and is unaffected.
- Adding a settings toggle to bring back inherit mode. The repo owner's direction is explicit: recipes always own their grind. If a future need for shared/synced grind across sibling recipes emerges, that is a new proposal.
- Any new write-coordination mechanism (deferred writes, additional self-write counters, reload gating). The simplification in Decision 2 is specifically chosen to avoid needing one.

## Decisions

### 1. Recipe grind becomes a plain, always-populated field — no inherit branch

`recipes.grind_pinned`/`rpm_pinned` stay as-is schema-wise (no migration needed for the columns themselves) but the "empty = inherit" reading is retired everywhere:
- `MainController::applyActivatedRecipe`'s grind block (`maincontroller.cpp:1151-1164`) drops the `else if (linkedBagId > 0)` bag-fallback branch — the live dye cache's grind/rpm always come from `recipe.grindPinned`/`rpmPinned` when a recipe is active.
- The live-edit-to-recipe stamp (`maincontroller.cpp:941-949`) drops its `if (!m_activeRecipe.value("grindPinned")...)` guard — it always stamps the active recipe now, since every active recipe owns its grind.
- `RecipeWizardPage.qml`'s `fGrindOverride` toggle and "Follows the bag" copy (`trInherited`, `fInheritedGrind`/`fInheritedRpm` used as a *live* mode) are removed as a mode; the same underlying bag-read machinery is repurposed as a one-time default-fill helper (Decision 4).

**Alternative considered**: keep inherit as an opt-in per-recipe choice but fix the coordination bugs properly (per-field self-write tracking instead of a shared counter, await-before-reactivate). Rejected — the repo owner wants the simpler mental model, and per-field write tracking is meaningfully more code for a feature (shared/synced grind across sibling recipes) nobody asked to keep.

### 2. Bag grind/rpm write-through stays live and becomes unconditional — no suspension, no deferral

Remove `SettingsDye::m_grindBagWriteThroughSuspended` and its setter entirely. `setDyeGrinderSetting`/`setDyeGrinderRpm` (`settings_dye.cpp:288-320`) call `writeThroughToBag` **unconditionally**, exactly like every other bag-backed dye field (dose, bean identity, etc.) already does — grind/rpm simply stop being special-cased. Whenever the live grind changes, the active bag's `grinderSetting`/`rpm` update immediately, whether the bag is active directly (no recipe) or because a recipe linking to it is active. In parallel, and independently, Decision 1's live-edit-to-recipe stamp keeps the active recipe's own `grindPinned`/`rpmPinned` current too — the same edit event fans out to both writers, neither gated on the other. The bag is thus always a live mirror of "whatever grind was last dialed," not a value deferred to shot-end; the existing "Grinder/dose correction on post-shot review" explicit-correction path (`coffee-bag-model`) is unaffected.

This fully resolves Bug B1 without inventing any new mechanism: the bug depended on the bag's stored grind being deliberately stale relative to a pinned recipe's private value at the moment a spurious resync reapplied it. With no suspension, the bag's stored grind converges to the live value within one ordinary async write — normally complete long before a shot even starts, let alone ends — so any later resync (the post-shot echo, app startup, whatever else) reapplies a value that already matches what's live. The observable "revert after the shot" symptom cannot occur. This also means an in-session grind edit is never at risk of being lost on an app restart before the next shot — the bag write-through already landed live, so there's nothing for a startup bag reload to clobber it with.

Two behavioral corollaries of the unconditional write, both intended (they follow directly from "the bag always mirrors the last dialed grind"), both spec'd in `coffee-bag-model` so they're deliberate rather than accidental:
- **Activating a recipe updates its linked bag's grind.** Activation applies the recipe's own grind to the live dial (Decision 1), and the live dial writes through — so selecting a recipe that selects a bag counts as dialing that bag. This is the repo owner's stated rule verbatim ("the bag value always gets updated when the grind changes and the bag is selected either directly or because a recipe that selects it is selected").
- **Bean-less recipe activation clears the active bag** (repo-owner decision, reversing the prior spec's "the active bag is not changed"): a recipe that carries no bean must not leave the session — grind write-through, shot bean-attribution — pointing at whatever bag happened to be laying around. With the bag cleared, `writeThroughToBag` is naturally a no-op (it already early-returns with no active bag), so no conditional write-gating is needed; the no-bag state does the right thing by construction. Implementation care: the bag-clear happens inside the activation flow, and the ingredient-swap deactivation watcher must treat "no bag" as matching a bean-less recipe's own (absent) bag ingredient so the clear doesn't self-deactivate the recipe; the `activeBagIdChanged → clearBrewOverrides()` connection firing on the clear is fine — the recipe's own yield/temp overrides apply after the bag stage, same ordering as bag-linked activation.

Note the residual uncounted writer: `onShotEnded`'s dose/yield/lastUsed bag stamp still bypasses `m_pendingSelfWrites`, so its echo still triggers one redundant `requestBag()`/`applyActiveBag()` round-trip after every shot. This is now harmless (every value it reapplies already matches the live state, so each setter early-returns) but it is wasted I/O and a residual trap for any future field with different timing. Optionally route it through a `SettingsDye`-counted helper as hygiene — cheap, not load-bearing for this change's correctness.

**Alternative considered** (drafted, then rejected as unnecessarily complex): defer all bag grind/rpm writes to a single post-shot stamp, fix `SettingsDye`'s self-write counting so that deferred write is safe, and further split `applyActiveBag()`'s reapplication logic to distinguish a genuine bag switch from a same-bag resync (since the bag would be legitimately stale between an edit and shot-end under that design, an app restart in that window would otherwise lose the edit). Reverting to always-live write-through needs none of that machinery — the bag is never in a state that needs protecting, because it's never allowed to go stale in the first place. Fewer new mechanisms, not more.

### 3. Recipe re-activation reconciles with in-flight self-writes instead of racing them (Bug B2)

When `MainController::activateRecipe(id)` is called for a recipe that is **already** the active recipe (`id == dye->activeRecipeId()`), and a grind (or other pinned-field) write for that same recipe is still in flight (`m_pendingRecipeSelfWrites > 0` for that field), `applyActivatedRecipe` must not let the fresh DB read stomp the live value. Two implementation shapes are viable; pick during implementation based on what's cleanest against `RecipeStorage`'s existing async plumbing:
  - (a) **Merge, don't overwrite**: `applyActivatedRecipe` compares the freshly-read row against `m_activeRecipe`'s already-optimistically-updated fields (which `stampActiveRecipe` already updates in-memory *before* the DB round-trip, `maincontroller.cpp:1370`) and keeps the in-memory value for any field with a write still outstanding.
  - (b) **Short-circuit same-id re-activation**: when re-activating the currently-active recipe id, skip the fresh DB read entirely and just re-push the current in-memory `m_activeRecipe` cache through the apply stages (this is what re-activation is actually trying to achieve per the #1471 fix's own reasoning — "make the live state match the cache" — the DB round-trip was never load-bearing for that goal, it just happened to also be a source of truth when nothing was in flight).

Recommend (b) as the primary approach: it's a smaller change, avoids adding field-level in-flight tracking, and directly serves the intent behind the #1471 fix (idempotent re-apply of what's already known) without the fresh-read race. Fall back to (a) only if (b) turns out to break some other caller's expectation that re-activation always re-reads (verify against `recipe-activation`'s "Activation applies the full bundle" requirement during implementation — a true cross-device/cross-session external edit to the recipe row, e.g. via MCP/web from another client, should still be picked up by *first* activation, just not spuriously re-fetched on a same-session re-tap while our own write is in flight).

### 4. New-recipe grind/rpm default-fill, not live inherit

**Recipe creation — specifically, linking a bag during creation — is the one and only moment a recipe's grind/rpm is ever read from a bag.** Nowhere else in the recipe's lifecycle does bag grind flow into a recipe: not at activation (Decision 1), not at relink to a different bag (`recipe-bag-lifecycle`), not on reopening an existing recipe for editing. Once a recipe exists, its grind is exclusively its own stored value until the user changes it directly.

`RecipeWizardPage.qml` keeps the existing bag-read logic that currently powers `fInheritedGrind`/`fInheritedRpm` (`refreshInheritedGrind`, ~lines 456-580, and the bag-selection handlers ~1114-1123, 2559-2565), but repurposes it purely as a **one-time prefill** for the grind/rpm text fields when a bag is (re)selected during creation — not as a live-follow mode with a toggle. RPM prefill only applies when `fEquipmentRpmCapable` is true (Decision 5 makes this reliably known earlier in the flow). Editing an existing recipe leaves its stored grind untouched unless the user explicitly changes the field — no re-prefill on every bag view. The migration-30 backfill (Migration Plan) is the one-time historical equivalent of this same creation-time read, applied retroactively to pre-existing inherit-mode recipes — not a new, separate read point.

The creation-time default must hold across **all** create surfaces, not just the wizard:
- **MCP/web creates**: these have no interactive field to offer a default in, so the rule moves to storage — `RecipeStorage::requestCreateRecipe` adopts the linked bag's current grind/rpm when the create map **omits** grind, mirroring the existing save-time bag-link normalization precedent in the same function (`recipestorage.cpp:399-405`). An **explicitly empty** grind in the map is stored empty (a deliberate "no grind" is respected; only omission means "use the default"). The wizard always sends an explicit value (whatever's on the field), so the storage rule never second-guesses a wizard user who cleared the field.
- **Promote-from-shot** (wizard prefill at `RecipeWizardPage.qml:427-428` and MCP `recipe_create_from_shot`): both currently store an **empty** grind when the shot has a bean — the old inherit encoding. Under the new model they default from the **shot's own recorded grind/rpm** (the exact dial that produced the shot being promoted — strictly better than the bag's since-moved dial), editable on the wizard summary before saving.

### 5. Equipment before grind in the wizard details step

Reorder the "details" step's card layout in `RecipeWizardPage.qml` so the Equipment section (currently ~lines 2245+) renders above the numbers/grind card (currently ~lines 1947-2170) — a pure layout reorder, no new wizard step, no change to the `drink → bean → profile → details → summary` step sequence (`RecipeWizardPage.qml:41`, step array ~1421-1422). The existing per-drink-type equipment default (`requestLastEquipmentForDrinkType`, ~lines 1082-1141) continues to resolve `fEquipmentRpmCapable` asynchronously as today; the reorder just guarantees the user *sees* the equipment choice (and can correct it) before they reach the field whose visibility depends on it, rather than after.

**Alternative considered**: promote equipment into its own wizard step before "details". Rejected for this change — a bigger navigational change than the bug warrants; revisit only if the in-step reorder proves insufficient (e.g. if profile filtering ever wants equipment context too).

### 6. Fixing the override-flag latch (Bug A)

`SettingsBrew::setTemperatureOverride`/`setBrewYieldOverride` (`settings_brew.cpp:792-801,823-844`) stop unconditionally setting their `has*Override` flags true. **Implementation constraint**: `SettingsBrew` is a settings-domain object with no knowledge of the loaded profile, so the "does this value differ from the profile default?" comparison cannot live inside the setters as-is. Either the setters gain a reference-value parameter (caller passes the profile default it is overriding against), or the flag decision moves to the call sites in `ProfileManager`/`MainController`, which do know the profile — pick whichever reads cleaner during implementation; the observable rule is identical. Then:
- A real `clearTemperatureOverride()`/`clearBrewYieldOverride()`-equivalent call is wired into every path that today calls `ProfileManager::clearBrewOverrides()` (`profilemanager.cpp:453-460`) — profile switch, recipe deactivation/switch — so the flags actually go false, not just the numeric value resyncing to the new default.
- `applyLoadedShotMetadata`'s re-stamp of a shot's frozen `temperatureOverride` (`maincontroller.cpp:661-748`, ~724-727) continues to set an override, but only marks it "has an override" when the frozen value genuinely differs from the freshly-loaded profile's default by more than the existing 0.1°C epsilon — matching the exact threshold `ShotPlanText.qml` already uses for the highlight, so "has an override" and "is highlighted" can never disagree.
- Remove dead code `SettingsBrew::clearAllBrewOverrides()` if nothing in this change ends up using it, or wire it in if it turns out to be the right shared clear point — implementation's call.
- Reconcile `CustomItem.qml:43-51`'s duplicate highlight rule (which also ORs in `brewByRatioActive`) with `ShotPlanText.qml`'s — either both should include brew-by-ratio or neither; pick one during implementation and note the choice in the PR description since it's a small user-visible behavior decision, not just a bug fix.

## Risks / Trade-offs

- **[Risk]** The recipe re-activation short-circuit (Decision 3b) could mask a genuine external edit (MCP/web client editing the same recipe from another session) if the user is actively editing it locally at the same moment. → **Mitigation**: this is an extremely narrow window (concurrent edit + same-session re-tap), no worse than the pre-#1471 behavior which skipped re-activation for an already-active recipe entirely; acceptable.
- **[Risk]** Migration backfill (existing inherit-mode recipes → pinned) runs once at upgrade and reads each recipe's linked bag at that moment; a recipe whose linked bag has since gone stale/finished gets whatever grind that bag last had, which may be old. → **Mitigation**: this matches the existing pattern from the bag_id migration (migration 29) — "recipes with no open bag migrated with stale data" is an accepted, documented precedent (see `docs/CLAUDE_MD/RECIPES.md`), not a new risk class.
- **[Trade-off]** This change intentionally removes a feature (grind inherit / sibling-recipe grind sync) some users may have relied on. → Accepted per explicit repo-owner direction; call out in the release notes as a deliberate simplification, not silently. (The MCP tool changes carry no compatibility risk — clients reload the tool schemas/descriptions on every connection, so the new grind semantics self-propagate; the omitted-vs-explicit-empty create rule additionally keeps the previously-recommended "omit grind" pattern doing the right thing.)

## Migration Plan

1. Ship the code changes (grind routing, write-through unconditional-ization, re-activation reconciliation, override-flag fix, wizard reorder) together — they're interdependent (e.g. the wizard can't stop offering inherit mode until activation stops supporting it).
2. New migration (30, next after the existing `recipes.bag_id` migration 29 in `src/history/shothistorystorage.cpp`): for every recipe row with an empty `grind_pinned` and a linked bag, copy that bag's current `grinderSetting`/`rpm` into `grind_pinned`/`rpm_pinned`. Skip rows whose linked bag has an empty `grinderSetting` (tea bags, never-dialed bags) — copying nothing must not write empty strings or fabricate grind for grind-less drink types. Recipes with no linked bag and empty `grind_pinned` are left as-is (nothing to copy from — already the accepted "no grind recorded yet" state, not a broken invariant). Idempotent and gated like migrations 26-29 (`>= 29 && < 30`).
3. **The same backfill must run on the device-transfer/backup import path.** `RecipeStorage::importRecipesStatic` (`recipestorage.cpp:1154`, invoked from `ShotHistoryStorage::importDatabaseStatic`) merges recipes from a source DB that may predate migration 30 — inherit-mode (empty-grind) recipes imported *after* this device's migration already ran would otherwise stay empty forever. Apply the migration-30 backfill rule to imported rows (against their bag links, after the bag id-remap) as part of the import.
4. **MCP/web contract changes are required, not merely possible** (verified against the code): `mcptools_recipes.cpp` exposes `grind.mode: "inherited"|"pinned"` on reads (~lines 125-155), `recipe_create`'s schema recommends empty-to-inherit (~lines 398, 426-429), `recipe_update` documents `''` as "return to inherit" (~lines 493-494), and `recipe_create_from_shot` describes the inherit rule (~lines 584-585); `shotserver_recipes.cpp` resolves `effectiveGrind` from the bag when `grindPinned` is empty (~line 99) and labels "(pinned)" in the web UI (~line 659). All of these need updating to the recipe-owned model (e.g. `grind.mode` collapses or is removed; `effectiveGrind` becomes the recipe's own value). No compatibility concern: MCP clients reload the tool schemas and descriptions on every connection, so the updated semantics self-propagate — this is straightforward update work (tools, descriptions, protocol-test stubs, `MCP_SERVER.md`), not a versioned-API migration.
5. No rollback path for the migration is provided beyond restoring a pre-upgrade backup — consistent with how migrations 25-29 are handled elsewhere in this codebase (one-way, best-effort, logged incomplete-and-retry on failure).
