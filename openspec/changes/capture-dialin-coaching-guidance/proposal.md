## Why

Today the AI advisor describes shots; it does not actively steer users toward good coffee. A mechanically-clean shot can still be off its profile's intent (e.g. a D-Flow pulled at ~10 bar — clean curves, no channeling, but off-guideline and not a 75), and the advisor has no per-profile expected-behavior signal to catch that or to coach the next change. The authoritative dial-in knowledge exists — but a source review (see design) shows it is **unevenly distributed**: only a small set of profiles has a true *two-sided* grind diagnostic at high (first-party / creator) confidence, while most profiles have first-party *prose* good for description but not a fault signature. There is **no structured expectation surface today**: the shipped KB (`resources/ai/profile_knowledge.md`) already resolves D-Flow-editor profiles per-profile via separate sections + `Also matches:` aliases (the #1160 correctness fix, already landed — archived `correct-dflow-variant-ugs`), but it carries no machine-readable per-profile dial-in expectation. This change **owns the entire `ProfileExpectation` seam** (struct + parser + the two-sided diagnostic grammar + active-coaching teaching), keying arms to those already-shipped per-profile sections. It supersedes the proposed-but-never-implemented `split-ugs-pressure-variants` (archived), whose envelope-first / archetype-5 / monolithic-"D-Flow-as-profile" design was falsified by the editor-vs-profile model (design D11). Scoped honestly to what the sources actually support, with the post-shot advisor (not a deterministic detector) as the first consumer. **Given this area's history of confident-but-wrong designs (#1155, monolithic-D-Flow, capped-variant, envelope-first), the rollout is staged behind pre-committed STOP gates (design D12 / tasks Phase 0→1→2): a ~1-hour zero-code spike that A/Bs one KB line on real shots before any build, then a minimal two-arm slice that is re-measured before the bulk seeding — each with the abandonment path written in.** The value hypothesis ("structured arms beat existing prose") is treated as unproven until a gate proves it.

## What Changes

- **Extend the `Expected*:` directive grammar to a two-sided diagnostic** (the deferred D9 grammar): a seeded profile may carry, alongside the centred `ExpectedPressureBar` / `ExpectedFlowMlS` / `ExpectedTimeSec`, a `GrindTooCoarse:` symptom signature, a `GrindTooFine:` symptom signature, a pressure-**shape qualifier** (`declining` / `flat` / `rising-bad` / `crash`), and a preinfusion-dripping grams figure with its own too-coarse/too-fine bounds. Each arm carries a per-arm `[SRC:...]` provenance tag and a confidence marker; **a missing arm is left absent, never fabricated to make a pair**. Every value is traceable to a cited line — **nothing invented** (the #1155 discipline). Where the symptom is stated in first-party *prose* but not as the pre-structured line (Blooming), the arm is tagged as synthesis-from-first-party-prose, not transcription.
- **Carry the per-profile suppression catalogue with the expectation** so a profile's "this is GOOD / do not flag" facts (Allongé channeling normal, Adaptive grind-tolerant by design, Blooming zero-flow bloom, LRv2 flow-control fallback, etc.) travel as machine-adjacent data, not only as free prose.
- **Two coverage tracks, scoped to the evidence (see design D-source for the audit):**
  - *Prose coverage (broad):* per-profile coaching prose for the full first-party-backed set (~15+ espresso profiles the `[SRC:*-video]` / `decent-guide` / `4mothers` sources cover). Prose tolerates directional hedging and the LLM softens it.
  - *Diagnostic-arm coverage (small, source-graded):* the structured arms are seeded **only** where a citation supports them — two-sided at high confidence for **Adaptive v2** (decent-guide + first-party video + Gagné/creator `adaptive-adastra`), **Londinium/LRv2-LRv3** (decent-guide + first-party video), **Blooming Espresso** (first-party video, synthesis-tagged; Rao's blog is philosophy-only), **D-Flow / Q** — note: *D-Flow is an editor type, not a profile; the profile is the name past the `/`* (see design D11). Two D-Flow-editor *profiles* carry the verbatim cited goal *"The goal is to grind for a pressure peak between 6 and 9 bar"* (`[SRC:profile-notes]`), each keyed to its own profile section: **`D-Flow / Q`** (Damian's + the maintainer's daily profile) — editor pressure limit **10.0 bar**, *above* the 9-bar target, so it is a **clean two-sided** arm (*too coarse* → peak < ~6 bar; *too fine* → peak > ~9 bar; unconfounded), the strongest D-Flow-editor arm and non-conditional; and **`D-Flow / La Pavoni`** — editor pressure limit **9.0 bar**, *exactly the top of the target band*, so only the **too-coarse arm is clean** (peak never reaches ~6 bar); its too-fine direction is *masked by the clamp* (a too-fine grind pegs flat at the 9.0 limit, which is the editor-clamp suppression case, not a distinct fault). `D-Flow / default` is the explicit starter/example and carries **no** stated target — do not attribute the 6–9 goal to it. The D-Flow *editor* contributes only an editor-level structural prior (`PressureShape: declining`, `[SRC:maintainer]`, video-corroborated) and a per-profile clamp **suppression** rule (pressure pegged at the resolved profile's own pressure limit is by-design, not a fault — `[SRC:de1app-dflow]`; this **replaces** the bogus "capped variant" — every D-Flow profile is clamped at its own limit). Other `D-Flow/…` profiles get the prior + their own clamp suppression and a numeric arm only if that specific profile carries a cited target; A-Flow is likewise an editor (prose track only); one honest single arm for **E61** (first-party video, too-coarse), **Allongé** (first-party video, too-fine + channeling-normal suppression), and **Turbo Shot** (EAF-only, conservative, explicitly `[SRC:eaf-profiling]`-tagged). A profile with no citation for an arm still gets no fabricated arm — absence stays absence.
- **Turn the directional teaching clause into an active-coaching instruction.** The `shotAnalysisSystemPrompt` "How to read structured fields" section gains a clause directing the advisor, when a shot's observed pressure/flow/time matches a profile's `GrindTooCoarse:` / `GrindTooFine:` signature, to *name the signature and recommend the grind direction* — while remaining directional, treating intentional in-profile variability as normal, AND-gating against the profile's suppression catalogue + channeling + unknown/very-fresh beans + grind-tolerant-by-design, and always deferring to taste (too sour → finer, too bitter → coarser outranks any curve verdict).
- **First consumer is the advisor prose on both surfaces** (in-app + `dialing_get_context`), via the existing KB-injection path — no deterministic detector, no new plumbing, no DB/storage, nothing persisted or exported to Visualizer.
- **Out of scope (unchanged):** the deterministic deviation detector that consumes the struct in code — that remains the deferred `shot-analysis-pipeline` change; this change only enriches the data + the LLM-facing prose/teaching. Trust-gating for #1159 consumes this guidance later; it is not built here.

## Capabilities

### New Capabilities
- None. This extends the data, grammar, and teaching surface owned by an existing capability.

### Modified Capabilities
- `dialing-context-payload`: extends the `ProfileExpectation` schema/parse contract from a one-sided envelope to a two-sided diagnostic + shape qualifier + preinfusion-dripping bounds + a per-profile suppression set, **with per-arm `[SRC]` provenance + confidence and an explicit no-fabricated-arm / arm-less-profile rule**; adds broad per-profile coaching *prose* coverage while scoping the structured *arms* to the evidence-graded set; and strengthens the `shotAnalysisSystemPrompt` "How to read structured fields" teaching from passive directional context into an active, taste-deferring, confounder-gated coaching instruction. UGS parsing, the RGS anchor algorithm, canonical/inferred semantics, and the no-deterministic-detector boundary are unchanged.
