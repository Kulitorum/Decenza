## Context

The shipped KB (`resources/ai/profile_knowledge.md`) already resolves D-Flow-editor profiles **per profile** — `## D-Flow`, `## D-Flow Q variant`, `## Damian's LRv2 / LRv3`, `## A-Flow`, `## Londinium` exist as separate sections with `Also matches:` alias keying (the #1160 grind-equivalence correctness fix; shipped, archived `correct-dflow-variant-ugs`). What does **not** exist is any machine-readable per-profile dial-in expectation. This change **owns the entire `ProfileExpectation` seam**: the struct + parser on `ProfileKnowledge` (parsed in `ShotSummarizer::loadProfileKnowledge()`, mirroring `ugs`/`ugsInferred`), the centred envelope, the two-sided diagnostic grammar (`GrindTooCoarse:`/`GrindTooFine:` + shape qualifier + preinfusion-dripping bounds + flow-stability), the per-profile suppression catalogue, evidence-graded seeding, and the active-coaching teaching clause — keying arms to the already-shipped per-profile sections. It **supersedes the proposed-but-never-implemented `split-ugs-pressure-variants`** (archived): that change's envelope-first / archetype-5 seeding and its monolithic "D-Flow-is-a-profile" split model were falsified by the editor-vs-profile model (D11), so it is replaced rather than extended. The deterministic detector remains deferred to a separate `shot-analysis-pipeline` change.

The motivating realization (this conversation; the #1155 inferred-rating rollback): a mechanically-clean shot is not a good shot. Detectors see channeling/truncation, not profile-intent adherence or taste. The signal that actually steers a user is the *symptom arm* ("peak below 7 bar ⇒ ground too coarse"), which is precisely what Decent's official guide leads with and what the deferred grammar omits. The constraint that makes this safe: the March-2026 ADR rejected an LLM-facing rigid min/max band fed as a rule; and #1155 proved a synthesized advisor signal that escapes to Visualizer / overwrites user state erodes trust fast.

## Goals / Non-Goals

**Goals:**
- Pull the deferred two-sided diagnostic grammar forward and make it the unit of value: per-profile `GrindTooCoarse:` / `GrindTooFine:` signatures, pressure-shape qualifier, preinfusion-dripping bounds, flow-stability qualifier.
- Carry the per-profile suppression catalogue as machine-adjacent data alongside the expectation.
- Scale seeded coverage to the full source-tagged set, then a citation-bounded long tail — boundary is "has a first-party/community citation," never invention.
- Make the post-shot advisor (in-app + `dialing_get_context`) the first consumer via the existing prose path, with an *active-coaching* teaching clause that names the matched signature and recommends a grind direction — directional, suppression-gated, taste-deferring.
- Stay additive and ADR-safe: parser growth only, loose parse contract, no struct code-consumer, no DB/storage, no Visualizer write.

**Non-Goals:**
- The deterministic deviation detector that consumes `ProfileExpectation` in code — remains the deferred `shot-analysis-pipeline` change.
- #1159 history trust-gating — a later consumer of this guidance, not built here.
- Re-doing the #1160 per-profile KB split — already shipped (archived `correct-dflow-variant-ugs`); this change keys arms to the existing sections and does not rename them.
- Persisting any coaching verdict on the shot record or syncing it to visualizer.coffee.

## Decisions

**D1 — Diagnostic-first, not envelope-first.** The seeded unit is the two-sided arm, not the centred band. Rationale: the Decent first-espresso comparison showed the centred envelope barely beats the prose the LLM already has for documented profiles, while the `GrindTooCoarse:` / `GrindTooFine:` symptom is the falsifiable, low-confounder, behavior-changing signal. *Alternative:* ship the envelope first (the superseded split-ugs envelope-first order), arms later — rejected: it front-loads the low-value half and defers the value, the exact inversion this change corrects.

**D2 — Two coverage tracks, each scoped to its evidence.** Prose coverage is broad (every first-party-backed espresso profile); diagnostic-arm coverage is *only* where a citation supports an arm. These are decoupled because they have different evidence bars and different false-positive costs: prose is hedged by the LLM, an arm is a near-verdict. *Alternative (the superseded split-ugs framing, and this change's own first draft):* "seed every profile that carries a source-tagged Official-dial-in/Grind-diagnostics line" — **rejected after the source review (D9)**: the literal repo grep shows that pre-structured line exists for only 2 profiles; "the full source-tagged set" was an overstatement for arms (true only for prose). The citation boundary (no invented numbers, no fabricated second arm) is non-negotiable — the #1160/#1155 anti-inference discipline.

**D9 — Source-review evidence grading (the audit that rescoped D2).** A review of every fetchable source in `docs/PROFILE_KNOWLEDGE_BASE.md` plus the three first-party Decent video transcripts (`docs/{light,medium,dark}_roast_profiles_transcript.txt`) established the real per-profile evidence:

Two structural findings fall out of the audit and are recorded so they are not re-derived: (a) the EAF guide *explicitly disclaims* visual/curve diagnosis ("looking at the bottomless is not likely to tell you how severe channeling is… judge by flavor") — this is not a weakness to route around but a first-class reinforcement of D4's taste-outranks rule and the deferred detector's false-negative bias; any EAF-sourced arm is conservative-only. (b) The `adaptive-search` source link in `PROFILE_KNOWLEDGE_BASE.md` returns HTTP 404 — a stale-link cleanup, in scope for the living-doc task, not a data source. *Alternative:* treat all `[SRC]`-tagged prose as arm-grade — rejected: it is exactly the inferred-on-inferred fragility #1160/#1155 exist to remove.

**D10 — A distinct `[SRC:maintainer]` provenance tier for first-hand operational knowledge.** Some structured guidance comes from the maintainer's first-hand daily operation, not a published page (e.g. the D-Flow-editor lever-decline structural prior and the "good D-Flow-editor shot lands ~5.5–9 then declines" expectation). High-trust (first-hand, controller author) but *not* a published citation, so it gets its own tier rather than being filed as `decent-guide`-grade. Rule: `[SRC:maintainer]` items are seeded like any other cited item, MUST carry the explicit tag + a confidence marker, and must be a concrete observable (a *shape* prior on the `PressureShape: declining` axis, not a raw level threshold — robust to per-profile number differences). A separate tier so future contributors and the deferred detector can tell published-corroborable from maintainer-asserted data (detector confidence gating depends on the provenance class). This does **not** loosen the no-fabrication rule — it widens *who counts as a citable source*, every value still attributable. *Alternative:* fold into `[SRC:profile-notes]` or treat as `decent-guide`-equivalent — rejected: misrepresents provenance and lets later edits inflate unpublished assertions to published-grade.

**D11 — D-Flow and A-Flow are editor *types*, not profiles (the model that corrects D9/D10).** Per `docs/CLAUDE_MD/RECIPE_PROFILES.md`, the Recipe Editor has four editor types (D-Flow `dflow`, A-Flow `aflow`, Pressure, Flow), each generating a frame structure; the editor is identified by title prefix (`D-Flow/…`, `A-Flow/…`). **The profile is the name past the `/`** — `D-Flow / default`, `D-Flow / Q`, `D-Flow / La Pavoni`, `A-Flow / …` — each a distinct profile with its *own* numeric targets (de1app `D_Flow/code.tcl` stock: default fill 3.0 / 1.7 mL/s / **8.5 bar limit** / 50 g; Q fill 6.0 / 1.8 mL/s / **10.0 bar limit** / 36 g; La Pavoni 1.2 / 2.4 mL/s / **9.0 bar limit** / 46 g). Consequences for this change:
- **Diagnostic arms attach to the *profile*, keyed by `profile_kb_id` to that profile's own KB section — never to "D-Flow."** "D-Flow" is not a diagnosable entity.
- **The D-Flow editor contributes only:** (a) a *structural prior* — soak → flow-driven pour with a per-profile pressure ceiling → lever-style decline (`PressureShape: declining`, `[SRC:maintainer]` + video-corroborated); and (b) a *suppression rule* — pressure pegging flat at *the resolved profile's own pressure limit* is by-design editor-clamp behavior, NOT a grind fault, true for every D-Flow profile at *its own* limit (`[SRC:de1app-dflow]`, new source). This **replaces and generalizes the bogus "capped Damian-advanced variant"**: there is no special capped variant — every D-Flow profile is clamped at its own limit; the video's "8.5 bar" was simply `D-Flow / default`'s limit.
- **Two cited per-profile numeric D-Flow arms: `D-Flow / Q` and `D-Flow / La Pavoni`** — both ship `notes` verbatim "grind for a pressure peak between 6 and 9 bar" (`[SRC:profile-notes]`). They differ by *clamp position*: Q's editor limit (10.0 bar) sits above the 9-bar target → cleanly **two-sided** (too coarse <6, too fine >9, unconfounded); La Pavoni's editor limit (9.0 bar) equals the target ceiling → only the **too-coarse** arm is clean, the too-fine direction is absorbed by the clamp suppression. `D-Flow / default` is the explicit starter/example with **no** stated target. Other D-Flow profiles get the editor structural prior + their own clamp suppression, and a numeric arm only if that specific profile carries a cited target.
- **A-Flow is likewise an editor** (hybrid pressure-then-flow). Any `A-Flow / …` guidance is per-profile; A-Flow currently has no cited arm (prose track only). Do not treat "A-Flow" as a profile.
*Alternative:* keep a monolithic "D-Flow" arm — rejected: it conflates distinct profiles and pressure regimes and is factually wrong about what D-Flow is.

**D10a — First-party source `[SRC:dflow-author-video]` (re-scoped under D11).** The D-Flow-editor livestream transcript (`docs/dflow_damian_profile_transcript.txt`; "Damian's D-Flow Profile", Decent Espresso, 2021-08-29; Damian + John Buckman) was added as a first-party source. Under the D11 model it: (1) **first-party-cites the editor-clamp suppression rule** — Damian: "doesn't matter if you grind too fine or too coarse, pressure won't go over the max… you still get the same shot" (this is the *editor* clamping at the profile's limit — `D-Flow / default`'s 8.5 bar in the video — not a special variant); (2) **taste-corroborates the too-fine direction** — Damian: ">9 bar gets harsh, you lose the fruity notes"; (3) corroborates the lever-decline structural prior. Prose-track first-party detail also mined: 3-bar soak; move-on-weight 0.3–0.5 g (clarity/wine) vs ~4 g (londinium thickness) vs 10–12 g (reverses); roast-graded flow/temp/dripping/ratio (dark → 88 °C, ~5 s drip, ~1.6–1.7 mL/s, ~1:2; ultralight → 94 °C, up to 30 s drip with end-of-bloom decline to 0, ~3–3.5 mL/s, ~1:2.8–3; raising flow needs coarser grind); grinder bimodal/unimodal effect on pour-stage pressure stability. ASR noise flagged in the file header; only the verbatim, unambiguous clamp + taste statements are promoted to citation.

**D10b — `D-Flow / Q` and `D-Flow / La Pavoni` are the cited per-profile D-Flow arms; their clamp position sets the arm shape.** Both `resources/profiles/d_flow_q.json` (title "D-Flow / Q") and `resources/profiles/d_flow_la_pavoni.json` (title "D-Flow / La Pavoni") ship `notes` stating verbatim *"The goal is to grind for a pressure peak between 6 and 9 bar"* — first-party stated dial-in goal shipped in the product → `[SRC:profile-notes]`, strongest provenance for a numeric bound. The decisive nuance is the **editor pressure limit relative to the 9-bar target ceiling**:
- **`D-Flow / Q`** — limit **10.0 bar** > 9-bar target → **clean two-sided** arm (too coarse → peak < ~6 bar; too fine → peak > ~9 bar, sitting *below* the 10.0 clamp so it is a genuine unconfounded signal). The cleanest, highest-value, non-conditional D-Flow-editor arm.
- **`D-Flow / La Pavoni`** — limit **9.0 bar** = the target ceiling → the too-fine direction cannot produce a peak meaningfully above the target without hitting the clamp, so it is **masked by the editor-clamp suppression**, not a fault. La Pavoni is therefore a **clean one-sided (too-coarse) cited arm** (peak never reaches ~6 bar); its fine direction is handled by the per-profile clamp suppression rule, NOT by fabricating a too-fine arm.

Both keyed to their own KB sections via `profile_kb_id`; neither inherits the other's numbers. `D-Flow / default` carries no stated target (starter/example). *Alternative:* give La Pavoni a symmetric too-fine arm like Q — rejected: its 9.0 limit == target ceiling makes a >9 peak the clamp case, so a too-fine arm there would fire on by-design clamp behavior (the exact false-positive the editor-clamp model removes). *Alternative:* treat either as `[SRC:maintainer]` — rejected: the bound is the profile author's published-in-product stated goal, a stronger class.

**Knock-on:** the sibling `profile-expectation-deviation-detector` change still records "D-Flow" as "provably arm-less, soft-note only" — superseded and *also conceptually wrong* (D-Flow is an editor). That change must be corrected to: (a) consume the `D-Flow / Q` `[SRC:profile-notes]` 6–9 peak bound as a clean two-sided, profile-keyed, **non-conditional** deterministic arm (top-tier, alongside Londinium and E61), **and** the `D-Flow / La Pavoni` `[SRC:profile-notes]` bound as a clean **one-sided (too-coarse) profile-keyed arm** (its 9.0 clamp masks the fine direction → that side is the clamp-suppression case, not an arm); (b) treat the lever-decline shape + per-profile pressure-limit clamp as an editor-level structural prior + suppression rule (keyed by editor type via title prefix), not a profile arm; (c) drop all "capped Damian-advanced variant" language.

**D3 — First consumer is advisor prose; detector stays deferred.** The enriched guidance reaches users through the existing KB-injection prose on both surfaces — no deterministic detector, no new plumbing. Rationale: this is the lowest-risk, immediate-value, ADR-safe path; it tests the right hypothesis ("does this steer users to better shots?") rather than the superseded envelope-first hypothesis ("does structuring rich profiles change LLM output?"). The struct stays code-unconsumed; the detector change owns code consumption + corpus regression.

**D4 — Teaching clause is active-coaching but discipline-split.** Prescriptive guidance ("for this profile expect ~9 bar peak, ~25s, ~4g drip; start one step finer than your last bean") may be generous. Diagnostic guidance must be symptom-specific (quote the matched `GrindTooCoarse:`/`GrindTooFine:` observable), AND-gated against the profile's suppression catalogue + channeling + unknown/very-fresh beans + grind-tolerant-by-design, and taste-deferring (too sour → finer, too bitter → coarser outranks any curve verdict). Same gating pattern the spec already uses for `tastingFeedback` / `beanFreshness`. *Alternative:* one uniform clause — rejected: prescriptive and diagnostic have different false-positive costs; a confident wrong "your good shot is wrong" is the #1155 trust-erosion failure.

**D5 — Suppression catalogue travels with the expectation.** Each seeded section's "is GOOD / do not flag" facts are recorded as adjacent directive lines, not only buried in prose, so the active-coaching clause can reference them and the deferred detector inherits a concrete per-profile suppression set (broader than the single `flow_trend_ok` flag). *Alternative:* leave suppression as free prose — rejected: it makes the gating non-systematic and forces the deferred detector to re-mine it.

**D6 — This change owns the whole seam; `split-ugs-pressure-variants` is superseded, not a predecessor.** It was proposed but never implemented (0/34), and this conversation falsified its conceptual foundation (D11: D-Flow is an editor, not a profile; its "split into pressure-distinct variant sections" framing is the monolithic error) and inverted its envelope-first/archetype-5 strategy (D1/D2). Continuing it would implement a wrong model; editing it in place would mean rewriting it against its own rationale. So it is archived as superseded and this change defines the `ProfileExpectation` struct + `Expected*:` parser + teaching clause itself — no external prerequisite, no apply-ordering dependency. The #1160 correctness fix it was conflated with already shipped separately (archived `correct-dflow-variant-ugs`), so closing split-ugs drops no bug fix. *Alternative:* heavily edit split-ugs in place — rejected: its design rationale (envelope-first, archetype-5, "D-Flow variant" sections) contradicts the corrected model throughout; a clean supersede is less confusing than an in-place rewrite that fights its own decisions.

**D7 — Hard constraint: nothing synthesized, persisted, or exported.** No computed confidence/quality score is fed to the advisor as a "treat as a hint" signal (the failed inferred-rating shape). No coaching verdict is written to the shot record or to visualizer.coffee. The only output is read-time prose on the AI surfaces. This is the durable #1155 lesson and it is a spec-level invariant, not a guideline.

**D8 — Parser is additive; parse contract is loose by construction.** This change defines the parse contract: populated when present, absent when not, malformed degrades to absent, range/pattern never collapsed to min==max. All directive lines (`ExpectedPressureBar:`, `ExpectedFlowMlS:`, `ExpectedTimeSec:`, `GrindTooCoarse:`, `GrindTooFine:`, `PressureShape:`, `PreinfusionDripG:`, `FlowStability:`, `Suppress:`) follow one prefixed-directive pattern mirroring the existing `ugs`/`ugsInferred` precedent; an un-seeded or partially-seeded section parses and behaves exactly as before this change (the parser is additive on `ProfileKnowledge`, no call-site change, no migration).

## Risks / Trade-offs

- **[Seeded data carries a transcription error]** → Every value traceable to an existing source-tagged line in `docs/PROFILE_KNOWLEDGE_BASE.md`; the seed task records the `[SRC:...]` line beside each; coverage + parse tests assert presence/shape; no invention. Coverage scales in citation-priority order so the highest-confidence data lands first.
- **[LLM derives a rigid pass/fail rule from the diagnostic tokens — the ADR failure]** → D4 clause frames diagnostics as directional, suppression-gated, taste-outranks; variable-by-design profiles carry pattern/wide values (`6-9 declining`) so the data encodes intentional variability; a spec scenario asserts the clause is present on both surfaces. Same mitigation the spec already relies on for `tastingFeedback`.
- **[Active coaching produces a confident false positive on an intentional behavior]** → AND-gate against the per-profile `Suppress:` catalogue + channeling-fired + bean-freshness-unknown + grind-tolerant-by-design before any "too coarse/fine" statement; taste always outranks. Grind-tolerant profiles carry wide patterns and a leniency note.
- **[Seam ownership / no predecessor]** → D6: this change defines the `ProfileExpectation` struct + parser itself; no external prerequisite. The #1160 per-profile KB sections it keys arms to already shipped (archived `correct-dflow-variant-ugs`), so there is no apply-ordering dependency and no dropped bug fix.
- **[Prose curation is real labor; arm set is small]** → Accepted and deliberate (D2/D9): the broad payoff is prose (citation-bounded, incremental); the high-value structured arms are a deliberately small evidence-graded set (3 two-sided + 3 single-arm), not a large probe. A future arm only lands if a future citation supports it.
- **[An arm looks one-sided / a profile looks arm-less and a contributor "completes" it]** → D9 makes absence intentional and explicit: a missing arm and an arm-less profile (D-Flow) are spec-enforced states with a no-fabrication scenario, not gaps to fill.
- **[Struct ships code-unconsumed until the detector lands]** → Not dead value: the prose twin is consumed by both AI surfaces from day one (D3); same proven additive pattern as `ugs`; parse-contract regression test prevents silent rot.

## Migration Plan

Resource/text growth (`profile_knowledge.md` broad coaching prose + extended directives/suppression on the D9 arm set with per-arm `[SRC]`+confidence tags; `PROFILE_KNOWLEDGE_BASE.md` kept in lock-step as the living seed/synthesis layer and its stale `adaptive-search` link fixed) + additive parser fields on `ProfileExpectation` + one strengthened `shotAnalysisSystemPrompt` clause + `tst_dialing_blocks` assertions. No DB migration, no storage/API/Visualizer surface, no rollout gating. The parse is additive and code-unread, so runtime behavior changes only by the LLM seeing richer prose + the strengthened clause on both surfaces. Rollback is a single revert with no state to unwind. Verification: existing Qt Test suite stays green; new assertions for the extended parse contract, the D9 arm-set seed coverage (present where cited, absent where not, no fabricated pair, D-Flow arm-less), and the active-coaching/taste-deferring/suppression-gated clause on both prompt surfaces; spot-check the rendered `dialing_get_context` KB section and the in-app advisor prompt.
