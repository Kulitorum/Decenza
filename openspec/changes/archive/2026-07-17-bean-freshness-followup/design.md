## Context

`CoffeeBag` (`src/history/coffeebagstorage.h`) already tracks a current portion's `frozenDate`/`defrostDate`, snapshots them onto every shot at save time, and feeds them into `DialingHelpers::buildBeanFreshness()` (`src/ai/dialing_helpers.h`), which flips `freshnessKnown` to `true` and switches the AI's instruction text once storage history exists. That machinery only fires for frozen beans. A bag that's never frozen — bought, opened, kept on the counter or in an airtight container — has no equivalent "this portion started aging on day X" anchor, so `freshnessKnown` stays `false` forever and the AI is told to ask about storage on every turn regardless of how much context the user has already given.

Separately, the `beanFreshness.instruction` strings and the `shotsummarizer.cpp` system-prompt guidance teach exactly one direction: "don't assume calendar age alone means staleness; count from the thaw date instead." Real dial-in sessions (see the #1032 follow-up comment) show the opposite failure mode also happens: a just-thawed or just-opened portion is under-rested and gassy, which chokes the puck and reads as "grind coarser," and this needs 3-5 days to settle. Nothing in the current instruction set teaches that.

Finally, `dialInSessions`/`bestRecentShot` (`src/ai/dialing_blocks.cpp`) only carry bean *identity* (brand/type, often hoisted to session level) for historical shots — never `frozenDate`/`defrostDate`/`openedDate`. Only the single resolved "current" shot gets a full `beanFreshness` block via `buildCurrentBeanBlock`. This means a best-rated anchor shot from a different (longer-rested) portion of the same bag is indistinguishable, in the payload, from one taken from the exact same portion as the shot being dialed in.

## Goals / Non-Goals

**Goals:**
- Give every bag — frozen or not — a "portion started aging" anchor date, so `freshnessKnown` reflects reality for the common non-freezer user too.
- Teach the AI both directions of the freshness clock: "don't assume stale from calendar age" (existing) and "a recent thaw/open may mean under-rested/gassy, not stale" (new).
- Give the AI the raw material to notice when a historical shot in `dialInSessions`/`bestRecentShot` predates the current portion's storage event, instead of the current total absence of that data.

**Non-Goals:**
- No reusable "storage container" equipment entity (no brand/model/identity like grinders or baskets get) — `storageHint` is a plain category tag on the bag, not a new equipment package type.
- No multi-portion/sub-bag data model. The existing pattern — "two bags of the same coffee, each with its own roast date / freeze state, is expected and fine" (`change-beans-dialog` spec) — continues to be how a freezer-rotation workflow is represented; this change does not introduce bag-to-bag linking.
- No explicit `freezeDate`-vs-`roastDate` gap tracking (the original #1032 proposal explicitly decided this doesn't matter enough to model).
- No change to Visualizer sync scope: `storageHint`/`openedDate` are local-only fields, same as `frozenDate`/`defrostDate` today (neither is in `touchesVisualizerFields()`).

## Decisions

**`storageHint` is a plain string enum column, not a new table.** Four fixed values (`counter`/`airtight`/`vacuum-sealed`/`fridge`) map directly onto a `QComboBox`-style dropdown and a single nullable TEXT column, following the same pattern as `roastLevel`. Considered a bitmask/int enum instead — rejected because every other CoffeeBag categorical field (`roastLevel`, `kind`) is already a plain string, and JSON serialization for MCP/Visualizer wants a human-readable string per the MCP conventions in CLAUDE.md ("use human-readable strings for enums").

> **SUPERSEDED IN PART by `fix-storage-hint-freezer-independence` (2026-07-17).** The conclusion below — "the dropdown is simply hidden while the freeze toggle is on, and `storageHint` is cleared to null when freezing" — was wrong and has been reversed. `storageHint` is the PLAN for out-of-freezer storage, orthogonal to the freezer axis rather than a competing way to express it, so it is always shown and never cleared; clearing it destroyed the plan (including MCP-written values) at the moment it mattered most. What survives is the narrower claim in this paragraph's first sentence: the enum has no `"frozen"` value and `frozenDate` remains the sole source of truth for frozen-ness. Read the current `bag-freeze-lifecycle` spec, not this paragraph's conclusion.

**`storageHint` has no "frozen" value — frozen state stays solely defined by `frozenDate` being set.** An earlier draft included a fifth `"frozen"` value and a requirement forcing it to stay in sync with the freeze toggle. Dropped on review: `isFrozen`/`fFreeze` are already purely *derived* from `frozenDate` presence today (`ChangeBeansDialog.qml:433`, `fFreeze = fFrozenDate.length > 0`; `BagCard.qml:28`, `isFrozen: ... frozenDate !== undefined && ... length > 0`) — there is no independent stored "is frozen" flag to sync against in the first place. Adding `storageHint = "frozen"` as a second, independently-settable way to say the same thing would introduce exactly the kind of two-sources-of-truth bug (`frozenDate` set but `storageHint = "counter"`, or vice versa) that doesn't exist today. The dropdown is simply hidden while the freeze toggle is on, and `storageHint` is cleared to null when freezing — no sync logic needed because there's nothing to keep in sync.

**`openedDate` is a sibling column to `frozenDate`/`defrostDate`, not a repurposing of either.** A bag could in principle be frozen AND later show an `openedDate` once thawed and moved to a counter jar — keeping the fields distinct avoids overloading `defrostDate`'s meaning ("left the freezer") with "left airtight storage." `buildBeanFreshness()`'s `known` check becomes `!frozenDate.isEmpty() || !defrostDate.isEmpty() || !openedDate.isEmpty()`, and the "age from" anchor becomes whichever of `defrostDate`/`openedDate` is most recent when both are present.

**"Mark Opened" mirrors "Thaw", and both keep the calendar picker but always default it to today rather than to the bag's existing date.** Today shipped behavior pre-sets the picker to the current `defrostDate` when one exists (`BagCard.qml:423`, `thawDatePicker.openWithDate(card.defrostDate)`) — so re-thawing a bag that already has a `defrostDate` opens on the *stale* previous value, not today, even though a fresh thaw event happening today is the overwhelmingly common case. The fix is narrower than removing the dialog: keep the picker (it's still the natural place to correct a wrong date or backdate a thaw), just always call `openWithDate("")`/pass no date so the dialog's existing "default to today" branch (`DatePickerDialog.qml:44-48`) always wins over any stored `defrostDate`. Considered making the action a single tap with no dialog at all (an earlier draft of this design) — reverted per explicit feedback: the picker itself is wanted, only the default value was wrong.

**The bag card shows the actual thaw/opened date, not just a relative day count.** Today's card renders only "Def {N}d" (`BagCard.qml:170`) — no absolute date — while the adjacent `roastDate` line already prefers the absolute date over "N days since roast" (`BagCard.qml:158-160` comment: "the actual roast date is more meaningful than days-since-roast"). This change brings the freeze/open line in line with that established preference: render the absolute date alongside the day count (e.g. "Thawed {date} ({N}d)" / "Opened {date} ({N}d)") rather than replacing one with the other, since both the exact date and the at-a-glance age are useful on the inventory card.

**Cross-portion visibility ships as a per-shot lifecycle snapshot only — no new instruction, no precomputed "discount" flag.** `shotToJson()` gains the shot's own snapshotted `frozenDate`/`defrostDate`/`openedDate` (already captured on `ShotProjection` today for frozen/defrost; extended to `openedDate`) alongside the existing identity-hoisting fields, following the same "hoist when uniform across the session, emit per-shot when it differs" discipline `dialInSessions.context` already uses. That's the entire fix. An earlier draft of this design also added a dedicated system-prompt instruction telling the AI to explicitly discount a lifecycle-mismatched shot — dropped on review: the payload already teaches the model, generally, that a per-shot field overriding the session context signals something meaningfully different about that shot (this is exactly how `grinderBrand`/`beanBrand` overrides already work), so a bespoke instruction for this one field would be the kind of narrow special-casing the `beanFreshness` design already avoids. Before this change, historical shots carried zero lifecycle data — there was nothing to reason about regardless of instructions. Exposing the raw dates is the fix; whether the model reliably acts on them without further prompting is worth checking against real transcripts before adding more machinery, not assumed up front. Considered precomputing a boolean `differentPortion` flag server-side instead of raw dates — rejected for the same reason: it repeats the "no precomputed day counts, let the AI do the subtraction" principle already established for `beanFreshness` (`dialing_helpers.h` comment: "the block still contains NO precomputed day count under any field name — the AI does the subtraction itself").

## Risks / Trade-offs

- [Risk: payload bloat] Adding lifecycle fields to every `dialInSessions` shot entry, even hoisted, adds JSON size to an already-dense payload. → Mitigation: reuse the existing hoist-to-session-context mechanism so the common case (uniform storage state across a session) costs zero extra per-shot bytes.
- [Risk: five-value enum drifts from real user workflows] A fixed `storageHint` list may not cover every real container type. → Mitigation: the field is advisory context for the AI, not a validated/required input — an unset value behaves exactly like today (falls back to date-only reasoning), so an imperfect taxonomy degrades gracefully rather than blocking.
- [Risk: instruction-text bloat further crowds the system prompt] Adding the second freshness direction still grows an already long `shotsummarizer.cpp` prompt (the cross-portion piece adds no new instruction text, per the Decisions section above). → Mitigation: fold the new guidance into the existing `beanFreshness`/`currentBean` prose blocks rather than adding a new top-level section, keeping it colocated with the rule it extends.
- [Risk: exposing the dates alone may not be enough] Without an explicit instruction, the model might not reliably notice or act on a per-shot lifecycle mismatch, even though the general "per-shot field overriding session context = meaningful difference" pattern already exists for other fields. → Mitigation: this is deliberately left as an open question rather than pre-solved with more instruction text (see Open Questions) — validate against real dial-in transcripts after shipping the data exposure before deciding whether a nudge is actually needed.

## Migration Plan

- One additive SQLite migration (next sequential migration number in `shothistorystorage.cpp`) adding nullable `storage_hint`/`opened_date` to **both** `coffee_bags` and `shots` — the bag-side columns alone are not sufficient. `frozenDate`/`defrostDate` are dedicated `shots` columns with their own INSERT (`shothistorystorage.cpp:1896`/`1957`), SELECT + `ShotRecord` field mapping (`2482`/`2542`), and device-transfer INSERT + source-column-presence index resolution for older databases (`3371`/`3412`); the two new fields need the identical plumbing or the shot-snapshot requirement in `coffee-bag-model`/`bag-freeze-lifecycle` cannot actually be satisfied. No backfill on either table (existing rows simply have both unset, same as a fresh `frozenDate`/`defrostDate`-less bag/shot today).
- `CoffeeBagStorage::importBagsStatic` (backup restore / device transfer) picks up the two new `coffee_bags` columns via the existing generic column-copy path — verify no explicit column allowlist needs updating. The `shots`-side transfer path is NOT generic (see above) and needs explicit column additions.
- No settings-migration or Visualizer-side changes required (local-only fields).
- Rollback: dropping the columns is not required for a revert — an older binary reading a DB with the extra columns ignores them; only a downgrade path that pins to an older schema-version check would need explicit handling, which is out of scope here (existing migrations don't support downgrade either).

## Open Questions

- Does exposing per-shot `frozenDate`/`defrostDate`/`openedDate` alone actually change AI behavior on the cross-portion scenario, or does it need a follow-up instruction after all? Check real dial-in transcripts post-ship (e.g. reproduce something like the #1032 follow-up's shot-937 session) before adding any prompt text for this. A cheaper pre-ship check than waiting for real transcripts: replay a synthetic lifecycle-mismatch payload (shot-937-shaped) through the actual system prompt in a one-off eval and see whether the model's response flags the mismatch, before committing to "no instruction needed" as a final answer rather than a starting bet.
