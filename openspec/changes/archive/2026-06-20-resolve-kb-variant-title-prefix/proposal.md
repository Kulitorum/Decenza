## Why

After the #1192 KB JSON restructure, `ShotSummarizer::matchProfileKey` resolves a profile title by an **exact** normalized-title lookup against `s_aliasToId`, then an editor-type default fallback. `normalizeProfileKey` only lowercases/trims/de-accents/` & `→` and ` — it does **not** split on `/` or strip a ` - <suffix>`. So a user-renamed variant of a documented recipe — `D-Flow / Q - Jeff`, `D-Flow / Q2`, `D-Flow / Q3`, `Damian's Q - decaf` — misses the exact alias (`D-Flow / Q`) and falls through to the **generic `d-flow` editor default** (UGS 0.5), losing the recipe-specific guidance the variant logically inherits (`d-flow-q-variant`: 6-bar / 84°C / UGS ~1.0). In the dialing-context path some show up as `no kbId resolvable` and contribute nothing.

The pre-restructure resolver had an order-dependent greedy `startsWith`/`contains` scan that caught this; it was **deliberately deleted** because it silently mis-resolved real profiles, and the new schema's `alsoMatches` is exact-only with no family/prefix concept. The existing `profile-knowledge-base` spec already carries a `D-Flow / Q - Jeff` scenario asserting it resolves to the D-Flow-Q variant id — a contract shipped behavior does not satisfy. This change closes that gap with a **deterministic, anchored, validator-and-test-gated** rule that is explicitly *not* the deleted greedy best-guess.

Crucial framing (confirmed during exploration): **editors are namespaces, not recipes.** `D-Flow`/`A-Flow` are the `defaultForEditorType` entries; the KB author's own `ugs.note` says *"D-Flow is the editor, not a profile."* The recipe identity is the segment *after* `editor /` (`Q`, `La Pavoni`, `default`). The editor namespace's "which editor" job is already done by the explicit editor-default fallback; recipe inheritance must anchor on the **recipe** alias, never the bare editor name.

Resolution policy (user-stated): **every built-in/shipped profile SHALL resolve by exact match** (they are all registered aliases — they never depend on the prefix rule). The recipe-alias prefix step is *exclusively* the **best-effort** path for **user-created profiles derived from a real profile** — the user kept the recipe name as a prefix (`D-Flow / Q - <my bean>`) and should inherit that recipe's dialing knowledge. The "save a profile" page of the user Manual SHALL document this convention so users know that keeping the source recipe's name as a prefix is what makes the AI inherit its knowledge.

## What Changes

- Add a deterministic **recipe-alias longest-boundary-prefix** resolution step to `matchProfileKey`, *between* exact-match (unchanged, still wins first) and the editor-type default fallback (unchanged, still last). Share the same step with `resolveKbInput` so legacy persisted variant-title kbIds on old shot records heal under the recompute-on-load contract (user decision: one shared resolver).
- **Rule (profile-general — not D-Flow/A-Flow-specific):** among registered aliases that do **not** belong to a `defaultForEditorType` entry (*recipe aliases* — the aliases of **any** documented profile: `Adaptive v2`, `Londinium`, `Allongé`, `Blooming Espresso`, `Cremina`, `80's Espresso`, the D-Flow/Q & La Pavoni variants, …), select those that are a *boundary-prefix* of the normalized title — the alias immediately followed by a separator ∈ { `/`, `-`, space, ASCII digit }. Resolve to the **longest** such alias's `id`. No match → fall through unchanged. D-Flow/Q is merely the motivating case in the issue; the same rule resolves `Adaptive v2 - Jeff` → `adaptive-v2`, `Londinium - Jeff` → `londinium`, etc. (verified by simulation against the shipped KB — 130 documented-profile aliases anchor, only the two editor entries are excluded).
- The issue's literal "reject if >1 distinct id matches" is **not** adopted: `D-Flow / Q - Jeff` boundary-prefixes both `d-flow` and `d-flow / q`, so reject-on-multiple would reject the primary target case. **Longest-wins** is the correct disambiguation; a true equal-length tie is impossible by construction (one prefix per length ⇒ equal length ⇒ identical alias ⇒ the validator's existing alias-collision rejection).
- Editor-name aliases never anchor a prefix (that is the exact arbitrary-substring footgun the restructure removed); the editor namespace stays resolved solely by the explicit `defaultForEditorType` fallback. **Prefix only, never `contains`/substring** — explicit non-goal.
- `tools/validate_kb.py`: add a **non-fatal informational lint** when one recipe alias boundary-prefixes another recipe alias mapping to a different `id` (specificity ordering is legitimate — author awareness, not a hard rule, consistent with the D9-lint precedent). No new hard validator rule is required.
- Tests: positive — both the D-Flow cluster (`D-Flow / Q - <any>`, `D-Flow / Q2`, `D-Flow / Q3`, `D-Flow / Q-Jeff`, `Damian's Q - decaf` → `d-flow-q-variant`) **and** non-D-Flow profiles proving generality (`Adaptive v2 - Jeff` → `adaptive-v2`, `Londinium - Jeff` → `londinium`, `Allongé - decaf` → `allonge`), relational (`D-Flow / La Pavoni 80s` → la-pavoni variant; strictly coarser than default), built-in invariant (every shipped/starter profile title resolves by step-1 exact match, never depending on the prefix step), negative (`D-Flow / Quark`, `D-FlowX` letter-blocked → no q-variant; fully-custom `My Morning Pull` → editor default unchanged). **Invert** the existing `tst_dialing_blocks.cpp::calibrationBlock_unresolvedCustomTitleEmittedAsRawHistoryRow`, which pins the *pre-#1198* contract (`D-Flow / Q - Jeff` stays an unresolved raw history row): the variant now resolves and its shots correctly aggregate into the `d-flow-q-variant` history group — the issue explicitly anticipates this dialing_blocks re-evaluation.
- **Documentation**: update the "save a profile" section of the user Manual wiki (`https://github.com/Kulitorum/Decenza/wiki/Manual`, separate repo `Kulitorum/Decenza.wiki.git`) to explain that built-in profiles are always recognized exactly, and a profile you create from a real recipe will inherit that recipe's AI dialing knowledge **as long as you keep the recipe's name as the start of your profile's title** (e.g. `D-Flow / Q - My Ethiopian`); fully-renamed custom profiles fall back to the generic editor default.

## Capabilities

### New Capabilities
- None.

### Modified Capabilities
- `profile-knowledge-base`: the resolver requirement gains a deterministic recipe-alias longest-boundary-prefix step. It remains exact-match-first and explicitly-unresolved-on-total-miss; the new step is anchored, total, and validator/test-gated — categorically distinct from the deleted order-dependent greedy best-guess, which stays prohibited. The `D-Flow / Q - Jeff` scenario now resolves via this rule rather than requiring a hand-registered full alias, and `resolveKbInput` shares it.

## Impact

- **Code**: `src/ai/shotsummarizer_kb.cpp` — one shared recipe-prefix helper; `matchProfileKey` step 2; `resolveKbInput` routes legacy titles through the same helper. **Validator**: `tools/validate_kb.py` — one non-fatal informational lint. **Tests**: `tests/tst_dialing_blocks.cpp` (new resolution fixtures + the built-in-exact-match invariant + invert the one pre-#1198 unresolved-custom-title test); possibly `tests/tst_shotsummarizer.cpp`. **Docs**: the `Kulitorum/Decenza.wiki` "save a profile" Manual page (separate repo — clone `Kulitorum/Decenza.wiki.git`).
- **Data**: none. No KB content change — the `D-Flow / Q` / `Damian's Q` recipe aliases already exist; the rule generalizes them to suffixed/numbered variants.
- **Behavior change (intended, issue-sanctioned)**: renamed/numbered variants of documented recipes again inherit the parent recipe's band/UGS/analysisFlags in shot analysis, AI dial-in context, and grinder-calibration grouping; previously-"unresolved" `D-Flow / Q - Jeff`-class shots now aggregate into the parent id group. No regression to exact-match resolution; undocumented fully-custom titles still fall to the editor default exactly as before. No DB migration, no schema/parser/grammar change. Rollback is a single revert.
- **Lineage**: the agreed follow-up to the dialing_blocks identity-grouping fix (#1199, merged). Issue #1198. Plain references, no auto-close.
