## Why

The no-AI Shot Summary is, for whole families of profiles, **silent on the axis experts actually dial on**. A code trace of `ShotAnalysis::analyzeShot` (recorded as the closing finding of `capture-dialin-coaching-guidance` design D13/D13a/D13b/D13c) established:

- **Pressure-defined profiles**: D-Flow/Q is dialed to keep the *pressure peak* in 6–9 bar (`[SRC:profile-notes]`); Adaptive v2 to 8–9 / under-7-is-too-coarse (`[SRC:decent-guide]`); A-Flow to 6–9 (`[SRC:aflow-repo]`); Londinium ~9 declining (`[SRC:decent-guide]`). The cascade has **no expert-band concept** for any of them — the recommended band is dial-in knowledge, not in the code.
- **Flow-defined profiles**: Rao/Blooming Allongé should *reach ~4.5 ml/s* (too fine → can't hit it / machine maxes out, `[SRC:light-video]`); E61 ~4 ml/s reads too-coarse in context (`[SRC:dark-video]`); D-Flow pour flow is roast-graded (`[SRC:dflow-author-video]`). The cascade already compares observed flow to the profile's **own commanded flow setpoint** (`analyzeFlowVsGoal`) — the profile-*intrinsic* reference — but has **no concept of the expert-recommended flow target** a source states, which is a different thing.
- D-Flow's flow-trend signal is additionally suppressed (`flow_trend_ok`). So on a normal D-Flow shot the no-AI summary says only some mix of *"Puck stable / Preinfusion X g in Y s / Could not analyze grind — check taste"*, while the recommended-band axis goes unmentioned.

An empirical probe (capture-dialin-coaching-guidance D13b: real shot, April 6 10:23 AM, peak 10.08 bar, user note *"Slightly bitter, pressure hit limiter (10 bar)"*) showed running **outside the expert band is a real, tasteable problem**, corroborated on the primary source (`docs/dflow_damian_profile_transcript.txt`: the author keeps the shot under ~9 bar because *">9 bar gets harsh"*). The firmware limiter is **not the subject** — it is a safety tool; a shot *hitting* it is only a corroborating symptom that something went wrong, never the reference.

This change surfaces one soft, observational, taste-deferring Shot Summary line when a shot runs **outside the operating band an expert source recommends for that specific profile** — on whichever axis that source states (pressure peak *or* extraction flow). Deterministic, AI-free, the band taken verbatim from a cited source per profile (no invention), consuming none of the deferred `ProfileExpectation` seam.

## What Changes

- **Emit one soft Shot Summary line when the shot is outside the profile's expert-recommended band, on the axis the source states.** A profile's classification carries an **axis** (`pressure-peak` or `extraction-flow`) and a **band**, both verbatim from a cited source. The line is **observational and taste-deferring, never a grind-direction verdict** — band-vs-actual is a confounded signal (capture-dialin-coaching-guidance D13/D13a: a coarser-grind shot can still taste bitter; the EAF guide explicitly warns curve diagnosis is unreliable; a deterministic verdict here is the #1155 failure). It says *"this shot's {peak pressure | extraction flow} ({X}) ran outside the band experts recommend for this profile ({Y–Z}, [cited]) — judge by taste."*
- **The firmware limiter is a corroborating symptom, not the anchor.** When a pressure-axis shot also pegged the machine limiter, the line MAY add *"…and it pegged the machine's safety limiter, which usually means something went wrong"* to raise salience. The limiter value is never the band; the line fires with no limiter hit; a limiter touch with the value still inside the recommended band does NOT fire.
- **Distinct from the existing intrinsic flow check.** `analyzeFlowVsGoal` already compares flow to the profile's *commanded* setpoint and is unchanged. This adds the orthogonal *expert-recommended* band layer (e.g. Allongé "should reach ~4.5 ml/s" is an expert target, not the frame's flow command).
- **A citation-graded band table — the only curated data this change adds**, reusing the evidence grading already done in `capture-dialin-coaching-guidance` design D9/D10/D10b (authoritative; not re-derived). Each entry = profile (resolved by canonical KB-section identity, D14) → axis → band → `[SRC:...]` → confidence. Seed set:
  - *pressure-peak*: **D-Flow / Q ≡ Damian's Q** & **D-Flow / La Pavoni** — 6–9 `[SRC:profile-notes]`; **A-Flow (all variants)** — 6–9 `[SRC:aflow-repo]`; **Adaptive v2** — 8–9 / <7 too coarse `[SRC:decent-guide]`; **Londinium** — ~9 declining `[SRC:decent-guide]`.
  - *extraction-flow*: **Rao Allongé / Blooming Allongé** — reach ~4.5 ml/s `[SRC:light-video]`; **E61** — ~4 ml/s in context `[SRC:dark-video]`.
  - citation-graded long tail (Blooming Espresso, Turbo, **lever profiles with a cited expert peak** — low-pressure-6-bar / two-spring-9-bar / "gentler 8.4 bar" families) seeded only where a source states a band; absence stays absence.
- **Self-classifying, citation-bound discriminator so additional lever/profiles auto-qualify**: a profile gets a line iff a cited source states a recommended pressure-peak or extraction-flow band for it. A profile whose own shipped `notes` state one auto-qualifies with zero curation. No cited band → **no line**, never a fabricated band or axis.
- **Reuse the existing one-place cascade, recomputed every open.** The line joins the `summaryLines` list `ShotAnalysisDialog.qml` already renders, as a low-authority `[observation]`/`[caution]` entry — no new badge column, no chip, no synthesized score, nothing persisted or exported. It follows the **same recompute-on-load contract as the badges**: only profile identity is persisted; the band table, margin, and gate are read fresh from shipped code/data on every open, so correcting/expanding the table or retuning the margin **retroactively improves every historical shot on its next open** (old shots always show the current best detection). Hard AND-gate (D8): silent if pour-truncated or channeling fired (the curve-knowable confounders `analyzeShot` can see); taste always outranks. Bean freshness is out of scope here — `analyzeShot` has no freshness input; freshness suppression stays the advisor-prose layer's job, unchanged.
- **Make the guidance discoverable: a quiet entry-affordance cue.** The expert-band line (and every other non-fault observation) is useless if nobody opens the dialog. The `QualityBadges` affordance that emits `summaryRequested()` (in `ShotDetailPage` + `PostShotReviewPage`) gains a **single calm tint whenever `DetectorResults::verdictCategory` is not `clean`** (the already-computed, already-serialized verdict): `clean` → current untinted appearance; **any non-`clean` verdict → the tint** ("worth opening"). It is **not** severity-graded into a red alarm: a deliberately-pulled experimental shot routinely yields a non-`clean` verdict, and an alarming entry point there is the same #1155 / over-claim failure D3/D11 guard against — the tint means "look", not "this shot is bad". It adds no judgment (pure function of the existing verdict, so it cannot false-fire beyond what the shipped detectors already decided) and works for all profiles today, gaining the expert-band signal for free because the band line flows through the same `verdictCategory`.

## Capabilities

### New Capabilities
- None. This adds one detector branch to the existing analysis pipeline.

### Modified Capabilities
- `shot-analysis-pipeline`: (a) a profile-aware expert-band check is added inside the single `ShotAnalysis::analyzeShot` cascade — it compares the shot's peak pressure *or* extraction flow (per the profile's cited axis) against a cited per-profile band and, when outside it, emits one soft, observational, taste-deferring `summaryLines` entry (optionally noting a coincident limiter hit on the pressure axis); and (b) the already-computed `verdictCategory` is surfaced on the Shot Summary entry affordance as a single calm tint applied whenever the verdict is not `clean`. The cascade-in-one-place, recompute-on-load, render-from-`summaryLines`, four-boolean-badge-projection, and "taste outranks / no synthesized score / nothing persisted or exported" invariants are preserved unchanged; no new badge column; `analyzeFlowVsGoal` (commanded-flow check) is untouched; the cue derives purely from the existing verdict and adds no judgment.
