## Context

`ShotAnalysis::analyzeShot` is the single no-AI cascade producing every Shot Summary observation: five profile-agnostic detectors → `summaryLines` (rendered by `ShotAnalysisDialog.qml`) + four boolean badges. A code trace (capture-dialin-coaching-guidance design D13c) established that none of it is aware of the *expert-recommended operating band* a profile is dialed against. That band may be a **pressure peak** (D-Flow "6–9 bar", author-stated; Adaptive v2 "8–9 / <7 too coarse", Decent guide) or an **extraction flow** (Rao Allongé "reach ~4.5 ml/s", light-roast video; E61 "~4 ml/s" in context). The cascade *does* already compare flow to the profile's own *commanded* setpoint (`analyzeFlowVsGoal`) — that intrinsic check is unchanged here and is orthogonal to the *expert-recommended* band, which is a separately-cited target. The full evidence chain, the empirical advisor probes, and the citation-graded per-profile band audit live in `capture-dialin-coaching-guidance` (design D9/D10/D10b/D13/D13a/D13b/D13c) and are treated here as authoritative inputs, not re-derived.

This change adds exactly one signal: *the shot ran outside the band an expert source recommends for this profile, on the axis that source states (pressure peak or extraction flow).* It supersedes `capture-dialin-coaching-guidance` (advisor rationale empirically flattened; sprawling seam unjustified) and replaces the scope of `profile-expectation-deviation-detector` (depends on that seam; would inherit a mis-specified blanket clamp-suppression).

## Goals / Non-Goals

**Goals:**
- One soft, observational, taste-deferring Shot Summary line when the observed peak pressure *or* extraction flow is outside the cited recommended band for the resolved profile, on the cited axis.
- Band + axis verbatim from a cited source, per profile, confidence-tagged — reusing the capture-dialin-coaching-guidance D9 grading. No invented band or axis.
- Self-classifying and citation-bound so additional lever/profiles auto-qualify the moment a source states a band (the explicit "more levers benefit" goal), zero invention.
- Seam-free: no `ProfileExpectation`, no `Expected*:`/`GrindTooCoarse:` grammar, no broad prose. Cited axis+band + the shot's observed peak/flow only.
- Backwards-compatible: absent classification → strict no-op, byte-identical to today for every legacy/imported/test caller.
- An objective shadow-validation gate over a labeled corpus before the line is ever shown.

**Non-Goals:**
- No grind-direction verdict. Band-vs-actual is confounded; the line never says "grind coarser/finer."
- No change to `analyzeFlowVsGoal` (the commanded-flow intrinsic check) or to any existing detector.
- No new badge/chip; no change to the four-badge projection.
- No LLM involvement; the advisor incidentally reads the same `summaryLines`.
- No persistence beyond the existing recompute-on-load line contract; no Visualizer write; no synthesized score.
- Not re-deriving the per-profile band/axis evidence — that is capture-dialin-coaching-guidance D9/D10's job; this change consumes it.

## Decisions

**D1 — The expert-recommended band is the anchor; the firmware limiter is only a corroborating symptom.** The validated reference is the per-profile expert-stated band, not the machine's pressure limiter. The limiter is a safety tool ("keeps things from being terrible"); a shot *hitting* it is a useful sign something went wrong, so when a pressure-axis peak is both outside the band AND pegged the limiter the line MAY append a clause to raise salience — but the limiter value is never the band, the line fires with no limiter present, and a limiter touch with the peak still inside the band does NOT fire. *Alternative (rejected):* anchor on the profile's own `lim:` (an earlier draft of this change) — rejected: it scoped the feature to ~4 limiter-bearing D-Flow profiles and conflated a safety cap with an expert target. **Implementation note (Phase A): the optional corroborating limiter clause is deferred.** It needs the resolved profile's flow-frame limit value to detect "pegged the limiter"; `analyzeShot` is the deterministic curve cascade with no profile-limiter input, and a curve-only "pegged" heuristic would be the fragile-guess pattern CLAUDE.md forbids. The spec wording is "MAY append", so Phase A ships the band line **without** the clause (fully spec-compliant and behaviorally complete — the line is gated purely on the band, never on the limiter). Threading the profile limiter as a small caller-resolved input (the `analysisFlags`/`expertBand` pattern) is a clean later-phase addition; same architectural reason as the D8 freshness scoping.

**D2 — The axis is per-profile and source-stated (pressure peak OR extraction flow).** Pressure-defined profiles cite a pressure-peak band; flow-defined profiles cite an extraction-flow band. The classification carries the axis; the cascade already has both observables (peak pressure on the pour-truncated path; extraction flow via `analyzeFlowVsGoal`), so no new computation. A profile is never given an axis a source did not state. *Alternative (rejected):* pressure-only — rejected: flow-defined profiles (Allongé, E61) have equally cited expert flow targets and are silent today.

**D3 — Observational and taste-deferring, never a grind verdict.** Band-vs-actual is the confounded class (capture-dialin-coaching-guidance D13/D13a: a coarser-grind shot tasted bitter; the LLM reasoned around it, a mechanical verdict misfires; the EAF guide explicitly disclaims curve diagnosis — axis-agnostic). A deterministic "your good shot is wrong / grind X" here is the #1155 trust-erosion failure on the highest-visibility surface. The line states the observation and the cited band and defers to taste; never a direction. This discipline is *what makes band-adherence safe to do deterministically at all*.

**D4 — Citation-graded axis+band table, reusing D9; no invention.** Each entry = profile (resolved by canonical KB-section identity, D14) → axis → band → `[SRC:...]` → confidence, seeded only where capture-dialin-coaching-guidance D9/D10/D10b grades a cited band. A profile with no cited band gets **no line**. *Alternative (rejected):* infer a band from frame setpoints — that is invention; the user's hard rule is cited bands only, and the frame setpoint is already covered by the orthogonal `analyzeFlowVsGoal`.

**D5 — Self-classifying / auto-extending.** The rule is "a cited source states a recommended band for this profile." A profile whose own shipped `notes` state a band qualifies automatically with zero curation; any future profile/lever does too the moment a source states one. Delivers the "more levers benefit" goal structurally rather than by a frozen list, and keeps absence intentional (no fabricated band/axis to "complete" a profile).

**D6 — Adaptive v2 is a normal cited-band entry, not a special suppression case.** Under the demoted-limiter framing its grind-adaptiveness is no longer a reason to suppress; it is simply *why D3's soft wording matters* (it legitimately wanders within its adaptation). Its cited band (8–9 / <7, `[SRC:decent-guide]`) is used observationally like any other. This dissolves the special-casing an earlier limiter-anchored draft needed. **Cross-reference (note for the implementer):** the predecessor `capture-dialin-coaching-guidance` D13c records Adaptive v2 as a *rail-**suppress*** case (the limiter-anchored model: it rides toward the ceiling by design, so a rail there must not fire). This change deliberately supersedes that treatment — under the band-anchored model the same `[SRC:decent-guide]` fact (8–9 / <7) is a *fire*-cited observational band, with the grind-adaptiveness handled by D3's soft, taste-deferring wording instead of by suppression. Same source, intentionally different role; do not carry the predecessor's suppress treatment forward.

**D7 — Lives in the one-place cascade; recomputed every open from current code+table; absent classification = strict no-op.** The check runs inside `analyzeShot`, so it inherits the badge **recompute-on-load contract**: it is recomputed every time a shot is opened (save / recompute-on-load / detail-load), not frozen at save. Crucially, the **only** thing persisted on the shot is profile identity (already persisted, already used to resolve profile KB); the citation-graded band table, the firing margin, and the gate logic are resolved **fresh from shipped code/data at every compute**, never snapshotted onto the shot record. Consequence (the explicit goal): correcting a band, adding a profile, or tuning the margin **retroactively improves every historical shot on its next open** — looking at an old shot always shows the current best detection, exactly like the existing badges. This is the *same coin* as D10 (nothing persisted): because no band/verdict value is frozen, the recomputed result is always authoritative and the serialized copy is a non-authoritative cache the recompute refreshes. Absent classification leaves the result byte-identical to today (same backwards-compatible pattern as `expectedFrameCount`); reuses the already-computed peak/flow; no new heavy parameter; no cascade fork.

**D8 — Hard AND-gate; bias to silence; freshness is out of scope.** Fire only if: a cited band exists for the resolved profile AND the observed value on the cited axis is outside it with margin AND the cascade did not already fire pour-truncated/channeling. Anything ambiguous → silent. Found during apply: `ShotAnalysis::analyzeShot` is the deterministic *curve* cascade — its inputs are pressure/flow/weight/phases/goals/flags; it has **no bean-freshness input** (freshness is roast-date metadata known only to the advisor/dialing layer). The pour-truncated and channeling confounders are curve-knowable and stay hard gates here; **bean freshness is deliberately not part of this work** — it cannot be evaluated in the deterministic emitter and freshness suppression already lives, unchanged, in the advisor-prose layer (`shotAnalysisSystemPrompt`, which sees the freshness block and the same `summaryLines` entry). The deterministic line is observational + taste-deferring, so "could be the beans" is absorbed by wording, not a structural gate it cannot compute. *Alternative (rejected):* thread bean freshness into `analyzeShot` as a new param — rejected: injects non-curve metadata into the curve analyzer (layering smell) and the save/serialize call-sites lack roast date; freshness gating correctly stays the advisor layer's responsibility.

**D9 — Shadow-validation gate before the line surfaces.** A `shot_eval` dry-run computes the line over `tests/data/shots/` without rendering; the known-good false-positive rate is measured and the margin tuned conservative before enabling. Same objective-offline-gate discipline as the project's prior changes (capture-dialin-coaching-guidance D12 culture).

**D10 — Hard invariant: nothing synthesized, persisted, or exported.** No confidence/quality score is fed anywhere as a hint; the only output is one prose line through the existing line contract; nothing new is written to the shot record, a column, or visualizer.coffee. The durable #1155 lesson, restated as a spec invariant for this surface.

**D11 — The expert-band line MUST NOT toggle the grind badge (or any badge).** The four-boolean projection (`decenza::deriveBadgesFromAnalysis`) stays driven solely by the existing *intrinsic crisp* detectors (`detectGrindIssue`/`GrindCheck`: choked puck, gusher, yield-overshoot, flow-vs-*commanded*-goal). The expert-band signal is the *confounded* class (D3) — a binary persisted verdict on a confounded, taste-overridable, possibly-intentional signal, on the highest-authority no-dialogue surface, is exactly the #1155 failure, and the sibling `profile-expectation-deviation-detector` D1 already rejected a 6th badge for this reason. Folding it into the existing grind badge would also corrupt that badge's meaning ("the puck physically misbehaved") with a different epistemic category ("outside what an expert recommends"). Key consequence: when the underlying fault is *real* (true gusher/choke) the grind badge is **already on via the intrinsic path**, and the band line renders as corroboration beside it *for free* — the band signal never needs to, and never does, flip a persisted bit. Graduated visibility for a band-miss is the severity choice (`[caution]` vs `[observation]`) + the limiter corroboration clause, not a badge. *Alternative (rejected):* let a band-miss set the grind badge — rejected on #1155 grounds and because it violates D10 (a badge column is persistence) and the recompute-on-load badge contract. A spec scenario asserts the four-badge projection is byte-identical with vs without the band line.

**D12 — A quiet entry-affordance cue is part of this change (the delivery for all the guidance), not a separate one.** The expert-band line and every non-fault observation are invisible if nobody opens the dialog; the cue is what makes the guardrail work actually get seen, so it belongs here. The `QualityBadges` affordance (emits `summaryRequested()`) gains a **single calm tint applied whenever `verdictCategory != "clean"`** (the already-computed, already-serialized verdict): `clean` → current untinted appearance; any non-`clean` verdict → the tint ("worth opening"). Binary and literal by the user's instruction — not a graded bucket. (One edge noted in Open Questions: `insufficientData` also tints under this rule; accepted as "not a clean shot, worth a glance".) It is a **pure function of the existing verdict** — it introduces no new threshold, score, or classification, so it cannot false-fire beyond what the shipped detectors already decided, and the band line reaches it for free via the same `verdictCategory`. It is deliberately **not** severity-graded into a red alarm: a deliberately-pulled experimental shot routinely yields a non-`clean` verdict, and an alarming entry point there is the same #1155 / over-claim failure D3/D11 (and `capture-dialin-coaching-guidance` D13b) are disciplined against — the cue means "look", not "this shot is bad". Plumb: surface `verdictCategory` on `shotData` exactly as the four fault flags already reach `QualityBadges`; add the accent in `QualityBadges.qml`; no C++ analysis change, nothing newly persisted (it only *reads* an already-serialized field). *Alternative (rejected):* a standalone change for the cue — rejected: it is the user-facing delivery of this change's guidance and shares its single source of truth (`verdictCategory`) and its #1155 discipline; splitting it orphans the payoff from the work. *Alternative (rejected):* graded green/amber/red — rejected per D1/D3/D11 (a confident verdict on an often-intentional signal at the highest-visibility surface).

**D13 — The band line keeps `type: observation`; a dedicated `verdictCategory` value (`expertBandDeviation`) makes it discoverable, not a `caution`.** Found during apply: `analyzeShot`'s verdict cascade computes `verdictCategory` only from line `type ∈ {warning, caution}` + the detector booleans, so an `observation`-only band finding on an otherwise-clean shot would resolve to `clean` and the D12 tint would never fire — defeating the discoverability purpose. Resolution: the band line's `summaryLines` `type` stays **`observation`** (lowest authority — D2/D3/D11 posture preserved, it is not a verdict), and a **new `verdictCategory` value `expertBandDeviation`** is set by a dedicated branch in the verdict cascade, ordered **below every real-fault and `hasWarning`/`hasCaution` verdict** (a true fault always dominates and the band line is then merely a corroborating summary line) and **above `cleanGrindNotAnalyzable`/`clean`** (a band-only otherwise-clean shot becomes `expertBandDeviation`, which is `!= "clean"`, so the D12 tint surfaces it). Its verdict-line text is non-directional and taste-deferring (e.g. "Ran outside this profile's expert-recommended band — judge by taste"). *Alternative (rejected): emit the line as `type: caution`.* Zero verdict-logic change and the tint would fire — but `caution` is louder than the deliberately-lowest-authority posture D2/D3/D11 fought for, and it routes through the `hasCaution` branch whose verdict *text* names a grind direction whenever the independent grind detector is off, risking a directional verdict adjacent to our deliberately non-directional line. The new value is **not a badge**: `verdictCategory` is not one of the four booleans, so `deriveBadgesFromAnalysis` stays byte-identical (D10/D11 intact); the only consumers of the new value are the verdict text and the D12 tint. This refines D12: the *tint* remains a pure function of `verdictCategory`; the one new classification lives in the band check (this decision), not in the tint.

**D14 — Keyed by canonical KB-section identity, not by profile-title kbId (KB prerequisite #1175 landed).** The band table is keyed by the resolved profile's **canonical `ProfileKnowledge` identity** (`ShotSummarizer::canonicalNameForKbId(profileKbId)`), the same dedup-by-`name` precedent `allKbUgsEntries()`/`ugsForKbId` already use — *not* by the raw normalized title kbId. Found during apply: `matchProfileKey` returns the matched *title* key (`"d-flow / q"`, `"damian's q"`, `"d-flow / la pavoni"`, `"d-flow / default"`), and several distinct titles resolve to one `ProfileKnowledge`. Keying by title would force duplicate entries for the D-Flow/Q ≡ Damian's Q twin; keying by canonical identity collapses them structurally (both alias `## D-Flow Q variant` → one canonical name → one entry, true zero-duplication) while keeping distinctly-sectioned profiles distinct. This required `D-Flow / La Pavoni` to have its **own** KB section (it was an `Also matches:` alias of `## D-Flow` default, so canonical keying alone could not separate La Pavoni from default). That root-cause KB-correctness fix shipped separately as **#1175 (`split-dflow-la-pavoni-kb-section`, merged)** — the deliberate alternative to an Option-A "two title keys → one record" keying hack, which would have papered over the data-model bug (#1160/#1155 lineage). With #1175 landed, canonical keying now cleanly yields: `D-Flow / Q` ≡ `Damian's Q` → one entry (`## D-Flow Q variant`); `D-Flow / La Pavoni` → its own entry (`## D-Flow La Pavoni variant`); `D-Flow / default` → no entry (`## D-Flow`, no cited band). Phase A's gold pair is therefore both the Q-twin **and** La Pavoni, with zero special-case keying logic. *Alternative (rejected):* key by title with duplicate twin rows / an Option-A behavioral-collapse — rejected: encodes a workaround for a KB modeling bug; fixed the model instead (#1175).

**D14a — The band re-resolves canonical identity *fresh* at recompute, never off the persisted `profileKbId` (defect found by the live A6-precursor check).** A live spot-check on real shot 819 (the flagship clamp case: `profileName "D-Flow / Q"`, pegged ~10 bar, bitter) wrongly produced "Clean shot. Puck held well." with no band line. Root cause: `profileKbId` is *persisted at save time*; a shot saved before a KB reorganization (#1160/#1175) carries a **stale** stored kbId (819's is `"d-flow / default"`, not `"d-flow / q"`). Resolving the band off that stored value silently mis-keyed it to the band-less `## D-Flow` section — so "recompute-on-load" refreshed the *analysis* but against a *stale identity*, breaking the D7/D14 retroactive-improvement promise for exactly the historical shots the change is meant to help. Fix (band-scoped, in `prepareAnalysisInputs`): re-resolve via `ShotSummarizer::computeProfileKbId(profileTitle, editorType)` against the **current** KB — mirroring the save-time call, run now — falling back to the stored kbId only if the title doesn't resolve; absent band stays a strict no-op. The profile title/editorType come from the profile JSON the helper *already parses* (zero new call-site/signature change, no extra parse). Scope boundary: `analysisFlags`/UGS deliberately keep using the persisted kbId — the same stale-kbId latent issue exists for them, but that is **pre-existing behavior, recorded here and explicitly out of this change's scope** (a separate correctness concern in the #1160 lineage, not to be conflated with the band feature). Regression: `tst_dialing_blocks::expertBand_staleKbId_freshTitleResolutionRecoversBand` pins it (stale → absent; fresh title → band; twin collapses; band-less stays absent). *Why caught here:* this is precisely the value of the D12-style live gate over test-green — the unit slice (synthetic `ExpertBand` passed directly) is byte-correct, but only a real historical shot exercised the stale-persisted-`profileKbId` path.

## Risks / Trade-offs

- **[Confounded band-verdict false positive — the #1155 surface]** → D3 (observational, taste-deferring, never directional) + D8 hard AND-gate + D9 corpus false-positive gate. Shot 938 (confounded bitter-but-coarse) is a seed-corpus must-not-misfire case.
- **[Seeded band/axis carries a transcription error]** → D4: every entry verbatim from a `[SRC]`-tagged line already graded in capture-dialin-coaching-guidance D9; no new derivation; coverage tests assert presence/shape.
- **[Over-reading the limiter again]** → D1: limiter is a corroborating clause only; spec scenario asserts a limiter touch with the peak inside the band does NOT fire and the band line fires with no limiter present.
- **[Confusing the expert-flow band with the commanded-flow check]** → D2: the table's `extraction-flow` band is the cited expert target (Allongé "reach 4.5"), explicitly distinct from `analyzeFlowVsGoal`'s commanded setpoint, which is untouched; a spec scenario asserts both can coexist without interference.
- **[Marginal value after gating]** → Accepted; D9's shadow run is the decision point — negligible conservative-tuned firing → the line is simply not enabled.
- **[A band-less profile gets a fabricated band]** → D4/D5 make absence intentional; a no-fabrication scenario in the spec.
- **[Supersede/replace ordering]** → Seam-free and self-contained; archiving the two predecessors drops no shipped behavior (both 0 implemented tasks); their evidence chains are cited, not lost.

## Migration Plan

Additive code only: a static citation-bound axis+band table + one comparison branch in the existing `analyzeShot` cascade emitting one `summaryLines` entry; absent classification is a strict no-op. No DB migration, schema/storage/badge/Visualizer change. Rollback is a single revert. Verification at the gate: the Qt Test suite stays green; new `tst_shotanalysis` assertions (outside-band fires on the cited axis; no-cited-band never fires; absent-classification byte-identical; recompute-on-load parity; limiter clause pressure-axis addendum-only; expert-flow band coexists with `analyzeFlowVsGoal`); and a `shot_eval` shadow run over the seed corpus — **April 6 10:23 AM** (peak ~10, outside 6–9: fire + limiter clause), **May 10 9:04 AM** (peak ~8.4, inside: silent), **May 15 2:32 PM** (confounded: at most observational, no direction), **May 15 2:37 PM** (peak ~4.6, below 6: fire, no limiter clause), plus a flow-axis fixture (Allongé well under ~4.5 ml/s: fire on the flow axis) — with the chosen conservative margin recorded in the change log before the line is enabled.

## Open Questions

- Margin outside the band before firing (per axis: bar vs ml/s) — deliberately not fixed here; the output of the D9 shadow run, recorded at apply time.
- Long-tail cut line (which citation-graded lever/flow profiles in the first applied increment) — a curation judgment against capture-dialin-coaching-guidance D9, not a spec decision.

Accepted as-is per the user's literal instruction; a one-token exclusion if it proves noisy. Also: whether the affordance should *appear* (not just tint) when the verdict is non-clean but no fault badge is set — leaning yes (the "worth opening, no hard fault" case), finalized against `ShotDetailPage`/`PostShotReviewPage` visibility at apply time.
