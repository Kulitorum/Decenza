## Why

The Grind quick-select widget offers whole-number steps (…7, 8, 9…) even though the app already knows the user grinds in 0.25 increments — the MCP `dialing_get_context` reports `smallestStep: 0.25` for the Niche Zero from the user's own shot history. That knowledge is walled off in the AI/MCP path; the widget instead reads a global `Settings.brew.grindQuickSelectStep` that defaults to `1.0` and must be tuned by hand. The widget should step the grinder the way the user actually uses it, from the data we already have.

Two secondary problems ride along: (1) the Grind and Ratio pills draw an opaque white capsule that clashes with a background image, where Beans/Milk/Equipment/Espresso and four other widgets already render transparently; and (2) the existing history-derived step is a raw minimum gap, so a single mistyped setting in history (e.g. an `8.1` among 7.5/8/8.5/9) permanently collapses the step to 0.1 for both the widget and the AI.

## What Changes

- **Derive the grind step from the user's own shot history, per grinder**, in one shared C++ helper consumed by both the widget and the MCP/AI dialing block — a single source of truth. The Grind widget stops reading `Settings.brew.grindQuickSelectStep` and instead asks the shot-history store for the step observed on the currently selected grinder.
- **Noise-filter the derivation.** Replace the raw minimum-gap with a robust modal-gap estimate that rejects lone outliers (a stray mistyped value no longer poisons the step), with ties broken toward the finer step and a sane floor. This improves both the widget's rows and the AI's `grinderContext`.
- **Handle "no grinder selected."** When no grinder is defined, derive the step from the full observed history (all grinders' numeric settings) rather than falling back to a fixed `1.0`. When history is too thin to derive (fewer than two distinct numeric observations), fall back to `1.0`.
- **Derive the RPM step too**, for variable-RPM grinders. The widget's RPM mode stops using a hardcoded 50-RPM step and instead runs the same noise-filtered estimator over the user's observed `shots.rpm` values for the active grinder, falling back to 50 when history is too thin. (RPM dials are continuous, so this reflects the user's most common logged move rather than a physical detent — an accepted trade for parity with burr stepping.)
- **Pair RPM *with* grind in the widget instead of toggling between them.** Today the widget is mutually exclusive — an RPM-capable grinder shows *only* RPM and hides the burr setting. Redesign it to a **single combined pill** showing both values (e.g. "8.75 · 900") that opens **one picker with two sections** — a Grind section stepped by the burr increment and an RPM section stepped by the RPM increment — so both halves of the dial-in are visible and settable. Non-RPM grinders still see grind alone. Gate the RPM half on the broad `grinderRpmCapable` (matching BrewDialog), which reconciles the widget's old narrow `isKnownRpmGrinder` gate with the rest of the app.
- **Share RPM alongside grind across every read/display/serialize surface** (a broad pairing pass — the audit found the storage layer already paired, but ~30 read/projection/display sites drop rpm). The linchpin is `ShotProjection::toVariantMap`/`fromVariantMap`, which drops `rpm` and thereby blinds MCP `shots_get_detail`/`shots_compare`, the shot-history list row, and the clone round-trip. Beyond that: the AI dial-in blocks (`dialInSessions` per-shot, `bestRecentShot`, `changeFromPrev`/`changeFromBest` diffs, the prose grind line, `grinderContext` observed-RPM summary), the remaining MCP reads/inputs (`shots_list`, `settings_get`/`settings_set`, `shots_update`, `bag_create`/`bag_update`), the shot comparison model + table (add an RPM row), bag-from-history, the Visualizer PATCH (which currently clobbers the uploaded `"<setting> <rpm>rpm"` back to bare setting), the ShotServer web forms, and the remaining QML surfaces (history row, auto-favorites, `%RPM%` custom token).
- **BREAKING (MCP payload):** rename `grinderContext.smallestStep` → `grinderContext.stepSize` to reflect the new "typical detent, noise-filtered" semantics (it is no longer strictly the minimum gap). The in-app AI user-prompt enrichment updates its label to match.
- **Remove `Settings.brew.grindQuickSelectStep` entirely** — the property, the Machine-tab "Grind step" control, and its backup/restore serialization. The step is a property of the *selected grinder* (a Niche's 0.25 detent is meaningless for a different grinder or a swapped burr set), so a single global value is a category error: it applies one grinder's increment to all of them. Now that the step is derived per grinder from history, the global override has no coherent meaning and is removed.
- **Transparent pills on a background image.** The Grind and Ratio quick-select pills adopt the existing `hasBackgroundImage ? "transparent" : Theme.surfaceColor` convention (fill + border), and move their value text off `Theme.primaryColor`-on-white to a color that reads directly on the background, matching Beans/Milk.
- **Teach the advisor to recommend and score RPM.** Add an `rpm` field to the `nextShot` structured-output schema (offered only for variable-RPM grinders, independent of `grinderSetting`), score it in adherence tracking (`rpmMatches`, ±25 RPM + no-movement guard), render it in the Predicted / “Your next shot” lines, and carry the anchor shot's actual RPM on the `grinderCalibration` `coffeeAnchor` so a variable-RPM recommendation can name the concrete RPM.
- **RPM brew override when starting a shot.** `ProfileManager::activateBrewWithOverrides` gains an optional `rpm` parameter, and the `machine_start_espresso` MCP tool accepts an `rpm` override alongside grind.
- **RPM in the ShotServer web forms.** The shot, bag, and recipe HTML edit forms get an RPM input (field + load + save), plus RPM in the bag/recipe list summary lines.
- **Update the wiki manual** entry for the Grind widget to describe automatic, history-derived stepping (and the removal of the Grind step setting), plus the combined grind+RPM pill and two-section picker.

## Capabilities

### New Capabilities
- `grind-rpm-pairing`: RPM travels with the grind setting across every shot serialize/project/read/input/display surface (the cross-cutting pairing rule beyond the dial-in payload and the widget).

### Modified Capabilities
- `layout-brew-widgets`: adds a Grind quick-select widget requirement (history-derived grind AND RPM steps, combined grind+RPM pill with a two-section picker, no-grinder fallback, transparent-on-background rendering) and updates the Ratio widget requirement to render transparently over a background image.
- `dialing-context-payload`: the `grinderContext` step field is renamed to `stepSize` and derived by the shared noise-filtered estimator; dial-in blocks pair `rpm` with `grinderSetting` (per-shot entries, `bestRecentShot`, diffs, prose line), `grinderContext` gains an observed-RPM summary, and the `nextShot` structured-output schema gains an `rpm` recommendation the adherence loop scores.
