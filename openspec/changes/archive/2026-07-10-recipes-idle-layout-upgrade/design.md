## Context

The idle layout is a JSON zone map persisted in `QSettings` (`layout/configuration`), generated by `SettingsNetwork::defaultLayoutJson()` when absent, and lazily migrated in `getLayoutObject()` (existing idempotent injections: equipment, recipes, statusBar, machineStatus merge). Whole-layout reset flows through `SettingsNetwork::resetLayoutToDefault()`, used by both `SettingsLayoutTab.qml` and the web editor. Recipe promotion-from-shot already exists (`recipe_create_from_shot` in `src/mcp/mcptools_recipes.cpp`), building recipe fields from a `ShotRecord` including the `steam_json` snapshot (`hasMilk`, `milkWeightG`, pitcher info). Recipe activation is a single path: `MainController::activateRecipe(recipeId)`. `qml/main.qml` already has a one-time `firstRunDialog` gated on a `Settings.value("firstRunComplete")` flag — the precedent for the upgrade dialog.

The current default centerTop is `recipes, espresso, steam, hotwater, flush`; bottomRight is `history, spacer, beans, equipment, autofavorites, settings`. The target (per Jeff's screenshot) is centerTop `recipes, beans, steam, hotwater`; bottomRight `flush, history, equipment, espresso, settings`.

## Goals / Non-Goals

**Goals:**
- Make recipes the **default** way of using the app while keeping them strictly **optional**: the profile-first workflow must remain fully functional and reachable after the upgrade (Profiles button in the bar keeps its pill row / selector / Brew Settings behaviors), and the existing ingredient-swap deactivation rule means picking a different profile drifts the user back to profile-driven brewing with no explicit opt-out.
- New recipes-first default for fresh installs and both Reset-to-default surfaces.
- One-time, opt-in upgrade for existing users that surgically edits their layout and seeds a starter recipe from the last shot.
- Zero data loss: decline = no-op; accept touches only espresso/recipes/autofavorites placements, and everything removed (Auto-Favorites slot, center Profiles) stays re-addable via the layout editor.

**Non-Goals:**
- No removal of the Auto-Favorites page, widget type, or catalog entry — only its slot in the default layout and (on accept) in upgraded layouts.
- No change to the recipes MRU/pill behavior, the promotion path's field semantics, or the recipe activation path.
- No forced migration: users who decline keep the old arrangement forever (they can still Reset to default later).

## Decisions

1. **Default layout: edit `defaultLayoutJson()` in place; keep item ids stable** (`recipes1`, `beans1`, `espresso1`, `flush1`, …) so nothing else keying on ids changes. Drop `spacer2` — the screenshot shows one evenly spaced right-aligned group. The existing equipment/recipes injection migrations are naturally no-ops on the new default (both types present).

2. **Upgrade transform lives in C++** (`SettingsNetwork::applyRecipesFirstUpgrade()`, `Q_INVOKABLE`), not QML — it must walk/edit zone JSON with the same care as existing migrations and persist once via `saveLayoutObject()`. It reuses the `insertRelativeTo` pattern from the recipes injection. Idempotency: calling it twice yields the same layout (guards: espresso-in-center check, has-recipes check, autofavorites removal is naturally idempotent).

2b. **Pristine detection: layout equals the migrated old default → full reset.** Before transforming, compare the user's zones against the *old* default composition after its injection migrations (structural comparison on type sequences per zone, ignoring ids/displayMode noise — decide the exact equivalence during implementation, but type-order per zone is the intent). Equal → call `resetLayoutToDefault()` so uncustomized users land exactly on the screenshot layout (Beans center, Flush bottom). Anything else → surgical transform. The old-default composition is frozen as a constant for this comparison since `defaultLayoutJson()` itself changes in this release.

3. **Offer gating: a new settings flag `layout/recipesUpgradeOffered` (bool)**, set true when the dialog is answered *and* on fresh-install first-run completion (so new users never see it). Eligibility check on idle-page readiness in `main.qml`, mirroring `firstRunDialog`: show when `firstRunComplete == true` and `recipesUpgradeOffered == false`. Ordering: if `firstRunDialog` is due (fresh install), set the upgrade flag and skip. Event-based, no timers (per project rule).

4. **Starter recipe: extract the promotion field-building from `mcptools_recipes.cpp` into a shared helper** (e.g. `RecipePromotion::fieldsFromShotRecord(record, name, hasMilkOverride, fallbackSteamJson)` in a small `src/recipes`-adjacent unit or on `RecipeStorage`), so the MCP tool and the upgrade share one implementation instead of a copy. The upgrade path: load latest shot id (background thread via `withTempDb`, per the no-main-thread-I/O rule) → build fields → `requestCreateRecipe` → on `recipeCreated`, `MainController::activateRecipe(id)`.

5. **Drink type: user chooses in the dialog; heuristic only pre-selects.** The snapshot's `milkWeightG` derives from `lastSteamMilkG`, a persistent last-steam-ever value never cleared by espresso-only shots, and `hasMilk` only appears when a recipe was active — so the snapshot can only say "has steamed milk before", not "this drink had milk" (confirmed against `main.qml` / `MainController::currentSteamSpecJson`). The dialog therefore carries an Espresso / Milk drink choice (shown only when a starter recipe will be created), pre-selected milk when `hasMilk == true` or `milkWeightG > 0`, espresso otherwise. The choice sets `hasMilk` on the created recipe (steam-heater intent on activation) and the default name.

6. **Starter recipe only when the user has zero recipes.** Recipes have not shipped in a release yet, so effectively every upgrading user qualifies; the guard is cheap insurance for anyone (Jeff, beta devices) who already has recipes — they keep their set and active recipe untouched.

7. **Default names (translated):** "My espresso" / "My milk drink" (`upgrade.recipe.espressoName` / `upgrade.recipe.milkName`). Short enough for MRU pills; the user can rename in the composer.

8. **Dialog copy previews the three edits** (Recipes takes center stage; Profiles moves to the bottom bar; Auto-Favorites button removed — page still reachable) plus, when a starter recipe will be created, a starter-recipe line with the Espresso / Milk drink choice (pre-selected per decision 5). The copy must land the "default, not mandatory" framing: profiles keep working exactly as before from the bottom bar, and nothing is lost — only rearranged. Buttons: "Upgrade layout" / "Keep my layout". Standard accessibility (focus, `Accessible.name`, dismiss = decline). On accept, a toast confirms the outcome (e.g. "Layout updated · Created 'My espresso'") so the change is discoverable.

## Risks / Trade-offs

- [User deleted the espresso button entirely] → transform steps are independent; recipes-presence and autofavorites removal still apply; no espresso item is fabricated.
- [Espresso button in a bar zone already] → leave it; only the center-zone instance is relocated. No duplicates.
- [Last shot predates steam snapshots] → empty `steam_json` classifies as espresso and the promotion helper falls back to current steam settings for the steam block — matches existing MCP behavior.
- [Heuristic pre-selects the wrong drink type] → the user sees and can flip the choice in the dialog before anything is created; `hasMilk` stays editable in the composer afterwards.
- [Backup/restore races the flag] → flag rides in normal settings backup like `firstRunComplete`; a restored old backup may re-offer once on a new device, which is acceptable and arguably correct.

## Migration Plan

Ships in one release. No data migration; the stored layout only changes on explicit accept (persisted via the existing `saveLayoutObject`). Rollback = users who accepted keep a valid layout (all widget types pre-exist); users who declined are untouched.
