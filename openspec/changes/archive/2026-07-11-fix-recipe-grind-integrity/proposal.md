## Why

GitHub issue #1468 reports that (1) the Shot Plan text/icon turns orange for no reason a user can see, and (2) a grind change made while a recipe is active can silently revert to its old value right after the shot completes. Investigation traced these to three independent defects: the temperature-override "orange" flag is a one-way latch that never truly clears (Bug A); a post-shot bag write bypasses the settings layer's echo-suppression and unconditionally re-applies the bag's stale grind over a pinned recipe's edit (Bug B1); and recipe re-activation does an uncoordinated fresh DB read that can race an in-flight grind write and lose it (Bug B2). Bug B1 exists specifically because today's grind model deliberately lets the bag's stored grind go stale relative to a pinned recipe's private value (to avoid sibling recipes following each other's pin) — that deliberate staleness is exactly what the spurious resync clobbers. Rather than patch around that fragility, this change also simplifies the model itself: a recipe's grind always lives on the recipe (no more inherit-from-bag mode at activation time), while the bag's stored grind stays what it already is today for the common case — always the most recently dialed value, updated live and unconditionally, whether driven directly or via an active recipe. Removing the special-cased suspension (rather than adding more coordination around it) removes the condition Bug B1 depends on.

## What Changes

- **Fix Bug A** — `SettingsBrew::hasTemperatureOverride`/`hasBrewYieldOverride` stop being one-way latches: they reflect a real, currently-active deliberate override and are genuinely cleared (not just value-resynced) on profile switch, recipe activation/deactivation, and "Clear" in BrewDialog, matching the `brew-overrides` spec's existing (currently unmet) "Overrides cleared on profile switch" promise. Reconcile the duplicate highlight rule in `CustomItem.qml` with `ShotPlanText.qml`'s.
- **BREAKING: Remove the recipe grind inherit/"follow bag" mode at activation.** A recipe's own grind (and rpm, when its grinder reports rpm) always lives on the recipe — a recipe never reads its grind *from* the bag at activation time. The wizard's override toggle ("Follows the bag" / pin) is removed; recipe grind is simply a field.
- New-recipe grind/rpm fields **default-prefill** from the linked bag's current `grinderSetting`/`rpm` at creation time only (a one-time copy offered as an editable default, not a live link); rpm only prefills when the selected equipment's grinder is rpm-capable. The same creation-time default applies on non-interactive create surfaces (MCP/web: an *omitted* grind adopts the bag's dial at save time; an *explicitly empty* grind stays empty), and **promote-from-shot defaults from the shot's own recorded grind/rpm** instead of today's empty/inherit encoding.
- **Bag `grinderSetting`/`rpm` write-through becomes unconditional, not deferred.** Every grind/rpm edit continues to write through to the active bag immediately (as it already does today for the common, non-pinned case) — now including while a recipe with its own owned grind is active. `m_grindBagWriteThroughSuspended` is removed entirely: there is no longer a reason to withhold the bag write, since a recipe's own grind is never read back from the bag. The bag and the active recipe's own field both update immediately, independently, from the same edit — this closes Bug B1 by removing the staleness the bug depended on, without adding any new write-coordination machinery.
- **Fix Bug B2** — recipe (re-)activation no longer blindly overwrites fields that have a self-write still in flight for the currently-active recipe; a same-id re-activation reconciles against pending writes instead of racing them.
- Recipe relink-to-a-different-bag (`recipe-bag-lifecycle`) simplifies: grind is never touched by a relink (the inherit branch of that behavior no longer exists).
- **Recipe wizard**: the details step reorders so equipment (and therefore grinder rpm-capability) is known before the grind field is reached, so the rpm field's visibility is correct the first time the user sees it instead of only after they separately visit the equipment section.
- **Data migration**: existing recipes currently in inherit mode (empty `grindPinned`) are backfilled once from their linked bag's current grind/rpm at upgrade time, so every recipe has an owned grind value going forward. The same backfill runs on the device-transfer/backup import path, since imported recipes can come from a pre-migration source DB.
- **MCP/web grind surfaces move to the recipe-owned model.** The MCP tools currently expose `grind.mode: "inherited"|"pinned"` and document `grindPinned: ''` as "inherit from the linked bag (recommended)"; the web surface resolves an `effectiveGrind` from the bag. All of this updates to the recipe-owned model. Not a compatibility concern: MCP clients reload the tool schemas and descriptions on every connection, so the new semantics self-propagate — the work is updating the tools, their descriptions, the protocol-test stubs, and `MCP_SERVER.md`.
- Update `docs/CLAUDE_MD/RECIPES.md`'s grind section and the `RecipeStorage` header comment to describe the new always-owned model.

## Capabilities

### New Capabilities
(none — this change modifies existing recipe/brew-override capabilities)

### Modified Capabilities
- `recipe-model`: "Grind inherit-or-pin" requirement is replaced by recipe-owned grind (always stored on the recipe; bag supplies only the one-time creation default).
- `recipe-activation`: activation's grind resolution drops the bag-inherit branch; grind/rpm always come from the recipe row.
- `recipe-wizard`: remove the inherit/override toggle from the details-step grind fields; add a requirement that equipment precedes grind in the details layout so rpm-capability is known when the grind field renders; update the "Drink-type-specific details fields" wording.
- `recipe-bag-lifecycle`: "Relink never rewrites grind" simplifies — there is no more inheriting-recipe branch to describe; relink is a bag-link-only operation, full stop.
- `coffee-bag-model`: "Bean/grinder edits write through to the active bag" is clarified to hold unconditionally, including while a recipe with its own owned grind is active (previously silently suspended in that case, contrary to what this requirement already promised).
- `brew-overrides`: "Persistent Override Storage" / "Overrides cleared on profile switch" is tightened and actually implemented — `hasTemperatureOverride`/`hasBrewYieldOverride` are real, clearable flags, not permanent latches.
- `plan-widgets`: "Override indicators" gains a scenario confirming the highlight only appears for a genuinely active, deliberate override (not a stale/coincidental one) — verifying Bug A's fix at the display layer.
