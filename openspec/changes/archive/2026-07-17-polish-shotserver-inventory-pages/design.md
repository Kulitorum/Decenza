## Context

The ShotServer serves `/beans`, `/recipes`, and `/equipment` as inline-HTML SPAs generated by `generateBeansPage()` / `generateRecipesPage()` / `generateEquipmentPage()` (`src/network/shotserver_bags.cpp`, `shotserver_recipes.cpp`, `shotserver_equipment.cpp`). Each ships a static skeleton, then JS `fetch()`es `/api/*` JSON and builds the DOM client-side.

They already share theme tokens and chrome via `webtemplates/` (`WEB_CSS_VARIABLES`, `WEB_CSS_HEADER`, `WEB_CSS_MENU`, `generateMenuHtml()`, `WEB_JS_MENU`, `WEB_JS_POWER_CONTROL`). Two problems make them read as unfinished, and a third makes them under-featured:

1. **Divergent header** — the three pages use a bare `<div class="header">` with a lone emoji title instead of the canonical `<header class="header">` (logo + back + `.header-right` burger) that Shot History and every other page use.
2. **Copy-pasted content CSS** — each file appends a character-for-character duplicate `<style>` block (bags:271-292, recipes:552-574, equipment:229-250) and repeats the same JS utilities (`esc`, `status`, `post`, list-render, modal open/save). There is no shared card/form/modal helper.
3. **Feature gaps vs. the app** — deliberate v1 deferrals: equipment puck-prep flags "stay in-app" (`shotserver_equipment.cpp:27`); `/beans` is "plain form fields (no URL/AI import)" and omits tea, full bean attributes, freeze actions, Bean Base search/linking, and AI extraction; `/recipes` lacks the search/sort bar, the full steam block, RPM, and the stale-bag re-point picker.

Crucially, the REST layer already accepts more than the forms expose: `kind` (coffee|tea), `yieldG`/`yieldRatio`, and `rpmPinned` are already parsed (`shotserver_bags.cpp:47-54,121-129`; `shotserver_recipes.cpp:138-139`). So much of "feature parity" is **surfacing existing API capability in the web UI**, not new backend work. The genuinely new backend surface is Bean Base search, AI page-extraction, and bean images.

The app's design system (`qml/Theme.qml`, `BagCard`/`RecipeDrinkCard`/`EquipmentCard`): rounded 16px surface cards, 1px border → 2px accent border when active, ~380px target card width with computed integer columns and 16px gutters, dot-joined dense attribute lines that omit missing fields, lifecycle-driven destructive action, and a small caption "Active" badge (recipes) or accent border (bags/equipment) as the active indicator.

## Goals / Non-Goals

**Goals:**
- Full **visual parity**: canonical header, responsive card grid, app-style card hierarchy, active-item accent highlight, chips, empty states — driven by one shared style.
- Full **feature parity**: every field/action the app offers on these three screens is reachable from the web, including Bean Base search + linking, AI page-extraction, bean thumbnails, tea, freeze actions, recipe search/sort + full steam block + RPM + stale re-point, and equipment puck-prep flags.
- **Do not duplicate code.** Replace the copy-pasted CSS/JS with shared helpers; reach parity features by **reusing existing backends** (`BeanBaseClient`, `CoffeeBagStorage`, `RecipeStorage`, `EquipmentStorage`) and the patterns already proven by the MCP tools and the async community endpoints — not by re-implementing them.

**Non-Goals:**
- No changes to the in-app QML, the data model, or existing REST endpoint semantics/payloads (additions only).
- Not re-skinning the whole ShotServer to the app's exact color palette (`#1a1a2e`/`#4e85f4`/`#e94560`); keep the web's coherent dark tokens and match layout/hierarchy/card-grammar. Palette parity is a separable follow-up.

## Decisions

### Decision: One shared embedded-page style, zero per-page CSS copies
Add a `webtemplates/` header (e.g. `management_css.h` → `WEB_CSS_MANAGEMENT`) with the card grid, card + `.active`, buttons, badges/chips, `#status`, empty state, form inputs, and dialog/modal. All three pages include it and delete their private `<style>` blocks. Likewise consolidate the repeated JS utilities (`esc`, `status`, `post`, a `joinWithBullet`-style dot-joiner, the card/list render + modal open/save helpers) into a shared JS constant.
- **Why:** Directly answers "avoid copying code" and is the root cause of the demo feel. One place to style, one place to fix.
- **Alternative:** A `generateManagementPage(title, emoji, bodyHtml, pageJs)` scaffold that also emits the header shell. Adopt for the header/shell (see next decision); keep bodies per-page because they diverge (recipes' archived section + search bar). The scaffold can absorb more later.

### Decision: Canonical header via a shared shell helper
Route all three pages through the same header the Shot History page uses (`☕ Decenza` logo → `/`, page title, back link, `.header-right` burger). Prefer a shared shell helper over pasting the header markup into each page.
- **Why:** Removes the most obvious "unfinished" tell and, done as a helper, avoids re-copying the header three times.

### Decision: Responsive grid in CSS, not JS column math
Use `grid-template-columns: repeat(auto-fill, minmax(~340px, 1fr))` (or equivalent flex) so cards wrap and collapse to one column on narrow screens without per-page JS.
- **Why:** Reproduces the app's `Flow` grid with pure shared CSS; stays touch-friendly on the DE1 tablet.

### Decision: Surface existing API fields in the forms (no new endpoints needed)
`kind`, `yieldG`/`yieldRatio`, `rpmPinned`, equipment link, freeze dates, and the full bean attributes are already accepted by the bag/recipe APIs. Parity here = adding the form controls and card lines that read/write these existing fields. Equipment puck-prep flags are the one **field-level** addition that also needs the API to accept/return them (extend the editable-key set + JSON in `shotserver_equipment.cpp`, routing through `EquipmentStorage`).
- **Why:** Cheapest path to most of the parity; avoids inventing endpoints for data the server already round-trips.

### Decision: Reuse `BeanBaseClient` for web search + AI extraction, mirroring the MCP tools
Add `GET /api/beans/search` and `POST /api/beans/extract` that call the **same** `BeanBaseClient` backend the app and the MCP tools use (`src/mcp/mcptools_beansearch.cpp` for `bean_search`; `src/mcp/mcptools_ai.cpp` for `bag_extract_details`). Search runs off the request thread; extraction is **async** and MUST follow the established ShotServer async pattern (`QPointer<QTcpSocket>` + `std::shared_ptr<bool>` fired guard + `QTimer` timeout, always emitting an error on timeout/rejection). Linking a bag to a canonical record reuses the existing bag-update write-through.
- **Why:** No re-implementation of Bean Base logic; the MCP tools are a working template for the exact same calls. Honors "avoid copying code" by reusing the backend rather than porting it.
- **Alternative considered:** Re-porting extraction/search into the web layer — rejected (duplication, drift, extra maintenance).

### Decision: Serve bean images via a small dedicated route
Add `GET /api/bag/<id>/image` (or equivalent) that returns the bag's cached photo, reusing whatever path the app's `BeanThumbnail` reads, with a placeholder fallback. The server already serves image bytes (e.g. the SVG route at `shotserver.cpp:2681`), so this follows an existing pattern.
- **Why:** Thumbnails are part of the app's card grammar; a single image route keeps the web card faithful without embedding image data in the JSON.

### Decision: Recipe search/sort is client-side
The `/recipes` list is already fully fetched; implement search + sort in the shared page JS over the in-memory list, matching the app's fields (date used / created / coffee / profile / name + direction).
- **Why:** No new endpoint; mirrors the app's behavior; trivial and fast for realistic recipe counts.

## Risks / Trade-offs

- [Scope is now large — look **and** features across three pages] → Mitigation: sequence the work (shared style/shell first, then per-page visual, then per-page feature parity, with the heavy `/beans` search/extract/image endpoints last); each task is independently verifiable.
- [Async extraction endpoint can hang a socket] → Mitigation: follow the proven async pattern exactly (fired guard + timeout + always-respond), and verify acceptance after invoking `BeanBaseClient` (mirror the community-endpoint "verify it was accepted" rule).
- [Bean Base is rate-limited / tier-gated] → Mitigation: reuse the client's existing 429 classification and error surfacing; the web `.catch()` must show a clear message, not a stuck spinner (per SHOTSERVER.md fetch rules).
- [Shared CSS/JS changes all three pages at once] → Both the benefit and the risk. Mitigation: verify all three pages (including the recipes archived section + modal) after each shared-helper change.
- [Palette still differs from the app] → Called out as a separable follow-up, not silently skipped.
- [Responsive/touch on the DE1 tablet browser] → Use widely-supported CSS grid/flex, keep targets ≥36px, test at tablet and phone widths.

## Migration Plan

Additive and presentation-facing. New endpoints are new paths; existing endpoints/payloads are unchanged, so existing clients keep working. Deploy is a normal app build (HTML/CSS/JS ship in the binary; new routes are compiled in). Rollback = revert the source change. No persisted state or schema is affected.
