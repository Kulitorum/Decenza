## Context

`resources/ai/profile_knowledge.md` is the Profile Knowledge Base shipped via Qt resource into both AI paths: the in-app advisor system prompt and the `dialing_get_context` MCP response. `ShotSummarizer::loadProfileKnowledge()` parses each `## ` heading into a `ProfileKnowledge` record. Per-section it extracts one `UGS:` value (stripping a leading `~` → `ugsInferred = true`, and stripping any trailing parenthetical), an `Also matches:` alias list, `AnalysisFlags:`, and the free-text behavioral content. Every alias key normalizes (lowercase, diacritics, `&`→`and`) and maps to that one record. The section *title* is additionally split on `" / "` and each part registered as its own key.

Today a single `## D-Flow` section enumerates seven variants via `Also matches:` and carries one `UGS: 0.5 (Damian's LRv2/LRv3 variants sit at 0 like Londinium; base D-Flow and Damian's Q at 0.5)`. The parenthetical is stripped at parse time, so all seven variants resolve to UGS 0.5 and the canonical name "D-Flow". Two cross-profile tables (a canonical one keyed to the videoblurb.com/UGS chart, and an inferred one) plus a "How to use this ordering" section give the AI its textual reasoning surface for cross-profile grind transfer.

The authoritative UGS chart was read verbatim from the calculator's JS source. It contains exactly one D-Flow entry — base D-Flow at 0.5 — and "Londinium / LRv3" at 0. D-Flow/Q, Damian's Q, LRv2, and LM Leva are absent from the chart entirely; any value for them is inferred. The chart's only two 6-bar profiles (Extractamundo Dos 1.5, Gentle & Sweet 2) sit 1.0–1.5 coarser than base D-Flow at 0.5.

The single consumer of the parsed `ugs` value in production code is the grinder-calibration anchor path in `dialing_blocks.cpp` (`buildGrinderCalibrationBlock`), which already skips any shot group whose `canonicalName` differs from `canonicalNameForKbId(kbId)` — i.e. variant profiles never anchor a calibration. All cross-profile *grind transfer* reasoning is done by the LLM reading the markdown tables, not by code.

Separately, most profile sections already carry a free-text `Expected curves:` line (Blooming "Pressure 6–9 bar; 6–8g dripping before rise"; Londinium "pressure peaks ~9 bar then stays high"; A-Flow, Adaptive v2, Gentle & Sweet, Allongé, and others). This prose is the *only* expression of per-profile quantitative expectation today, and it is unreachable from code — the parser drops it into the opaque `content` blob. The reporter's broader ask (the shot summarizer flagging a shot that did not meet profile expectations) is therefore not D-Flow-specific: it is blocked KB-wide by the absence of any machine-readable expectation field. The D-Flow split is the trigger for this change, but the expectation schema is a general capability that happens to be seeded by the sections this change already rewrites.

## Goals / Non-Goals

**Goals:**

- Pressure-distinct D-Flow variants resolve to distinct UGS positions and distinct canonical names so the AI produces a directional grind adjustment instead of "no change" when transferring across them.
- The authoritative chart value for base D-Flow (0.5) and for LRv3 (== "Londinium / LRv3" == 0) is encoded exactly; inferred values are marked `~`.
- Shared behavioral false-positive suppression (declining pressure, pressurized soak, LRv2 flow-switch, LRv3 9-bar hold) is preserved for every variant.
- The textual cross-profile tables and the "How to use" guidance the LLM reads are internally consistent with the split.
- Introduce a general, profile-agnostic optional structured `Expected*:` schema parsed into `ProfileKnowledge`, applicable to any section, and **seed an archetype-diverse initial 5** (Adaptive v2, Damian's LRv2/LRv3, Blooming Espresso, E61, Allongé) from `docs/PROFILE_KNOWLEDGE_BASE.md`'s existing source-tagged dial-in targets — validating the capability before a gated follow-up scales it.
- Make the enriched per-profile guidance consumable by both AI surfaces now, as framed qualitative prose plus a directional-reading teaching clause; keep the future deterministic detector deferred.
- Minimal additive code change (one parser field, mirroring `ugs`; one system-prompt teaching clause); regression tests lock the UGS contract, the expectation-schema parse contract, and seed coverage.

**Non-Goals:**

- No change to the RGS anchor-selection algorithm or canonical/inferred `source` semantics. The only parser change is the additive optional `ProfileExpectation` parse alongside the existing `ugs` parse.
- Not splitting all seven D-Flow variants into seven sections — only along pressure/temperature fault lines.
- Not building the deviation detector that consumes the `ProfileExpectation` struct. That is a deferred `shot-analysis-pipeline` change. The struct is code-unconsumed here; its prose twin is consumed by the LLM now.
- Not populating `Expected*:` beyond the **archetype-diverse initial 5**. The rest of the well-documented set (base D-Flow, D-Flow/Q, standalone Londinium, Gentle & Sweet) and the long tail are a follow-up change gated on confirming value; all stay valid with an absent expectation, and none get invented numbers.
- Not feeding rigid pressure/flow bands to the LLM. The ADR rejected an LLM-facing hard-threshold rule; the LLM here consumes framed qualitative prose plus a directional-reading teaching clause (D10), and the structured struct is consumed (later) only by a deterministic detector that must AND-gate against existing suppression — see D8/D10.
- Not fixing the fuzzy-prefix matcher weakness for custom user-titled variant profiles.
- Not reconciling the pre-existing 80's Espresso spec-example/KB value drift, nor touching the 80's Espresso UGS value.

## Decisions

**D1 — Split into three sections, not seven; split on pressure/temperature, not on every distinct profile.**
The reporter's mechanism is pressure-target → grind. The fault lines that matter are: base ~9-bar/88°C, the 6-bar/84°C-fill Q, and the pure ~0-UGS Londinium-R lever sims. La Pavoni and LM Leva stay with base (no chart entry, ~8–9 bar, behaviorally and quantitatively adjacent; inventing separate inferred positions for them would re-introduce the exact inferred-on-inferred fragility #1160 warns about). Three sections capture the real quantitative spread while keeping the behavioral grouping that suppresses false positives. *Alternative considered:* the issue's other option, a single section with a pressure-offset annotation. Rejected — the parser strips parentheticals/annotations, so an offset would need a parser change (production code, broader blast radius) to have any effect; the section split needs none.

**D2 — D-Flow/Q at `UGS: ~1.0`, inferred.**
Not on the chart, so it must be `~`. Bounded below by base D-Flow (0.5) and above by the chart's 6-bar profiles (Extractamundo/G&S at 1.5–2.0). D-Flow/Q is a pressurized-soak flow profile with a 6-bar *approach* (coarser pull) but a low 84°C fill that reduces solubility (pulls finer), so it should land below the flat-6-bar profiles, not at them. ~1.0 = ~0.5 UGS coarser than base — unambiguously "noticeably coarser, do not transfer 1:1," without overclaiming a precise position the chart never measured. *Alternatives:* ~1.25 (aligns numerically with Adaptive v2/Flat-9 but those are not 6-bar — coincidental, weaker rationale); 1.5 (the full Extractamundo value — overstates, ignores the finer-pulling low fill temp). The value is a documented inferred estimate, not a measurement; the test asserts the *ordering invariant* (Q strictly coarser than base), not the literal 1.0, so the exact number can be tuned later without breaking the contract.

**D3 — Q section title is `## D-Flow Q variant` (no `" / "`).**
The parser splits section titles on `" / "` and registers each part as a key. A `## D-Flow / Q` title would register the bare key `d-flow`, colliding with the base section (last-parsed wins, order-fragile). Titling it `## D-Flow Q variant` registers the inert key `d-flow q variant`; the real aliases `"D-Flow / Q"` and `"Damian's Q"` are carried via `Also matches:`, which is comma-split only (never `" / "`-split), so they register cleanly. `## Damian's LRv2 / LRv3` deliberately *does* use `" / "` — there both parts ("damian's lrv2", "damian's lrv3") are desired real keys and neither collides.

**D4 — Exact-alias match guarantees correct routing for built-ins.**
`matchProfileKey` checks `knowledge.contains(normalized)` (exact) before the fuzzy prefix loop. Every built-in title ("D-Flow / Q", "Damian's Q", "Damian's LRv2", etc.) is registered verbatim as an alias on the correct new section, so built-in profiles get an exact hit and route correctly. The fuzzy-prefix weakness only affects custom user titles that match no alias — pre-existing, explicitly out of scope, and the AI still has the corrected tables to reason from.

**D5 — Re-anchor 80's Espresso off base D-Flow.**
The inferred table currently rationalizes 80's Espresso relative to "D-Flow / Q" — an inferred value anchored on what was an undifferentiated value and is now itself explicitly inferred. Re-anchor the *rationale text* to base D-Flow (canonical 0.5). The 80's Espresso UGS value itself is untouched (out of scope; pre-existing spec drift noted separately).

**D6 — Mirror `docs/PROFILE_KNOWLEDGE_BASE.md`; review-only for `docs/UNIVERSAL_GRIND_SETTING.md`.**
`PROFILE_KNOWLEDGE_BASE.md` is the human-facing twin of the shipped resource and has the same `## D-Flow` block and inferred table — it must stay in sync or it becomes misleading. `UNIVERSAL_GRIND_SETTING.md` is an architecture decision record describing the *approach*, not the per-profile data; reviewed for stale claims. Its item-3 "keep the KB qualitative — do not formalize into rigid JSON rules" gets a clarifying note (see D8), but its substance stands.

**D7 — A general, profile-agnostic `Expected*:` directive schema, not a D-Flow feature.**
The expectation capability is introduced as a KB-wide convention: optional per-section directive lines `ExpectedPressureBar:`, `ExpectedFlowMlS:`, `ExpectedTimeSec:`, parsed into an optional `ProfileExpectation` struct on `ProfileKnowledge`. This mirrors exactly how `ugs`/`ugsInferred` were added — a typed optional field next to the free-text content, populated from a prefixed directive line, ignored when absent. Rationale for *general, not D-Flow-scoped*: the blocker the reporter hit (no machine-readable expectation anywhere) is KB-wide; ~25 sections already have free-text `Expected curves:` prose that this schema formalizes; scoping the schema to D-Flow would force a second KB-wide restructure later (the double-touch the reporter explicitly wants to avoid). Seeding is an **archetype-diverse initial 5**, not D-Flow alone and not the whole well-documented set (see D12 for the pick + the gated follow-up). Decent's official "Your first Decent espresso" dial-in guide independently validates both the schema shape and generality: it expresses per-profile expectations as exactly peak-pressure-bar, shot-time-sec, and a preinfusion-dripping-grams figure, and it covers **Adaptive v2** (peak 8–9 bar, ~25s, ~4g drip — a non-D-Flow archetype) and **Londinium LRv2** — both in the initial 5, seeded with real first-party data. *Alternatives:* (a) schema + soft detector now — rejected for this change: bigger blast radius, needs `tests/data/shots` corpus regression, reopens the ADR directly while #1160's real ask is the split; (b) a separate sibling OpenSpec change for the schema — rejected: it would re-touch every section after this change lands, the exact double-restructure to avoid; (c) seed D-Flow only — rejected by the user: it would ship a general capability with only its narrowest instance populated; (d) seed all ~9 well-documented profiles now — narrowed by the user to a validate-then-scale approach: prove value on 5 distinct archetypes, then a gated follow-up adds the rest.

**Scope at a glance:**

| | Now, in this change | Deferred |
|---|---|---|
| **General (all profiles)** | `ProfileExpectation` schema + parser seam; cross-surface consumability; directional-reading teaching clause; living-doc structural guidance | the deterministic deviation detector (`shot-analysis-pipeline`) |
| **Seeded data** | the **initial 5** archetypes: Adaptive v2, Damian's LRv2/LRv3, Blooming Espresso, E61, Allongé | the rest of the well-documented set (base D-Flow, D-Flow/Q, standalone Londinium, Gentle & Sweet) + the long tail — a follow-up change gated on confirming value |
| **D-Flow-specific** | the #1160 UGS split + distinct-position content fix (base D-Flow & D-Flow/Q sections are split/UGS-corrected/prosed here, just not `Expected*:`-seeded) | — |

The seed values for the `## Damian's LRv2 / LRv3` section are taken from Decent's official guide (which documents this exact profile as "Londonium LRv2"): 3 bar held 12s preinfusion → 9 bar with a slow gradual decrease; peak 9 bar declining; flow does not dip significantly during extraction; shot time 30–35s; too-coarse = pressure crash, too-fine = pressure exceeds 9 bar + reduced flow + much longer time. This replaces inferred numbers with a first-party citation and is recorded in tasks §1.4 / §3.4.

**D8 — Schema shape is detector-facing directive lines, not an LLM-facing JSON band; ADR-consistent.**
The UGS ADR rejected `"extractionPressure": {"min":6,"max":9}` *fed to the LLM as hard rules* — it over-applied and flagged intentional D-Flow behavior (March 2026). This schema differs on both axes that caused that failure: (1) it is consumed by a future *deterministic* detector, not injected as LLM rules; (2) that detector is contractually required to AND-gate against the existing `AnalysisFlags` / `flow_trend_ok` suppression and emit a *soft* "could be better" advisory line, never a hard error badge — structurally it cannot reproduce the false-positive that flagged correct declining-pressure D-Flow shots. To keep that guarantee legible, expectation values for intentionally-variable profiles are written as patterns/wide ranges (e.g. base D-Flow `ExpectedPressureBar: 6-9 declining`) so the data itself encodes "variable by design," and the ADR doc gets a one-line note recording this distinction. In this change there is no consumer at all, so the guarantee is trivially met; the note exists so the deferred detector change inherits the constraint.

The Espresso Aficionados profiling guide (`[SRC:eaf-profiling]`) independently hardens this: it *deliberately avoids* rigid graph-based grind/channeling diagnosis ("simply looking at the bottomless is not likely to tell you how severe your channeling issues are"; "just because a shot visually appears to need a taper doesn't mean that it does") and routes users to taste. The deferred detector therefore SHALL prefer first-party per-profile numeric targets (Decent's official guide; the doc's "Official dial-in target" lines) over generic graph heuristics, SHALL stay a soft advisory, and SHALL defer to taste — the universal rule (too sour → finer, too bitter → coarser) outranks any curve-shape verdict.

**D9 — `PROFILE_KNOWLEDGE_BASE.md` is the synthesis/seed layer; the schema must hold a two-sided diagnostic, not just a band.**
The link-mining is already done: `docs/PROFILE_KNOWLEDGE_BASE.md` is a source-tagged synthesis of all ~30 references (Decent blogs/videos, EAF, Scott Rao, Coffee ad Astra, the Basecamp community index). The future KB-wide population pass SHALL draw `Expected*:` values from that doc's existing per-profile **"Official dial-in target"** + **"Grind diagnostics"** lines, not by re-fetching raw links. Reading those lines surfaces the real shape of what is deterministically flaggable: the well-researched profiles encode (a) a centred target (Adaptive "8–9 bar peak, ~25s, ~4g drip"; Londinium "9 bar declining, 30–35s, 2–8g drip"; E61 "9 bar flat, 22–25s"), (b) a *two-sided* directional diagnostic ("too coarse → peak < 7 bar / pressure crash"; "too fine → > 9 bar / much longer time"), (c) a pressure-shape qualifier (declining / flat / rising-again-is-fines-migration / crash), and (d) a preinfusion-dripping figure with its own too-coarse/too-fine bounds (Londinium ">17g = too coarse"). It also surfaces a per-profile **suppression catalogue** (Allongé channeling normal; Advanced Spring Lever "pressure notch is GOOD"; Gentle & Sweet rising flow expected; Cremina steep decline intentional; LRv2 flow-control fallback not a fault; Adaptive grind-tolerant by design; Blooming zero-flow bloom intentional) — i.e. the detector's suppression set is per-profile, broader than the single `flow_trend_ok` flag. The initial 5 archetypes are **populated now** from these exact lines (no raw-link refetch); the rest of the well-documented set and the long tail are a gated follow-up. This change does NOT widen the directive *grammar* (minimal tokens — value range/pattern only), but the finding is recorded so the deferred grammar extension is deliberate, not rediscovered: the eventual `Expected*:` grammar will need target + too-coarse/too-fine + shape-qualifier + dripping, and the per-profile suppression must travel with it. The spec's parse contract ("populated/absent/malformed-degrades, pattern not collapsed to min==max") is intentionally loose enough to absorb that growth without a spec break.

The deferred detector's required confounder gates are enumerated here so that change inherits a concrete list, drawn from the false-positive sources Decent's own dial-in guide names: (1) channeling / poor puck prep (dry spots, spraying, inconsistent time) — already detected; the off-target advisory must suppress when channeling fired; (2) bean freshness unknown or very fresh (stale beans pull very fast — a short shot is then expected, not a defect) — gate on the existing `beanFreshness` knownness already in `dialing-context-payload`; (3) grinder cannot grind fine enough (equipment limit, not a dial-in fault); (4) profiles that are grind-tolerant *by design* (Adaptive v2 adapts flow to grind while capping pressure < 9 bar) — such sections must carry a wider/pattern expectation and the detector must treat them more leniently than a tight profile like LRv2. The advisory is "soft" in Decent's exact sense: aim for the recommended time + curve shape first, then let taste (too sour → finer, too bitter → coarser) lead — the detector never overrides a taste-based judgement.

**D10 — Consumable by both AI surfaces now, as framed qualitative prose; ADR-safe.**
The user's requirement is that the enriched guidance helps the MCP and the in-app advisor, not only a future detector. This is satisfied without re-opening the ADR because the *prose* is the LLM surface and the *struct* is the detector surface. KB section content already ships verbatim into both `dialing_get_context` and the advisor system prompt (the parser strips only `Skip-Catalog:` / `Purpose:`), so the split's per-variant pressure/flow/time/grind-direction narrative is consumed by both paths with zero new plumbing and zero divergence — they read the same KB section. The ADR's rejected design failed on two specific axes (a rigid min/max band, fed to the LLM *as a rule*); the prose twin is neither — it is qualitative, per-variant, and framed. To prevent the LLM from re-deriving a rigid rule out of the `Expected*:` tokens that also sit in the section text, the `shotAnalysisSystemPrompt` "How to read structured fields" section gains a teaching clause: expectations are directional context, intentional in-profile variability is normal and not a fault, and taste outranks any expectation-vs-actual comparison — the exact gating pattern the spec already uses for `tastingFeedback` / `beanFreshness`. *Alternative considered:* strip `Expected*:` tokens from LLM-facing content (like `Skip-Catalog:`) and feed the LLM only the prose. Rejected — the tokens are terse and harmless beside the prose, stripping adds parser special-casing for no benefit, and the teaching clause already neutralises the rigid-rule risk. The struct stays code-unconsumed; "consumable" here means the LLM reads the prose, which it always has.

**D11 — `PROFILE_KNOWLEDGE_BASE.md` is a living artifact, updated as we learn.**
The doc is not a write-once synthesis. As this and future work surfaces what is important to flag, the doc is updated in lock-step: this change documents the `Expected*:` directive convention beside the existing `UGS:` / `AnalysisFlags:` field-line docs, and records the D9 structural guidance (two-sided diagnostic shape; per-profile suppression catalogue) so the next contributor inherits it rather than re-deriving it. The existing tasks drift-check (resource vs. doc parity) is the mechanism that keeps "living" from becoming "diverging."

**D12 — Initial seed is 5 archetype-distinct profiles (validate-then-scale); the boundary is "has a source-tagged target," not "is D-Flow."**
Two filters apply. First, eligibility: a profile may be seeded only if `docs/PROFILE_KNOWLEDGE_BASE.md` already carries a source-tagged "Official dial-in target" / "Grind diagnostics" line for it — traceable to a first-party or community-vetted citation, so seeding introduces zero new inference (the long tail is excluded precisely because inventing a numeric target without a citation would reintroduce the inferred-on-inferred fragility #1160 exists to remove). Second, the user narrowed the initial cut to **5 profiles chosen for archetype uniqueness**, to confirm the capability is a valuable addition before bulk population: `Adaptive v2` (grind-adaptive — the hardest "variable by design" case), `Damian's LRv2/LRv3` (declining-pressure lever; also the #1160 section, so zero marginal section cost), `Blooming Espresso` (bloom + hard never-max bound + zero-flow suppression), `E61`/Flat 9 Bar (flat constant, the tight non-variable opposite), `Allongé`/Rao Allongé (high-flow turbo extreme + channeling-normal suppression). These five span the full set of extraction shapes; every deferred profile (base D-Flow, D-Flow/Q, standalone Londinium, Gentle & Sweet, then the long tail) is a *variant* of one of the five, so success across them confirms the schema, prose, and cross-surface consumability generalize. The rest follow in a **separate change gated on that confirmation**. Deferring base D-Flow and D-Flow/Q is cheap and not a double-touch: their sections are split/UGS-corrected/prosed here for #1160; adding one `Expected*:` directive line later is an append, not a re-restructure (the expensive double-touch the reporter feared was re-doing the split itself). The `## D-Flow Q variant` UGS (`~1.0`, inferred) remains the one deliberately-inferred value, justified separately in D2 and asserted only as an ordering invariant.

## Risks / Trade-offs

- **[Inferred ~1.0 for Q is a judgment call, not a measurement]** → Marked `~` (flows through to `source: "extrapolated"` in RGS, matching existing semantics for inferred values). The regression test asserts only the ordering invariant (Q coarser than base, LRv3 == 0), so the literal value can be retuned from real shot history without breaking the spec or the test.
- **[Splitting changes `canonicalNameForKbId` for Q/LRv shots]** → In `buildGrinderCalibrationBlock` the canonical-name-mismatch guard already excludes variant shots from anchoring; after the split a Q shot's group canonicalName ("D-Flow / Q") still differs from base and from its own new canonical name unless titled exactly, so behavior is unchanged or strictly more correct. Covered by the existing `tst_dialing_blocks` calibration tests plus the new resolution assertions.
- **[Variant sections lose the umbrella's full shared prose]** → Each variant section restates or cross-references the shared "DO NOT flag" behavioral notes so no false-positive suppression is lost; the base section remains the canonical home of the shared mechanics.
- **[Two docs can drift again]** → Tasks include an explicit sync step and a grep check that the `## D-Flow` heading count and inferred-table variant rows match between the resource and the design doc.
- **[LLM may still over-generalize "same family"]** → The new explicit "same family ≠ same grind across a pressure-target change" caveat in "How to use this ordering" directly counters this, with the D-Flow vs D-Flow/Q example called out by name.
- **[`ProfileExpectation` struct ships code-unconsumed until the deferred detector lands]** → The *struct* is unread by code, but it is not dead value: its qualitative prose twin is consumed by both AI surfaces from day one (D10), so the user-visible benefit lands now. The struct is the same proven additive pattern as `ugs` (which also shipped before all consumers existed), is optional/absent on every non-seeded section, and has a parse-contract regression test so it cannot silently rot. The deferred detector change owns code consumption + corpus regression; this change stops at the structural seam.
- **[LLM derives a rigid rule from the `Expected*:` tokens in section text — the ADR failure mode]** → The "How to read structured fields" teaching clause (D10) explicitly frames expectations as directional, intentional-variability-normal, taste-outranks-curve; values for variable profiles are pattern-encoded (`6-9 declining`) not min/max; and the spec has a scenario asserting the prompt carries that clause. Same mitigation shape the spec already relies on for `tastingFeedback` / `beanFreshness`.
- **[MCP and advisor surfaces present divergent guidance]** → Both render the same KB section content via the existing injection path; the spec adds a scenario asserting the per-profile guidance is the same on both surfaces. No separate copy is authored.
- **[Expectation values for variable-by-design profiles could be mis-encoded as tight bands]** → Per D8 they are written as explicit patterns/wide ranges (`6-9 declining`, grind-adaptive wide band) so the data encodes intentional variability; the parse test asserts a pattern value round-trips without being coerced to a degenerate min==max range.
- **[Seeded data could carry a transcription error]** → Only 5 sections are seeded; every value is traceable to an existing source-tagged line in `docs/PROFILE_KNOWLEDGE_BASE.md` (D12), the seed task records the source line beside each value, the coverage test asserts presence for exactly the 5, and no value is invented (the only inferred number is the D-Flow/Q UGS, justified in D2). The validate-then-scale cut keeps this change's diff small and its data trustworthy.
- **[Initial 5 might prove the capability is *not* valuable]** → That is the explicit purpose of the gated follow-up: the 5 are a deliberately diverse probe. If review/usage shows the expectation surface is noise, the follow-up is simply not done and the 5 cost little; if valuable, the follow-up scales it with the boundary already proven. Nothing in the general schema/parser/teaching surfaces depends on the seed count.

## Migration Plan

Resource/text (the D-Flow split + `Expected*:` seeded on the initial 5 sections + per-section prose) + one additive parser field + one system-prompt teaching clause + tests. No DB migration, no storage/API surface, no rollout gating. The `ProfileExpectation` parse is additive and code-unread, so it changes no runtime behavior; the only behavioral change is the LLM seeing richer per-profile prose plus the directional teaching clause on both surfaces. Rollback is a single revert of the commit with no state to unwind. Verification is the existing Qt Test suite (must stay green) plus the new `tst_dialing_blocks` assertions for the UGS contract, the expectation-parse contract, and seed coverage; spot-check by reading the rendered `dialing_get_context` KB section and the in-app advisor prompt for the new headings, the corrected tables, the seeded `Expected*:` lines + prose across the initial 5, and the teaching clause.

## Open Questions

- Should D-Flow/Q's inferred ~1.0 be revisited once enough real same-bean Q-vs-base shot pairs exist in user history to measure the actual offset? (Tracked as a future tuning task, not a blocker — the contract is the ordering, not the literal.)
- Final `Expected*:` value grammar (deferred, scoped by D9): the eventual grammar will need a **two-sided diagnostic** — a centred target plus directional `too-coarse:` / `too-fine:` signatures — a pressure-**shape qualifier** (`declining` / `flat` / `rising-bad` / `crash`), a **preinfusion-dripping** grams figure with its own bounds (Londinium ">17g = too coarse"), and a flow-stability qualifier (LRv2 "flow does not dip"). The per-profile **suppression catalogue** (D9) must travel with it. This change fixes only the minimal tokens the three D-Flow variants need; the deferred detector change extends the grammar deliberately. The parse contract ("populated when present, absent when not, malformed degrades to absent, pattern not collapsed to min==max") lets the grammar grow without a spec break, so none of this is added now.
- Which detector surface consumes it (a new soft advisory `summaryLines` entry vs. an existing badge path) — owned by the deferred `shot-analysis-pipeline` change, explicitly not decided here.
