## Why

GitHub issue #1160 (parent #1147, reporter @fredphoesh): the Profile Knowledge Base collapses every D-Flow variant — base D-Flow, D-Flow/Q, Damian's Q, LRv2, LRv3, LM Leva, La Pavoni — into one `## D-Flow` section carrying a single `UGS: 0.5`. The parser strips the section's parenthetical, so the "LRv2/LRv3 at 0, Q at 0.5" nuance never reaches code. The AI therefore reads every variant as grind-equivalent and transfers a grinder anchor 1:1 across profiles with materially different pressure targets. D-Flow/Q uses a 6-bar approach; base D-Flow peaks ~9 bar. The authoritative UGS chart (videoblurb.com/UGS calculator source, read verbatim) has exactly one D-Flow entry (base, 0.5) and "Londinium / LRv3" at 0 — D-Flow/Q, Damian's Q, LRv2, LM Leva are not on the chart at all. The chart's own two 6-bar profiles (Extractamundo 1.5, Gentle & Sweet 2) sit a full 1.0–1.5 UGS coarser than base D-Flow, confirming the reporter's mechanism: lower pressure target → coarser grind.

The family abstraction is correct for *behavioral* guidance (the shared "don't flag declining pressure / the soak / the LRv2 flow-switch" notes) and the UGS architecture decision record explicitly warns against per-variant JSON thresholds. It is too coarse only for the *quantitative* surface — UGS, pressure, fill temperature. The fix splits the data along pressure/temperature fault lines while keeping behavior grouped.

Investigating the fix surfaced a broader, general gap: the KB has rich free-text "Official dial-in target" / "Grind diagnostics" prose for the well-documented profiles (Adaptive v2, Londinium, Blooming, E61, Allongé, Gentle & Sweet, the D-Flow family) but nothing machine-readable, so neither AI surface nor any future deterministic check can leverage it consistently. This change therefore also introduces a **general, profile-agnostic** expected-dial-in capability — schema, parser seam, cross-surface consumability, directional-reading teaching, living documentation — and seeds it for an **archetype-diverse initial 5** to confirm value before bulk population; the rest follow in a gated follow-up. The D-Flow UGS split is the #1160 instance of the same "the KB must be per-profile accurate" principle; it is not the whole change.

## What Changes

- Split the single `## D-Flow` section in `resources/ai/profile_knowledge.md` into three sections along pressure/temperature fault lines:
  - `## D-Flow` — base ~9-bar / 88°C family. `UGS: 0.5` (canonical, confirmed by chart). Aliases: D-Flow / default, D-Flow / La Pavoni, Damian's D-Flow, Damian's LM Leva (~8-bar/88°C, inferred, behaviorally and quantitatively close to base). Retains all shared "DO NOT flag" behavioral guidance and `AnalysisFlags: flow_trend_ok`.
  - `## D-Flow Q variant` — 6-bar approach, 84°C fill, medium-light. `UGS: ~1.0` (inferred — between base 0.5 and the chart's 6-bar profiles at 1.5; the low fill temp pulls slightly back finer). Aliases: "D-Flow / Q", "Damian's Q". Section title deliberately omits `" / "` because the parser splits titles on that separator (a `## D-Flow / Q` title would register the key `d-flow` and collide with the base section); aliases are carried via `Also matches:`.
  - `## Damian's LRv2 / LRv3` — pure Londinium-R lever simulations. `UGS: 0` (LRv3 == chart's canonical "Londinium / LRv3"); note LRv2 trends slightly coarser than LRv3. Carries lever-decline behavior plus the LRv2 flow-control-switch and LRv3 9-bar-hold specifics.
- Rewrite the two cross-profile ordering tables so the AI's textual reasoning matches the split: keep the canonical table's base D-Flow row at 0.5, and replace the inferred-table row `~0–0.5 | Damian's LRv2, LRv3, LM Leva, Q | … treat as the same family.` (the single line that tells the AI they are grind-equivalent) with per-variant rows.
- Add an explicit "same named family ≠ same grind when the pressure target differs — never transfer a grinder anchor 1:1 across a pressure-target change" caveat to the "How to use this ordering" section, citing D-Flow ~9 bar vs D-Flow/Q 6 bar.
- Re-anchor the 80's Espresso inferred-table rationale off **base D-Flow** instead of off D-Flow/Q (currently inferred-anchored-on-inferred — the exact fragility #1160 calls out). The 80's Espresso UGS *value* is unchanged.
- Mirror the equivalent edits in `docs/PROFILE_KNOWLEDGE_BASE.md` and keep it current as the **living synthesis layer**: document the new optional `Expected*:` directive convention next to the existing `UGS:` / `AnalysisFlags:` field-line docs, and record the structural guidance this work surfaced — the two-sided diagnostic shape (target + too-coarse/too-fine + pressure-shape qualifier + preinfusion-dripping bounds) and the per-profile suppression catalogue. The doc is updated as we learn what is important, not just read from.
- Add a general, profile-agnostic, **optional** structured-expectation schema to the KB: per-section directive lines `ExpectedPressureBar:`, `ExpectedFlowMlS:`, `ExpectedTimeSec:` parsed into an optional `ProfileExpectation` struct on `ShotSummarizer::ProfileKnowledge` — the same additive parser pattern the existing `ugs` / `ugsInferred` fields established. The schema formalizes the free-text `Expected curves:` prose that already recurs across many profile sections. It is **not D-Flow-specific**: any section may carry it, and a section without it parses and behaves exactly as today.
- **Seed an archetype-diverse initial set of 5**, chosen to validate the schema across every distinct extraction shape before bulk population: `Adaptive v2` (grind-adaptive), `Damian's LRv2/LRv3` (declining-pressure lever — also the #1160 section, so zero marginal cost), `Blooming Espresso` (bloom + hard never-max bound), `E61`/Flat 9 Bar (flat constant), `Allongé`/Rao Allongé (high-flow turbo). Every seeded value is traceable to that profile's existing source-tagged "Official dial-in target" / "Grind diagnostics" line in `docs/PROFILE_KNOWLEDGE_BASE.md` (Decent blogs/videos, official Decent guide, EAF, Scott Rao, community index); nothing invented. The remaining well-documented profiles (base D-Flow, D-Flow/Q, standalone `Londinium`, `Gentle & Sweet`) and the long tail stay valid with an **absent** expectation and are populated by a **separate follow-up change, gated on confirming this is a valuable addition**. The general schema/parser/consumability/teaching surfaces are fully delivered here regardless — only the count of seeded data is the validation set.
- Make the enriched guidance **consumable by both AI surfaces**. Every seeded section carries its pressure / flow / time and grind-direction expectation as **qualitative prose**, not solely terse `Expected*:` tokens. Because KB section content already ships verbatim into both the in-app advisor system prompt and the `dialing_get_context` MCP response, this guidance reaches both paths with no divergence. The `shotAnalysisSystemPrompt` "How to read structured fields" section gains a clause teaching that per-profile expectations are **directional context, not a hard pass/fail rule** — intentional in-profile variability (declining pressure, grind-adaptive pressure, etc.) is normal, and taste (too sour → finer, too bitter → coarser) outranks any expectation-vs-actual comparison. This mirrors the existing directional gating taught for `tastingFeedback` / `beanFreshness` and keeps the LLM path ADR-safe (the rejected March-2026 design was a rigid min/max band fed as a rule; this is framed qualitative prose plus a teaching clause). No code consumes the structured struct in this change; the LLM consumes the qualitative prose twin, as it already does for all KB content.
- Add regression coverage in `tests/tst_dialing_blocks.cpp` asserting (a) the UGS content contract: `ugsForKbId("d-flow / q")` is strictly coarser than `ugsForKbId("d-flow / default")`; Damian's LRv3 resolves to UGS 0; D-Flow/Q resolves to a canonical name distinct from base D-Flow; variant titles resolve to expected kbIds via `computeProfileKbId` / `canonicalNameForKbId`; (b) the expectation-schema parse contract: a seeded section exposes a populated `ProfileExpectation`; an un-seeded section exposes an absent one and is otherwise byte-identical; malformed values degrade to absent rather than throwing; a variability pattern is not collapsed to min==max; and (c) seed coverage: each of the five initial profiles (Adaptive v2, Damian's LRv2, Blooming Espresso, E61, Allongé) resolves to a present `ProfileExpectation`, while a deferred well-documented profile (base D-Flow) resolves to an absent one.

The only production code change is the additive parser field: `loadProfileKnowledge()` gains an optional `ProfileExpectation` parse + a struct member on `ProfileKnowledge`, mirroring the existing `ugs` precedent. No code reads the struct; the only "consumer" enabled now is the LLM reading the enriched prose on both AI surfaces (already the case for all KB content) plus the new directional-reading teaching clause. Nothing else changes. The parser already supports multiple `## ` sections and the `~`-inferred convention; the only consumer of `ugsForKbId` (the grinder-calibration anchor path) already excludes variant profiles by canonical-name mismatch and continues to work correctly under the split. **The deviation detector that would consume `ProfileExpectation` (in `ShotAnalysis`) is explicitly out of scope here** — it is a deferred follow-up against the `shot-analysis-pipeline` capability and, per the UGS ADR, must AND-gate against the existing `AnalysisFlags` / `flow_trend_ok` suppression so it cannot reproduce the March 2026 false-positive.

## Capabilities

### New Capabilities
None.

### Modified Capabilities
- `dialing-context-payload`: the capability that owns "ProfileKnowledge SHALL expose UGS as a parsed numeric field", the relative-grinder-setting / anchor-selection requirements, and the `shotAnalysisSystemPrompt` "How to read structured fields" teaching section. Four ADDED requirements, general-first: (1) `ProfileKnowledge` SHALL expose an optional, **profile-agnostic** structured `ProfileExpectation` parsed from `Expected*:` directive lines, absent by default, malformed-degrades, variability-preserving; (2) an **archetype-diverse initial 5** (Adaptive v2, Damian's LRv2/LRv3, Blooming Espresso, E61, Allongé) SHALL be seeded with traceable `Expected*:` values, the remaining well-documented profiles deferred to a gated follow-up and staying absent; (3) per-profile expected guidance SHALL be carried as qualitative prose, identical across the in-app advisor and `dialing_get_context` surfaces, with the system prompt teaching directional, taste-deferring reading; (4) the #1160 instance — the KB SHALL assign distinct UGS positions to pressure-target-distinct D-Flow variants. The UGS parser behavior, the RGS anchor-selection algorithm, and the canonical/inferred `source` semantics are unchanged. The structured struct's *code consumption* (a deviation detector) is deferred to a future `shot-analysis-pipeline` change and is not specified here; only parsing, the seeded data, and the LLM-facing prose/teaching surface are in scope.

## Impact

- **Data / resources**: `resources/ai/profile_knowledge.md` — the `## D-Flow` section split into three; both cross-profile tables; the "How to use" caveat; the 80's Espresso rationale re-anchor; `Expected*:` schema seeded on the **initial 5** (Adaptive v2, Damian's LRv2/LRv3, Blooming Espresso, E61, Allongé) with per-section qualitative expectation prose. Shipped via Qt resource into both AI paths.
- **Code**: one additive parser change in `src/ai/shotsummarizer.{h,cpp}` — a `ProfileExpectation` struct member on `ProfileKnowledge` plus its parse in `loadProfileKnowledge()`, mirroring the existing `ugs` field. Plus the `shotAnalysisSystemPrompt` "How to read structured fields" section gains the directional-expectation teaching clause. No new struct consumer; no change to existing call sites.
- **AI surfaces**: both the in-app advisor system prompt and the `dialing_get_context` MCP KB surface present the enriched per-profile guidance for every seeded profile (same KB content → no divergence) plus the new teaching clause. This is the existing KB-injection path; no new plumbing.
- **Docs**: `docs/PROFILE_KNOWLEDGE_BASE.md` is kept current as the living synthesis layer — mirrored split/table edits, the new `Expected*:` directive convention documented next to the existing field-line docs, and the two-sided-diagnostic + per-profile-suppression structural guidance recorded. `docs/UNIVERSAL_GRIND_SETTING.md` reviewed; the ADR's "keep the KB qualitative — do not formalize into rigid JSON rules" item gets a clarifying note that an optional, deterministic-detector-gated structured field plus a qualitative LLM-facing prose twin is consistent with that decision (the rejected design was a rigid min/max band fed to the LLM as a rule).
- **Tests**: new assertions in `tests/tst_dialing_blocks.cpp` for the UGS contract, the expectation-schema parse contract, and seed coverage across the well-documented set. No DB migration, no schema/storage change, no API change.
- **Behavioral**: the AI now sees D-Flow/Q as coarser than base D-Flow and LRv3 as finer (#1160), and reads an explicit per-profile expected dial-in signature for every well-documented profile on both AI surfaces (directional, taste-deferring). Behavioral false-positive suppression (declining pressure, soak, flow-switch, grind-adaptive pressure) is preserved via the per-profile suppression notes and the directional-reading teaching clause.
- **Closes**: tracked under #1147; #1160 stays open with a plain reference (cc @fredphoesh) until the reporter confirms — not auto-closed.
- **Out of scope**: the deviation detector that consumes `ProfileExpectation` (a future `shot-analysis-pipeline` change — must AND-gate against existing `AnalysisFlags`/`flow_trend_ok` suppression *and* the confounders Decent's own dial-in guide enumerates: channeling/poor puck prep, unknown/very-fresh beans, grinder fineness limits, and by-design grind-tolerant profiles; must prefer first-party per-profile numeric targets over generic graph heuristics, which the EAF profiling guide explicitly warns are unreliable; soft advisory only, taste leads); populating `Expected*:` beyond the initial 5 — the remaining well-documented profiles (base D-Flow, D-Flow/Q, standalone Londinium, Gentle & Sweet) and the long tail are a **separate follow-up change gated on confirming the capability is valuable**; all stay valid with an absent expectation until then; reconciling the pre-existing drift between the `dialing-context-payload` spec's 80's Espresso example value (`canonical UGS 0.25`) and the KB's current `~-0.5`; the fuzzy-prefix matcher weakness for custom user-titled variant profiles that do not exactly match an alias (pre-existing; built-in profiles match exactly and are unaffected); the remaining #1147 sub-issues (#1159).
