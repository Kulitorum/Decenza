## Context

`profile-auto-load` (shipped, `openspec/specs/profile-auto-load/spec.md`) pins a single profile filename in `Settings.app.autoLoadProfileFilename` (empty string = disabled), plus a shared `Settings.app.autoLoadRevertMinutes` (0..60, clamped). `ProfileManager::loadAutoLoadProfileIfNeeded()` (`src/controllers/profilemanager.cpp:993`) is the single entry point: no-op if unconfigured, self-clears with `autoLoadStaleCleared()` if the filename no longer resolves to a Selected-list profile, otherwise calls `loadProfile()`. Three call sites in `qml/main.qml` invoke it: app startup (`:931`), DE1 `Sleep → Idle` wake (`:495-508`, guards against BLE-reconnect false-fires), and a dedicated 1-minute countdown `Timer` (`:441-458`) that only decrements while the Idle page is shown and no machine operation is active, independent of the auto-sleep timer. UI surfaces: a pin icon beside the profile title on the auto-load row, an "Auto-load:" status strip at the top of `ProfileSelectorPage`, and a Set/Disable action inside a modal overflow `Dialog` (not inline on the card) — no confirmation step either way. Three MCP tools (`profiles_get/set/clear_auto_load`) mirror this in `src/mcp/mcptools_profiles.cpp` and `mcptools_write.cpp`.

Recipes are DB rows (`src/history/recipestorage.h/.cpp`, `recipes` table), identified by `qint64 id`, not files. The existing single-select-by-int-id pattern already used for `SettingsDye::activeBagId`/`activeRecipeId` (`src/core/settings_dye.cpp:604-623`) is the natural fit for a recipe-side auto-load key, in place of the filename string the profile side uses. Recipe activation is a single cascading entry point, `MainController::activateRecipe(qint64 recipeId)` (`src/controllers/maincontroller.cpp:1202`), which applies profile → bag → equipment → grind → dose → yield/temp → steam → hot-water → `activeRecipeId` in one pass; auto-loading a recipe should reuse this path unchanged rather than duplicating any of its cascade logic.

The recipe card (`qml/pages/RecipesPage.qml:464-538`) already has a `footer: Flow` action row (Edit → Clone → Archive/Delete → Restore) supplied to the shared `RecipeDrinkCard.qml`. This is where the new button lands, per the user's request ("to the left of edit") and the confirmed answers: reuse `pin.svg`, no confirmation dialog, share the existing revert-minutes timer.

## Goals / Non-Goals

**Goals:**
- Let exactly one of {a profile, a recipe} be the auto-load target at any time, enforced by mutual clearing, not by validation-and-reject.
- Reuse every existing mechanism that already generalizes: the three trigger call sites, the revert-minutes timeout, the stale-target self-clearing behavior, the MCP tool shape, and recipe activation's existing cascade.
- Keep the recipe-side card interaction lighter than the profile page's (no overflow dialog) since the user explicitly asked for a single card button — a toast is enough feedback for an immediate toggle, plus a compact on-card status row for the one card that is the current target (see Decisions).

**Non-Goals:**
- No page-level or toolbar status strip on `RecipesPage` — the `ProfileSelectorPage` strip pattern (a persistent banner above a filter row, or a compact version anchored to the search bar) was considered and rejected twice in favor of putting the status directly on the card itself; see Decisions.
- No new coordinator/dispatcher class unifying "auto-load target" into one polymorphic concept. The two settings stay separate properties on their existing domain objects (`SettingsApp`, `SettingsDye`); the mutual exclusion is two setters each clearing the other's key, which is simpler than introducing an abstraction for a two-member set.
- No change to how recipe activation itself works (`activateRecipe`'s cascade, bag/equipment resolution, same-id re-activation shortcut) — auto-load only decides *when* to call it, never *how*.

## Decisions

**Storage: `SettingsDye::autoLoadRecipeId` (int, sentinel `-1`), not `SettingsApp`.** `activeRecipeId`/`activeBagId` already live in `SettingsDye` (`src/core/settings_dye.h/.cpp`) since recipes/bags are DYE-domain concepts; keeping the new key beside them avoids splitting recipe-related settings across two domain objects for no reason. This does mean the two auto-load keys live in different `Settings*` sub-objects (`app` for the profile filename, `dye` for the recipe id) — acceptable, since each key belongs with its existing domain, and the cross-clear is a two-line call from one setter into the other's setter, not a shared-storage requirement.

**Mutual exclusion lives in the setters, not the callers.** `SettingsApp::setAutoLoadProfileFilename(filename)`, when `filename` is non-empty, calls `settings->dye()->setAutoLoadRecipeId(-1)` (needs a `Settings*` back-reference or a signal `MainController`/`ProfileManager` connects across, mirroring how `SettingsDye::setActiveBagId` already reaches into `SettingsDye`'s own bean fields as a clearing side-effect at `src/core/settings_dye.cpp:619-629`). Symmetrically, `SettingsDye::setAutoLoadRecipeId(recipeId)`, when non-negative, must clear `autoLoadProfileFilename`. Doing the clear inside the setter (rather than at each MCP/QML call site) means every path that sets either value — MCP tools, the new recipe card button, any future UI — gets the exclusion for free and can't forget it. Alternative considered: validate-and-reject in each call site (like `profiles_set_auto_load`'s existing "not in Selected list" rejection) — rejected because the user's confirmed answer was "switch silently + toast," which is a clearing side-effect, not a validation error.

**Two independent trigger calls at each site, not a unifying dispatcher.** `qml/main.qml`'s three trigger sites (`:453`, `:506`, `:931`) each currently call `ProfileManager.loadAutoLoadProfileIfNeeded()`. This change adds a second call, `MainController.loadAutoLoadRecipeIfNeeded()`, at all three sites. Each entry point independently no-ops when its own setting is empty/`-1`, so calling both is safe and — because the setters guarantee mutual exclusion — at most one of the two ever does real work at a given trigger. Alternative considered: a single `AutoLoadCoordinator::performAutoLoad()` that checks both settings and dispatches to the right one — rejected as an unnecessary abstraction layer for two call sites that are already individually no-op-safe; it would also need to duplicate the existing BLE-reconnect false-fire guard and idle-countdown gating that already live correctly in `main.qml` for the profile path.

**Recipe eligibility: not archived, still exists.** Mirrors "Selected-list" eligibility on the profile side (`isProfileInSelectedList`). `MainController::loadAutoLoadRecipeIfNeeded()` checks `m_recipeStorage` for the recipe's current `archived` flag (see `recipestorage.h:99`) before activating; if the id doesn't resolve or resolves to an archived recipe, clear `autoLoadRecipeId` and emit a `autoLoadRecipeStaleCleared()` signal (mirroring `ProfileManager::autoLoadStaleCleared()`) so `main.qml`'s existing toast wiring (`:4172`) can show the recipe-flavored message. `RecipeStorage::requestArchiveRecipe`/`requestDeleteRecipe` additionally clear `autoLoadRecipeId` proactively at the point of archiving/deletion (mirroring `ProfileManager::deleteProfile`'s hidden/deleted cleanup), rather than waiting for the next trigger to discover the staleness — this matches the immediate-feedback expectation set by the card button itself (toggle off, or the button visibly changing) rather than leaving a dangling reference until the next app start.

**Card button, no dialog.** Unlike the profile page (overflow `Dialog`, because the profile row doesn't have room for another inline icon), the recipe card's `footer: Flow` already holds 2-4 `StyledIconButton`/`AccessibleButton` siblings at `Theme.scaled(36)` — a fifth `StyledIconButton` using `pin.svg`, `visible: !card.archivedCard`, inserted before the existing Edit button, matches both the user's placement request and the established sizing/spacing convention (`spacing: Theme.scaled(6)`). Active state: tint via `Theme.primaryColor` (matching the profile row's pin-icon treatment) when `card.recipe.id === Settings.dye.autoLoadRecipeId`; outline/untinted otherwise. Tap toggles immediately (set if not active, clear if active) and shows a toast — no modal, consistent with the confirmed "switch silently + toast" answer and the fact the card already has room, unlike the profile row.

**On-card status row, not a page-level or toolbar strip.** Two other placements were considered and rejected before landing here. First, porting `ProfileSelectorPage`'s full status strip (`ProfileSelectorPage.qml:44-173`) to the top of `RecipesPage` as its own banner — rejected because a recipe card already carries far more identifying content (image, name, category, stats) than a profile row does, so a second banner naming the same recipe again is pure redundancy, and `RecipesPage` shows many cards at once rather than one selectable row. Second, a compact version of the same strip anchored to the far right of the search/sort toolbar row — also rejected, for the same redundancy (it would repeat the auto-loaded recipe's name) plus crowding an already-tight single-line toolbar. Landing the indicator on the card that **is** the auto-load target sidesteps both problems: the card's own title already supplies the name, so the row only needs to add what the title doesn't say — that this specific card is the auto-load target, spelled out in words (not just the pin tint) because, per explicit user feedback, a bare pin icon alone doesn't communicate its meaning to someone unfamiliar with the convention. Content: a small pin icon + an "Auto-load" text label + the shared revert-minutes `ValueInput` stepper (same 0..60 range, live `onValueModified` write, "off" display at 0, matching `ProfileSelectorPage.qml:117-149`). No recipe name (redundant with the card title) and no clear/× button (redundant with the toggle button from the "Card button, no dialog" decision above, which already clears auto-load on a second tap) — trimming both keeps the row to the one piece of information and one piece of functionality the rest of the card doesn't already provide. Implementation: an additional item in the card's `footer: Flow` (`RecipesPage.qml:466`), `visible: card.recipe.id === Settings.dye.autoLoadRecipeId`, wrapping naturally alongside the action buttons since `Flow` already handles overflow — no new row or layout container needed.

**MCP naming: `recipe_get_auto_load` / `recipe_set_auto_load` / `recipe_clear_auto_load`**, singular `recipe_` prefix, matching every existing recipe tool (`recipe_activate`, `recipe_archive`, `recipe_clone`, `recipe_create`, `recipe_get`, `recipe_list`, `recipe_update` — all singular), rather than mirroring the profile side's plural `profiles_*_auto_load` naming. Each tool's error semantics mirror the profile trio: `recipe_set_auto_load` validates `recipeId` is present, resolves to an existing non-archived recipe, before hopping to the GUI thread to write the setting (same `QMetaObject::invokeMethod(qApp, ..., Qt::QueuedConnection)` pattern as `mcptools_write.cpp:1477`); `recipe_clear_auto_load` clears only the recipe id, leaving `autoLoadRevertMinutes` untouched, matching `profiles_clear_auto_load`'s "preserve the timeout" behavior. Tool descriptions on both the recipe trio and the existing `profiles_set_auto_load`/`profiles_clear_auto_load` are updated to state the cross-clear explicitly, since an MCP client scripting "set recipe X as auto-load" needs to know it silently disables a previously-configured profile auto-load.

**Shared revert timer, no new setting.** Per the confirmed answer, `autoLoadRevertMinutes` is read by both entry points unchanged — no new `SettingsDye`/recipe-side revert-minutes key. This also means the existing `qml/main.qml` idle-countdown `Timer` (`:441-458`) and its `autoLoadCountdownReload()` gating function need their empty-check widened from `Settings.app.autoLoadProfileFilename === ""` to also treat `Settings.dye.autoLoadRecipeId === -1` as "nothing configured," and the `Connections { target: Settings.app; ... }` block needs a parallel `Connections { target: Settings.dye; function onAutoLoadRecipeIdChanged() { root.autoLoadResetCountdown() } }` so changing the recipe target resets the countdown exactly like changing the profile target already does.

## Risks / Trade-offs

- **[Risk]** Two settings in two different domain objects (`SettingsApp` vs `SettingsDye`) each need a live reference to the other to enforce the cross-clear, which is a small layering wrinkle (domain sub-objects are meant to stay decoupled per `SETTINGS.md`). → **Mitigation**: `Settings` (the façade) already owns both sub-objects and can wire a direct signal/slot connection between them at construction time (e.g. `connect(app(), &SettingsApp::autoLoadProfileFilenameChanged, dye(), ...)`) instead of giving each sub-object a pointer to the other — keeps the two domain classes themselves ignorant of each other, with the coupling confined to `Settings`'s own wiring code.
- **[Risk]** Adding a second no-op-checked call at each of the three `main.qml` trigger sites doubles the per-trigger settings reads (cheap, but worth noting) and means a future third auto-load-able entity (if ever) would need a third call at every site — a smell that the "two independent calls" decision could stop scaling. → **Mitigation**: acceptable at N=2; if a third target type is ever proposed, that's the trigger to revisit the coordinator-abstraction alternative rejected above, not before.
- **[Risk]** Proactively clearing `autoLoadRecipeId` on archive/delete needs to run on the correct thread (recipe storage work happens off the main thread). → **Resolved during implementation**: rather than adding a clearing hook inside `RecipeStorage` itself, `MainController` hooks the existing `recipeUpdated`/`recipeDeleted` signals — the same ones it already uses for the active-recipe cache refresh — which `RecipeStorage::runAsync` (`recipestorage.cpp:271-282`, via `SerialDbWorker`) already marshals back to the GUI thread before emitting. No new thread-hop code was needed.

## Migration Plan

No data migration: `autoLoadRecipeId` defaults to `-1` (QSettings key absent), and no existing installs have a value to migrate. Ship as a normal feature release; no rollback concerns beyond a standard revert (removing the setting key is harmless — `-1` default behavior is unaffected by having stale data from a rolled-back build, since the key would simply go unread again).
