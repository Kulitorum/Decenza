# Recipes

A **recipe** is the whole drink: profile + bean link + equipment package + dose/yield/temperature + grind routing + steam block + hot-water block + a drink type + a name. It is the fourth rung of the optionality ladder — profile alone always pulls a shot; beans, equipment, and recipes are strictly additive. No flow ever requires a recipe, and a recipe works with whatever rungs the user tracks (bean-less and equipment-less recipes are valid). **Profile is required unless the recipe carries a hot-water block with `hasWater` true** (a profile-less hot-water TEA recipe, add-recipe-wizard-tea) — the shared rule is `Recipe::saveValidationPasses`, enforced identically by the wizard, MCP, and web.

OpenSpec changes: `add-recipes` (archived) and `add-recipe-wizard-tea`. The promoted specs under `openspec/specs/` are the behavioral source of truth.

## Data model (`src/history/recipestorage.{h,cpp}`)

- `recipes` table in the shot-history DB (migration 25; `hot_water_json` added migration 27; `drink_type` added migration 28; `bag_id` added migration 29), managed by `RecipeStorage` — a structural clone of `CoffeeBagStorage` (kCols single-source column table, async request/ready signals, `SerialDbWorker`).
- **Recipes link a SPECIFIC bag, not a bean** (recipes-bag-links-ui-polish — this reversed the original bean-level rule): `bag_id` is the hard link; activation applies exactly that bag, never an MRU resolution (users running two bags of one bean at different ages were silently landing on the wrong bag). The bean identity fields (`beanbase_id`, roaster+coffee) are retained as the display fallback and the **matching key for automatic relinking** — never an activation input. `resolveOpenBagStatic()` survives only as the relink matching helper and the one-time migration-29 data pass (which resolved each existing recipe's bean identity to its then-open bag; recipes with no open bag migrated with `bag_id` NULL → stale). `requestCreateRecipe` normalizes at save time: bean identity without a bag resolves to that bean's open bag once (older MCP/web clients); a bag without identity adopts the bag's identity fields.
- **Relink lifecycle (recipe-bag-lifecycle): silent, event-driven, zero options.** *Roll-on-finish*: `CoffeeBagStorage::bagFinished` (emitted by any update carrying `inInventory=false`) → `requestRelinkForFinishedBag` moves the finished bag's non-archived recipes to the **newest open** bag of the same bean identity (id DESC — freshest, not MRU). *Wake-on-restock*: `bagCreated` → `requestRelinkForRestockedBag` relinks stale recipes matching the new bag's identity, MRU-first (twins wake one at a time). Both apply the **dup-guard**: a relink is skipped when the target bag already carries a non-archived recipe with the same (profile title, drink type) — the one rule that protects deliberate two-bag comparison pairs; dose/yield/temp differences do NOT make recipes distinct. Every move emits `recipesRelinked(movedIds, targetBagId, targetBagName)` → the courtesy toast in main.qml; MainController refreshes the active-recipe cache when it moved. No dialogs, no settings, no timers.
- **Stale = linked bag not in inventory** (or a bean identity with no link at all). Never stored — computed by the inventory query (`stale` in the QML maps, `bagStale` on MCP/web reads) — and **never a gate**: a stale recipe lists, activates fully (the finished bag's grind is applied — its last dial beats nothing), and its card shows "Bag finished — tap to choose beans" (opens a bag picker scoped to the recipe → `requestRelinkRecipeToBag`, which is just a `bagId` update: the bag's identity follows, grind pin/inherit is untouched). Idle pills dim (`dimmed` preset flag in `PresetPillRow`).
- **Grind: inherit by default, pin by exception.** `grind_pinned` empty = the recipe follows its **linked bag's** grind (re-dials update all sibling recipes of that bag; sibling bags of the same bean are independent). Non-empty = the recipe's private dial. Bean-less recipes keep grind in the pin by construction. Grind text is opaque (free-form strings — no deltas). Relinks never rewrite grind.
- **Steam block** (`steam_json`, compact JSON): `{hasMilk, milkWeightG, pitcherName, durationSec, flow, temperatureC}`. The pitcher is snapshotted **by value** (snapshot-not-reference, like Bean Base); activation re-selects the preset by name and *recreates it from the snapshot if deleted*.
- **Hot-water block** (`hot_water_json`, compact JSON): `{hasWater, vesselName, volume, mode, flowRate, temperatureC, order}` — the opt-in added-hot-water that makes an **Americano** (or **long black**) a recipe. `order` is `"after"` (americano, default) or `"before"` (long black). Hot water is vessel-carried: the selected **water vessel** supplies amount/temperature/flow — there is no separate per-recipe amount. The vessel is snapshotted **by value** like the pitcher (activation re-selects by name, recreates from the snapshot if deleted). Composer stores a full snapshot; the web form stores a name-only block and activation resolves the live vessel's values (`applyActivatedRecipe`). Independent of the steam block — a recipe may carry either, both, or neither. No heater hold (hot water needs no pre-warm), so a milk-less hot-water recipe never lights the steam heater.
- **Drink type** (`drink_type`, add-recipe-wizard-tea): user intent recorded by the wizard — `espresso | filter | americano | long_black | latte | tea | tea_hotwater`. Presentation only (pill icons, wizard re-entry, MCP/web field): activation and machine behavior read the blocks, never this. Empty (legacy rows) = derive via `Recipe::deriveDrinkType(recipe, profileBeverageType)` — precedence: profile-less+water → tea_hotwater; tea_portafilter profile → tea; milk → latte (milk wins over added water); water order → americano/long_black; filter/pourover → filter; else espresso. `requestUpdateRecipe` re-derives it when a block/profile field changes without an explicit drinkType (stored tea/filter categories survive block stamps — see the bev-synthesis comment there). Installed profiles embed no profile JSON, so their `beverage_type` resolves from the catalog: `ProfileManager::beverageTypeForTitle` (surfaces, main thread) / `beverageTypeByTitleSnapshot` (captured on the main thread, passed by value into bg-thread recipe-JSON builders); MCP/web updates attach a transient `profileBeverageType` hint that `requestUpdateRecipe` strips before the column write.
- **Lifecycle mirrors bags**: 0 shots = hard-deletable; any shots = archive only (`archived` flag), so `shots.recipe_id` provenance never dangles. Hot-water-only recipes never accumulate shots (hot-water dispenses aren't shots) — always hard-deletable.
- Shots record `recipe_id` + `steam_json` + `hot_water_json` snapshots at save time (all three metadata build sites in MainController), so promote-from-shot round-trips the whole drink.
- **Transfer / backup**: `RecipeStorage::importRecipesStatic` merges recipes during device transfer and `.dcbackup` restore (called from `ShotHistoryStorage::importDatabaseStatic` after bags), remapping `equipment_id` through the package id-map, `bag_id` through the bag id-map (an unmatched source bag becomes NULL → stale; wake-on-restock re-homes it later), and `shots.recipe_id` through the recipe id-map. Dedup identity = case-insensitive name + profile_title + bean identity (unchanged — not the bag id). Recipes ride inside `shots.db` — a backup with **zero shots** skips the whole import (pre-existing early-return, same as bags/equipment).

## Activation (single path)

`MainController::activateRecipe(id)` → `RecipeStorage::requestRecipeForActivation` (one background pass: recipe + its LINKED bag loaded by `bag_id` — applied whether or not the bag is still in inventory; only a deleted bag row degrades to bean-less) → `applyActivatedRecipe`:
profile by title with stored-JSON fallback → bag via `setActiveBagKeepFields` + explicit field applies from the bundle's bag map (deterministic; no async `applyActiveBag` race) → equipment → grind routing (suspension flag set **before** the setter when pinned) → queued dose write (beats loadProfile's deferred recommended dose) → yield/temp overrides + re-upload → steam block + heater hold (`hasMilk` → `startSteamHeating("recipe-activated")` now, and `sendMachineSettings` keeps it on while the recipe is active; never fights `keepSteamHeaterOn`) → hot-water block (re-select the snapshotted vessel, push its values via `applyHotWaterSettings()`; **no** heater hold) → `activeRecipeId` last.

QML pill taps, MCP `recipe_activate`, and the web `/activate` route all call this one path. Terminal status: `recipeActivated(id, success)` (queued, lands after the dose write).

**Profile-less recipes** (hot-water tea): activation skips the profile-load, dose-write, and yield/temp-override stages entirely — bag, equipment, and the hot-water block still apply, the loaded espresso profile is untouched, and NO steam-heater hold is ever taken (`activeRecipeHasMilk()` returns false for profile-less recipes so re-sends can't re-assert one). The machine action is the user starting Hot Water.

- **Tweaks write through; ingredient swaps deactivate.** Dose/yield/temp/steam/milk edits and the hot-water vessel selection stamp the active recipe (`stampActiveRecipe`/`stampActiveRecipeHotWater`, echo-guarded like the bag pattern). Grind edits go to the bag when inherited (`SettingsDye` write-through) and to the pin when pinned (`setGrindBagWriteThroughSuspended` blocks the bag write). Changing profile/bag/equipment deactivates — the watchers compare against the recipe's **own** ingredients, so re-selecting the same thing (or startup auto-load) never deactivates.
- Active id persists in `SettingsDye` (`dye/activeRecipeId`); startup restores the cache + pin routing without re-applying values.

## Surfaces

- **QML**: `RecipesItem` idle widget (BeansItem clone — MRU pills of the last 5 with drink-type icons via `PresetPillRow`'s optional per-preset `icon`, double/long-press → `RecipesPage`), `RecipeWizardPage` (replaced `RecipeComposerPage`, add-recipe-wizard-tea: drink-type-first steps with auto-advance + breadcrumb chips; the SUMMARY step is the edit page — edit/clone/promote land there via the same `mode`/`editRecipeId`/`prefill`/`promoteShotId` contract; templates pre-seed blocks per drink type but the summary's Add/Remove milk/water buttons keep every combination expressible). Profile step filters by the profile's `beverage_type` per drink type and ranks: ① used with this bean (`requestRankedProfilesForBean`), ② KB-recommended — coffee: `roastAffinity` from the profile knowledge base (`ShotSummarizer::roastAffinityForTitle` via `ProfileManager.kbProfileSuitsRoast`; authored per-entry only where the profile's own docs state roast suitability); tea: type-matched stock profiles (`DrinkTypes::teaProfileMatchesType`) — + similar beans, capped to 5, ③ rest (tea: temp proximity to the bag's `brewTempC`). Grind hint on the details step: latest grind for the bean (any profile; same-roast fallback — `requestLatestGrindForBean`) + UGS direction via `ShotSummarizer::grindDirectionBetween` when the picked profile differs (direction only, never a computed number — the KB's cross-profile rule). Details step prefills from the latest shot with the bean+profile pair (`requestLatestShotForBeanProfile`) → tea-bag brewing data → profile defaults; portafilter tea only seeds a temp override when the profile is NOT type-matched. Equipment defaults per drink type (`requestLastEquipmentForDrinkType`). Tea profile step has a fixed "Just hot water" row → profile-less recipe. Promote buttons on ShotHistoryPage (beside Load), ShotDetailPage header, AutoFavoritesPage rows. Navigate target is `recipeList` — bare `recipes` is the pre-existing profile Recipe Editor.
- **Shared card component** (`qml/components/RecipeDrinkCard.qml`, design D4): the management-page card AND the wizard summary's WYSIWYG hero are the same component (plain properties, no page-specific lookups; a `footer` alias carries the management page's action row). Card lines: name anchor + Active badge / drink icon + SHORT type label + profile (always shown) + milk weight ("200g milk", never a bare "milk") / bag + shot count (or the stale re-point affordance) / plan line (or the vessel snapshot for profile-less hot-water tea). Lines wrap, never elide the profile. Drink-type labels/icons live in the `DrinkType` QML singleton (`qml/components/DrinkType.qml`): `shortLabel()` for cards/pills/auto-names, `longLabel()` only for the wizard's drink-type picker; `suggestName()` uses short labels and skips the type word when the bean name already ends with it.
- **MCP**: `mcptools_recipes.cpp` — see `MCP_SERVER.md` for the tool table. Register-stub gotcha applies (`tst_mcpserver_session/protocol`). Recipe reads expose `bagId` + a `bag` display object + `bagStale` (human-readable string); create/update accept `bagId`; inherited grind resolves from the linked bag.
- **Web**: `shotserver_recipes.cpp` (+ `shotserver_bags.cpp`, `shotserver_equipment.cpp`) — see `SHOTSERVER.md`.

## Gotchas

- `ShotProjection` gained `recipeId`/`steamJson`/`hotWaterJson`; the composer prefill reads them from `shotReady`.
- Tea (add-recipe-wizard-tea): bags carry a creation-time `kind` ("coffee"/"tea", migration 28) — see BEAN_BASE.md for the Add Tea flow and tea extraction. Tea recipes never store grind; equipment packages may be grinder-less (basket-only) for tea setups.
- Adding a recipe column = a kCols row in recipestorage.cpp **plus** the CREATE TABLE + a migration step (same rule as bags). A shot-snapshot column (like `hot_water_json`) also needs the `ShotRecord`/`ShotSaveData`/`ShotMetadata` structs, the shot INSERT + read SELECT, and the `ShotProjection` mapping — see the `steam_json` sites.
- The steam heater takes **5–9 minutes** to warm (measured). An active milk recipe therefore HOLDS the heater on: `sendMachineSettings` treats `activeRecipeHasMilk()` like `keepSteamHeaterOn`, so re-sends (wake/reconnect/edits) keep it warm; deactivation or a milk-less switch releases the hold. Milk-less recipes never force the heater off (keep-on users unaffected). Hot water, by contrast, has **no** heater hold.
- Device transfer / backup import now copies the `recipes` table (`importRecipesStatic`, mirroring bags/equipment). Recipes ride inside `shots.db`, so a zero-shot backup still skips them (pre-existing early-return).
