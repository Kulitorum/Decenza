# Recipes

A **recipe** is the whole drink: profile + bean link + equipment package + dose/yield/temperature + the recipe's own grind + steam block + hot-water block + a drink type + a name. It is the fourth rung of the optionality ladder — profile alone always pulls a shot; beans, equipment, and recipes are strictly additive. No flow ever requires a recipe, and a recipe works with whatever rungs the user tracks (bean-less and equipment-less recipes are valid). **Profile is required unless the recipe carries a hot-water block with `hasWater` true** (a profile-less hot-water TEA recipe, add-recipe-wizard-tea) — the shared rule is `Recipe::saveValidationPasses`, enforced identically by the wizard, MCP, and web.

OpenSpec changes: `add-recipes` (archived) and `add-recipe-wizard-tea`. The promoted specs under `openspec/specs/` are the behavioral source of truth.

## Data model (`src/history/recipestorage.{h,cpp}`)

- `recipes` table in the shot-history DB (migration 25; `hot_water_json` added migration 27; `drink_type` added migration 28; `bag_id` added migration 29; `temp_offset_c` added migration 31), managed by `RecipeStorage` — a structural clone of `CoffeeBagStorage` (kCols single-source column table, async request/ready signals, `SerialDbWorker`).
- **Recipes link a SPECIFIC bag, not a bean** (recipes-bag-links-ui-polish — this reversed the original bean-level rule): `bag_id` is the hard link; activation applies exactly that bag, never an MRU resolution (users running two bags of one bean at different ages were silently landing on the wrong bag). The bean identity fields (`beanbase_id`, roaster+coffee) are retained as the display fallback and the **matching key for automatic relinking** — never an activation input. `resolveOpenBagStatic()` survives only as the relink matching helper and the one-time migration-29 data pass (which resolved each existing recipe's bean identity to its then-open bag; recipes with no open bag migrated with `bag_id` NULL → stale). `requestCreateRecipe` normalizes at save time: bean identity without a bag resolves to that bean's open bag once (older MCP/web clients); a bag without identity adopts the bag's identity fields.
- **Relink lifecycle (recipe-bag-lifecycle): silent, event-driven, zero options.** *Roll-on-finish*: `CoffeeBagStorage::bagFinished` (emitted by any update carrying `inInventory=false`) → `requestRelinkForFinishedBag` moves the finished bag's non-archived recipes to the **newest open** bag of the same bean identity (id DESC — freshest, not MRU). *Wake-on-restock*: `bagCreated` OR `bagRestocked` (an update returning a bag to inventory) → `requestRelinkForRestockedBag` relinks stale recipes matching the bag's identity, MRU-first (twins wake one at a time). Both apply the **dup-guard**: a relink is skipped when the target bag already carries a non-archived recipe with the same (profile title, drink type) — the one rule that protects deliberate two-bag comparison pairs; dose/yield/temp differences do NOT make recipes distinct. Every move emits `recipesRelinked(movedIds, targetBagId, targetBagName)` → the courtesy toast in main.qml; MainController refreshes the active-recipe cache when it moved. No dialogs, no settings, no timers.
- **Stale = linked bag not in inventory** (or a bean identity with no link at all). Never stored — computed by the inventory query (`stale` in the QML maps, `bagStale` on MCP/web reads) — and **never a gate**: a stale recipe lists, activates fully (with its own grind — the bag's state is irrelevant to grind), and its card shows "Bag finished — tap to choose beans" (opens a bag picker scoped to the recipe → `requestRelinkRecipeToBag`, which is just a `bagId` update: the bag's identity follows, the recipe's grind is untouched). Idle pills dim (`dimmed` preset flag in `PresetPillRow`).
- **Grind: always the recipe's own** (fix-recipe-grind-integrity — the old "empty = inherit from the bag" mode is retired; migration 30 backfilled inherit-mode rows from their bag, and `importRecipesStatic` runs the same backfill for pre-migration source DBs). The linked bag's dial is read **exactly once, at creation**, as the grind field's editable default: the wizard offers it (shot history for the bean+profile pair beats it when available), and `requestCreateRecipe` adopts it when a bag-linked create map *omits* grind (MCP/web) — an explicitly empty grind stays empty. Editing grind while a recipe is active stamps the recipe AND writes through to the bag in parallel (the bag always mirrors the last dialed grind; see below). Sibling recipes never follow each other — no recipe ever reads grind from the bag after creation. Grind text is opaque (free-form strings — no deltas). Relinks never rewrite grind. Grind-less drink types (tea) store none, and a dial edit while a tea recipe is active never stamps grind onto it.
- **Yield is a SPEC, not a number** (add-yield-ratio-anchor, migration 34): `yield_value` (double) + `yield_mode` (`none` | `absolute` | `ratio`) — one value column plus an explicit discriminator, so a recipe can never hold both an absolute yield and a ratio (`src/core/yieldspec.h` is the shared vocabulary). `mode = none` = the recipe designs no yield and the resolution ladder falls through (recipe → bag → profile `target_weight`). A `ratio` value is a dose multiplier (2.0 = 1:2); its gram target is derived at use time (`ProfileManager::targetWeight()` — the ladder's single evaluation point), so a dose change moves the effective target and a stale absolute can never be manufactured — the dose analogue of the temperature offset below. **The anchor rule**: whichever of {ratio, yield} was last written is the anchor; UI surfaces show the other derived (dimmed), never blank, and only the anchor is stored. **Resolution to grams happens before `MachineState::setTargetWeight`** — `WeightProcessor`, the quality detectors, `shots.yield_override`, MQTT `target_weight`, and DYE/Visualizer only ever see grams. Shots additionally record the anchor that produced their target (`shots.yield_mode` + `yield_anchor_value` — intent alongside outcome), so promotion is a straight copy (`RecipePromotion`) and a post-shot dose correction can't mint a ratio nobody chose. A ratio session anchor **survives a profile switch** (1:2 is 1:2 on any profile; an absolute clears — `SettingsBrew::clearProfileScopedBrewOverrides`), and the dose used for ratio resolution is **latched at `espressoCycleStarted`** so a mid-shot dose write can never move the live SAW target. The legacy `yield_g` column is dead in place (the `temp_override_c` precedent); wire surfaces (MCP `recipe_*`, web) expose sparse mutually-exclusive `yieldG`/`yieldRatio` keys and reject both-present loudly.
- **Temperature is a stored OFFSET, not an absolute** (recipe-relative-temp-offset, migration 31): `temp_offset_c` is a SIGNED delta in °C against the recipe's profile (`tempOffsetC` in maps/MCP/web; 0 = brew at the profile's temperature). The effective brew temperature is always computed `profile espresso_temperature + offset` at use time (`applyActivatedRecipe`, `activeBaselineTemperatureC`), so a profile temperature edit ("Save temperature to profile") moves every recipe on it and can never manufacture a phantom offset — the pre-31 absolute `temp_override_c` did exactly that (the "−5° I never set" report on #1492). The column uses a SIGNED bind (`COL_DBL_SIGNED` — the default `bindDbl` null-collapses ≤ 0 and would drop negative offsets). **Migration**: `temp_override_c` stays in place, dead; migration 31 adds the column with NULL = "unconverted", and the data pass (`convertLegacyTempOffsetsStatic`, run deferred by `MainController::requestRecipeTempOffsetConversion` with a main-thread profile-catalog snapshot) converts `absolute − profile temp` (catalog by title → embedded `profile_json` → drop the pin). Transfer/backup: a legacy-version source (no `temp_offset_c` column) has its absolutes staged into the dest's dead column + NULL marker and converted after import; a current-version source imports its offset VERBATIM — reconverting from the dead column would resurrect cleared offsets. **Portafilter-tea recipes share the same field**: the wizard's tea temperature UI stays absolute and converts at its load/save boundary (`fTeaTempC = profileTemp + offset` / `offset = entered − profileTemp`); it disables when the profile can't be resolved (no anchor → the stored offset is preserved untouched). Hot-water tea has no profile and stores **no** temperature pin — the water vessel is the single source of its temperature (recipe-model hot-water-block rule), so the wizard hides the field and the summary shows the vessel's temperature. Promote-from-shot converts the shot's frozen absolute against the shot's profile snapshot.
- **Steam block** (`steam_json`, compact JSON): `{hasMilk, milkWeightG, pitcherName, durationSec, flow, temperatureC}`. The pitcher is snapshotted **by value** (snapshot-not-reference, like Bean Base); activation re-selects the preset by name and *recreates it from the snapshot if deleted*.
- **Hot-water block** (`hot_water_json`, compact JSON): `{hasWater, vesselName, volume, mode, flowRate, temperatureC, order}` — the opt-in added-hot-water that makes an **Americano** (or **long black**) a recipe. `order` is `"after"` (americano, default) or `"before"` (long black). Hot water is vessel-carried: the selected **water vessel** supplies amount/temperature/flow — there is no separate per-recipe amount. The vessel is snapshotted **by value** like the pitcher (activation re-selects by name, recreates from the snapshot if deleted). Composer stores a full snapshot; the web form stores a name-only block and activation resolves the live vessel's values (`applyActivatedRecipe`). Independent of the steam block — a recipe may carry either, both, or neither. No heater hold (hot water needs no pre-warm), so a milk-less hot-water recipe never lights the steam heater.
- **Drink type** (`drink_type`, add-recipe-wizard-tea): user intent recorded by the wizard — `espresso | filter | americano | long_black | latte | tea | tea_hotwater`. Presentation only (pill icons, wizard re-entry, MCP/web field): activation and machine behavior read the blocks, never this. Empty (legacy rows) = derive via `Recipe::deriveDrinkType(recipe, profileBeverageType)` — precedence: profile-less+water → tea_hotwater; tea_portafilter profile → tea; milk → latte (milk wins over added water); water order → americano/long_black; filter/pourover → filter; else espresso. `requestUpdateRecipe` re-derives it when a block/profile field changes without an explicit drinkType (stored tea/filter categories survive block stamps — see the bev-synthesis comment there). Installed profiles embed no profile JSON, so their `beverage_type` resolves from the catalog: `ProfileManager::beverageTypeForTitle` (surfaces, main thread) / `beverageTypeByTitleSnapshot` (captured on the main thread, passed by value into bg-thread recipe-JSON builders); MCP/web updates attach a transient `profileBeverageType` hint that `requestUpdateRecipe` strips before the column write.
- **`recipeCreated` is a broadcast — correlate with a request token.** Creates race across four surfaces (wizard, MCP, web, clone); every one-shot listener passes a unique transient `requestToken` in the create map (clone: third arg) and filters on `recipe["requestToken"]`, which is echoed on failure too and never stored on the row. An uncorrelated listener receives OTHER surfaces' results.
- **Lifecycle mirrors bags**: 0 shots = hard-deletable; any shots = archive only (`archived` flag), so `shots.recipe_id` provenance never dangles. Hot-water-only recipes never accumulate shots (hot-water dispenses aren't shots) — always hard-deletable.
- Shots record `recipe_id` + `steam_json` + `hot_water_json` snapshots at save time (all three metadata build sites in MainController), so promote-from-shot round-trips the whole drink.
- **Transfer / backup**: `RecipeStorage::importRecipesStatic` merges recipes during device transfer and `.dcbackup` restore (called from `ShotHistoryStorage::importDatabaseStatic` after bags), remapping `equipment_id` through the package id-map, `bag_id` through the bag id-map (an unmatched source bag becomes NULL → stale; wake-on-restock re-homes it later), and `shots.recipe_id` through the recipe id-map. Dedup identity = case-insensitive name + profile_title + bean identity (unchanged — not the bag id). Recipes ride inside `shots.db` — a backup with **zero shots** skips the whole import (pre-existing early-return, same as bags/equipment).

## Activation (single path)

`MainController::activateRecipe(id)` → `RecipeStorage::requestRecipeForActivation` (one background pass: recipe + its LINKED bag loaded by `bag_id` — applied whether or not the bag is still in inventory; only a deleted bag row degrades to bean-less) → `applyActivatedRecipe`:
profile by title with stored-JSON fallback → bag via `setActiveBagKeepFields` + explicit field applies from the bundle's bag map (deterministic; no async `applyActiveBag` race); a **bean-less recipe clears the active bag** (`setActiveBagId(-1)` — the session must not stay attributed to, or write grind into, whatever bag was laying around) → equipment → grind (always the recipe's own `grindPinned`/`rpmPinned`; empty leaves the current dial untouched; the setters' unconditional bag write-through mirrors it onto the linked bag — selecting a recipe that selects a bag counts as dialing it) → queued dose write (beats loadProfile's deferred recommended dose) → yield/temp overrides + re-upload → steam block + heater hold (`hasMilk` → `startSteamHeating("recipe-activated")` now, and `sendMachineSettings` keeps it on while the recipe is active; never fights `keepSteamHeaterOn`) → hot-water block (re-select the snapshotted vessel, push its values via `applyHotWaterSettings()`; **no** heater hold) → `activeRecipeId` last.

QML pill taps, MCP `recipe_activate`, and the web `/activate` route all call this one path. Terminal status: `recipeActivated(id, success)` (queued, lands after the dose write).

**Profile-less recipes** (hot-water tea): activation skips the profile-load, dose-write, and yield/temp-override stages entirely — bag, equipment, and the hot-water block still apply, the loaded espresso profile is untouched, and NO steam-heater hold is ever taken (`activeRecipeHasMilk()` returns false for profile-less recipes so re-sends can't re-assert one). The machine action is the user starting Hot Water.

- **Tweaks write through; ingredient swaps deactivate.** Dose/steam/milk edits and the hot-water vessel selection stamp the active recipe (`stampActiveRecipe`/`stampActiveRecipeHotWater`, echo-guarded like the bag pattern). **Yield/temp do NOT write through** (recipe-aware-brew-settings): they are per-brew overrides living in `Settings.brew` only, never auto-stamped from the live dial — the recipe's yield spec/`tempOffsetC` change only through explicit recipe edits (Brew Settings' single yield/ratio Update button — it sits on whichever row is anchored, labels *Update Recipe*/*Update Bag* by the ladder, and writes `{yieldValue, yieldMode}` via `requestUpdateRecipe`/`persistYieldSpecToBag`; the temp button persists `dialed − profileTemp`, never the absolute — plus the composer and MCP/web `recipe_update`), mirroring how a profile's target/temperature never follow the dial. **The bag splits on the measurement/intent line** (add-yield-ratio-anchor): grind/rpm/dose are dial-in — things the user physically did — and keep their unconditional write-through + the dose stamp on shot save; the bag's yield spec is intent and is button-protected — no shot save, Brew Settings OK, dose capture, or bag selection ever writes it (both old auto-writers are gone). Activation applies the stored yield SPEC verbatim — value and mode: a ratio anchors the session unconditionally in a queued write (so it resolves against the recipe's own dose, never the stale pre-activation one), while an absolute keeps the Bug-A default-comparison (matching the profile default is not an override — absolute only); the temperature applies as `profileTemp + tempOffsetC`, armed only when the offset is non-zero. **In Brew Settings a recipe's `yieldG`/offset-derived temperature are the BASELINE, not overrides of the profile** (recipe-baseline-not-override, #1485): when a recipe is active, the override highlight, the Temp Delta zero-point, the Ratio baseline, `Clear`, and both `Update Recipe` enable-gates all measure against the recipe's own value via `recipeTempBaseline` and the MODE-AWARE anchor pair `baselineYieldValue`/`baselineYieldMode` (`MainController.activeBaselineYieldValue/Mode` — the recipe's own spec when it designs a yield, else the bag's, else the profile; the derived row's baseline computes through the dose so neither row spuriously highlights on a dose change). Consequences: a recipe's designed yield/temp render **un-highlighted** (they're the baseline, not phantom overrides), `Clear` reverts to the recipe's values (not the profile), and `Update Recipe` is enabled ⟺ the shown value deviates from the recipe — so resetting to the profile default is persistable (the old build greyed the button out exactly then). The enable-gate is literally the field's `overridden` state (`Update enabled ⟺ value highlighted`); comparing against the raw stored value instead wrongly enables the temp button at delta 0 when the override is unset. The **live Shot Plan** widget follows the same baseline: `ShotPlanItem` injects the active recipe's resolved yield baseline (`MainController.activeBaselineYieldG` — mode-aware, so a ratio recipe's baseline is its ratio × the dose) and offset-derived temperature into `ShotPlanText` (`baselineYieldG`/`recipeBaselineTemp`), so a recipe's own yield shows as a plain "40.0g" (no "36.0 → 40.0g" arrow) and neither yield nor temp is tinted until a per-brew tweak goes *beyond* the recipe. **Temperature is shown as the recipe's OWN temps, not "profile + delta"**: `TemperatureDisplay::format` / `ProfileManager::temperatureDisplay` take a `baselineShiftC` (the recipe's stored offset) that shifts the frame temps, so a recipe with offset −3 on an `84 · 94°C` profile reads **"81 · 91°C"** on the Shot Plan and **"Recipe: 81 · 91°C"** in Brew Settings (agreeing with the Temp Delta's `0°`) — *a baseline is a baseline*. **Centralized helper**: `MainController` exposes `activeBaselineTemperatureC`/`activeBaselineYieldG` and `temperatureIsRealOverride`/`yieldIsRealOverride` (NOTIFY `brewBaselineChanged`) — one source of truth for "is this a real override vs the active baseline", consumed by the temperature readout (`TemperatureItem`) and the custom-brew-widget highlight (`CustomItem`), so those widgets don't re-derive the recipe-vs-profile comparison and drift. (CustomItem's `%TARGET_TEMP%` token shows the *effective* brew temp — the `Settings.brew` override when set — to stay aligned with `%TARGET_WEIGHT%`.) The **display-layer injection props** (`baselineYieldG`/`recipeBaselineTemp` on `ShotPlanText`, and the formatter's `baselineShiftC`) default to 0/off. **Recipe cards render the same baseline, resolved against their own profile** (recipe-relative-temp-offset, revised for [#1482](https://github.com/Kulitorum/Decenza/issues/1482)): each card still resolves **its own profile's** frame temps (`profileStepTemps` → `temperatureDisplayForSteps`; `RecipesPage.refreshProfileNumbers` resolves title → embedded JSON → omit the segment, never the loaded profile's frames) — a card can't use the live baseline props above because `ProfileManager.temperatureDisplay()` reads the machine's currently-loaded profile in C++, and a card must stay correct while a *different* profile is loaded — but the recipe's stored offset is now folded silently into those frames as a shift (`recipeTempOffsetC` passed as `temperatureDisplayForSteps`'s `baselineShiftC`) instead of rendered as a separate highlighted tag, and the yield shows the plain resulting value with no arrow (`yieldOverridden: false`). So a card and the live Shot Plan now agree exactly (card: `81 · 91°C`; live: `81 · 91°C`), with no highlight on either — a card has no live override to report, only a static recipe definition. The wizard's summary hero renders by the card rule (`fProfileStepTemps`). **Shot Review / Shot Detail** still show what was overridden at shot time (relative to the shot's own profile). Grind/rpm edits stamp the active recipe's own `grindPinned`/`rpmPinned` AND write through to the bag in parallel, off the same edit — there is no suspension flag and no routing decision (tea recipes are the one exception: the recipe stamp is skipped so a dial edit can't turn a tea recipe into one that grinds). Changing profile/bag/equipment deactivates — the watchers compare against the recipe's **own** ingredients, so re-selecting the same thing (or startup auto-load) never deactivates. **Same-id re-activation** (re-tapping the active recipe after an edit, the #1466/#1471 flow) re-pushes the in-memory `m_activeRecipe` cache instead of doing a fresh DB read — a fresh read can race the user's own in-flight write-through and revert the edit (Bug B2); a bag-link change since activation (`bagId != resolvedBagId`) falls through to the full fresh read.
- Active id persists in `SettingsDye` (`dye/activeRecipeId`); startup restores the pill highlight + the active-recipe cache without re-applying values.

## Surfaces

- **QML**: `RecipesItem` idle widget (BeansItem clone — MRU pills of the last 5 with drink-type icons via `PresetPillRow`'s optional per-preset `icon`, double/long-press → `RecipesPage`), `RecipeWizardPage` (replaced `RecipeComposerPage`, add-recipe-wizard-tea: drink-type-first steps with auto-advance + breadcrumb chips; the SUMMARY step is the edit page — edit/clone/promote land there via the same `mode`/`editRecipeId`/`prefill`/`promoteShotId` contract; templates pre-seed blocks per drink type but the summary's Add/Remove milk/water buttons keep every combination expressible). Profile step filters by the profile's `beverage_type` per drink type and ranks: ① used with this bean (`requestRankedProfilesForBean`), ② KB-recommended — coffee: `roastAffinity` from the profile knowledge base (`ShotSummarizer::roastAffinityForTitle` via `ProfileManager.kbProfileSuitsRoast`; authored per-entry only where the profile's own docs state roast suitability); tea: type-matched stock profiles (`DrinkTypes::teaProfileMatchesType`) — + similar beans, capped to 5, ③ rest (tea: temp proximity to the bag's `brewTempC`). Grind hint on the details step: latest grind for the bean (any profile; same-roast fallback — `requestLatestGrindForBean`) + UGS direction via `ShotSummarizer::grindDirectionBetween` when the picked profile differs (direction only, never a computed number — the KB's cross-profile rule). Details step prefills from the latest shot with the bean+profile pair (`requestLatestShotForBeanProfile`) → tea-bag brewing data → profile defaults; portafilter tea only seeds a temp override when the profile is NOT type-matched. Equipment defaults per drink type (`requestLastEquipmentForDrinkType`). Tea profile step has a fixed "Just hot water" row → profile-less recipe. Promote buttons on ShotHistoryPage (beside Load), ShotDetailPage header, AutoFavoritesPage rows. Navigate target is `recipeList` — bare `recipes` is the pre-existing profile Recipe Editor.
- **Shared card component** (`qml/components/RecipeDrinkCard.qml`, design D4): the management-page card AND the wizard summary's WYSIWYG hero are the same component (plain properties, no page-specific lookups; a `footer` alias carries the management page's action row). Card lines: name anchor + Active badge / drink icon + SHORT type label + profile (always shown) + milk weight ("200g milk", never a bare "milk") / bag + shot count (or the stale re-point affordance) / plan line (or the vessel snapshot for profile-less hot-water tea). Lines wrap, never elide the profile. Drink-type labels/icons live in the `DrinkType` QML singleton (`qml/components/DrinkType.qml`): `shortLabel()` for cards/pills/auto-names, `longLabel()` only for the wizard's drink-type picker; `suggestName()` uses short labels and skips the type word when the bean name already ends with it.
- **MCP**: `mcptools_recipes.cpp` — see `MCP_SERVER.md` for the tool table. Register-stub gotcha applies (`tst_mcpserver_session/protocol`). Recipe reads expose `bagId` + a `bag` display object + `bagStale` (human-readable string) + the recipe's own grind as `grind: {value, rpm}` (no inherited/pinned mode — nothing resolves from the bag); create/update accept `bagId`; a create that omits grind adopts the linked bag's dial at save time.
- **Web**: `shotserver_recipes.cpp` (+ `shotserver_bags.cpp`, `shotserver_equipment.cpp`) — see `SHOTSERVER.md`.

## Gotchas

- `ShotProjection` gained `recipeId`/`steamJson`/`hotWaterJson`; the composer prefill reads them from `shotReady`.
- Tea (add-recipe-wizard-tea): bags carry a creation-time `kind` ("coffee"/"tea", migration 28) — see BEAN_BASE.md for the Add Tea flow and tea extraction. Tea recipes never store grind; equipment packages may be grinder-less (basket-only) for tea setups.
- Adding a recipe column = a kCols row in recipestorage.cpp **plus** the CREATE TABLE + a migration step (same rule as bags). A shot-snapshot column (like `hot_water_json`) also needs the `ShotRecord`/`ShotSaveData`/`ShotMetadata` structs, the shot INSERT + read SELECT, and the `ShotProjection` mapping — see the `steam_json` sites.
- The steam heater takes **5–9 minutes** to warm (measured). An active milk recipe therefore HOLDS the heater on: `sendMachineSettings` treats `activeRecipeHasMilk()` like `keepSteamHeaterOn`, so re-sends (wake/reconnect/edits) keep it warm; deactivation or a milk-less switch releases the hold. Milk-less recipes never force the heater off (keep-on users unaffected). Hot water, by contrast, has **no** heater hold.
- Device transfer / backup import now copies the `recipes` table (`importRecipesStatic`, mirroring bags/equipment). Recipes ride inside `shots.db`, so a zero-shot backup still skips them (pre-existing early-return).
